1. مقدمة وهيكل الكتاب

كتاب Bug Bounty Playbook يمثل دليلاً انتقالياً رائعاً لمن يمتلك أساسيات الشبكات والويب ويريد البدء الفعلي في كسب مكافآت برامج Bug Bounty. يتميز هذا الكتاب بتركيزه التقني والتطبيقي المباشر بعيداً عن التنظير البحت، حيث يركز الكاتب Alex Thomas (Ghostlulz) على بناء خطة لعب منظمة مقسمة إلى ثلاث مراحل أساسية:

  1. مرحلة التجهيز (Pre-Game): إعداد الأدوات والبيئة الخاصة بك.
  2. مرحلة جمع المعلومات (Reconnaissance): رسم الخريطة الكاملة للهدف.
  3. مرحلة استكشاف الثغرات واستغلالها (Exploitation): الهجوم الحقيقي للوصول للثغرة وإثبات الأثر.
القيمة الجوهرية للكتاب: يساعدك الكتاب على تجنب العشوائية في الفحص والتنقل بين الأنظمة، ويعطيك "كتيب إرشادات" واضحاً للتحقق من ثغرات محددة بناءً على نوع النظام والتقنية المكتشفة.

2. إعداد البنية التحتية (Pre-Game)

يركز الكاتب على أن الفشل في إعداد بيئتك بشكل احترافي سيتسبب في ضياع الكثير من الثغرات. يوصي الكتاب بالآتي:

أ. إعداد سيرفر VPS (Virtual Private Server)

تشغيل سكريبتات الفحص والاستطلاع بشكل دائم (24/7) دون الاعتماد على موارد جهازك الشخصي وسرعة الإنترنت لديك. هذا يتيح لك الاستمرار في عمل Brute Forcing و DNS resolution بدون انقطاع.

ب. إعداد نطاق شخصي (Custom Domain)

ضروري جداً لتلقي اتصالات الـ Out-Of-Band (مثل تفاعلات ثغرات Blind XSS أو SSRF) باستخدام أدوات مثل Interactsh أو إعداد خادم DNS خاص بك. عندما تقوم بحقن كود يستدعي خادماً خارجياً، سيتصل الخادم بالنطاق الخاص بك لتأكيد وجود الثغرة.

ج. تنظيم قاعدة البيانات والملفات

تنظيم المخرجات في مجلدات مقسمة لكل هدف، وتوثيق جميع النطاقات الفرعية المكتشفة لمتابعة الفحص لاحقاً. يوصي الكاتب بإنشاء هيكل مجلدات شبيه بالتالي:

~/bugbounty/ ├── target_name/ │ ├── recon/ │ │ ├── subdomains.txt │ │ ├── alive.txt │ │ └── ports.txt │ └── vulns/

3. استراتيجية الاستطلاع والاستكشاف (Recon)

عملية جمع المعلومات هي المفتاح الأساسي. كلما جمعت معلومات أكثر عن الهدف، زادت مساحة الهجوم المتاحة وازدادت فرصك في العثور على ثغرات فريدة.

أ. جمع النطاقات الفرعية (Subdomain Enumeration)

الجمع يعتمد على الدمج بين الاستطلاع السلبي (Passive Recon) والاستطلاع النشط (Active Recon):

  • الاستطلاع السلبي (Passive): الاستعلام من قواعد البيانات والشهادات المؤرشفة دون إرسال أي حزم مباشرة للهدف.
    subfinder -d target.com -all -o passive_subs.txt
  • الاستطلاع النشط (Active Brute Forcing): تخمين النطاقات الفرعية باستخدام قواميس كلمات ضخمة ومحللات DNS سريعة:
    puredns bruteforce wordlist.txt target.com -r resolvers.txt -o active_subs.txt

ب. كشف المنافذ والخدمات (Port Scanning & Web Server Verification)

بعد تجميع النطاقات الفرعية وتنظيفها من التكرار، يتم فلترتها لمعرفة النطاقات النشطة التي تقدم خدمات ويب فعلاً:

cat subs.txt | httpx -title -status-code -content-length -o alive.txt

ثم فحص المنافذ المفتوحة الأخرى للبحث عن خدمات غير محمية مثل FTP, SSH, Database, or Jenkins:

nmap -sV -sC -p- -T4 -iL alive.txt -oA nmap_results

ج. فحص المسارات والمجلدات (Directory Brute Forcing)

البحث عن الملفات الحساسة والمنسية (مثل ملفات النسخ الاحتياطي `.bak`, `.conf`, ملفات التكوين `.git`, `.env`). استخدام أداة سريعة مثل `ffuf`:

ffuf -w wordlist.txt -u https://target.com/FUZZ -mc 200,301,302,403 -e .bak,.zip,.git,.env

4. صيد الأهداف سهلة المنال (Easy Wins)

ثغرات لا تتطلب كتابة كود معقد ولكنها بالغة الخطورة وتكتشف سريعاً بمجرد انتهاء الاستطلاع.

أ. ثغرات السيطرة على النطاقات الفرعية (Subdomain Takeover)

تحدث عندما يشير سجل CNAME لنطاق فرعي لخدمة خارجية (مثل AWS S3, GitHub Pages, Heroku, Shopify) ولكن هذه الخدمة تم حذفها أو لم يتم حجزها بالخدمة. يمكن للمهاجم حجز هذا الاسم في الخدمة الخارجية لصالحه ليعرض محتوى خبيث تحت اسم النطاق الأصلي للشركة!

بصمات الخدمات الشهيرة (Fingerprints):
  • GitHub Pages: 404 Not Found / "There isn't a GitHub Pages site here."
  • AWS S3: NoSuchBucket / "The specified bucket does not exist."
  • Heroku: "There's nothing here, yet."

أداة التحقق التلقائي السريع:

subjack -w subs.txt -t 100 -timeout 30 -o takeovers.txt -ssl

5. استغلال أنظمة إدارة المحتوى (CMS)

الكثير من الشركات تستخدم أنظمة إدارة المحتوى كواجهة لمدوناتها أو مواقعها التعريفية. الأنظمة القديمة أو المكونات الإضافية (Plugins) غير المحدثة هي مناجم ذهب للهكرز.

أ. فحص الووردبريس (WordPress Pentesting)

البحث عن المكونات الإضافية الضعيفة وحقن الملفات وتسريب المستخدمين:

wpscan --url https://target.com --enumerate vp,vt,u --api-token YOUR_WPSEC_TOKEN

يركز الكتاب على التحقق من ملفات التثبيت المنسية (مثل wp-config.php.bak) أو فحص واجهات الـ APIs الحساسة للووردبريس مثل /wp-json/wp/v2/users لتسريب أسماء المستخدمين.

6. ثغرات الحقن والويب الشهيرة (OWASP)

الشرح التقني التفصيلي للثغرات البرمجية في منطق التطبيق وكيفية العثور عليها يدوياً.

أ. حقن الاستعلامات (SQL Injection - SQLi)

التلاعب بالمدخلات لتنفيذ أوامر بقاعدة البيانات. ينقسم الحقن إلى:

  • Union-Based: استخراج البيانات مباشرة بدمج استعلام المهاجم مع استعلام الموقع الأصلي باستخدام كلمة `UNION`.
  • Blind SQLi: عندما لا تظهر البيانات مباشرة، نستخدم الحقن الزمني (Time-Based) أو المنطقي (Boolean-based):
    # حقن زمني في قاعدة بيانات MySQL http://target.com/product.php?id=1 AND SLEEP(5)

يوضح الكتاب كيفية التحقق اليدوي البسيط قبل استخدام `sqlmap` لتجنب كشف فحصك بواسطة الـ WAF (جدار الحماية):

sqlmap -u "http://target.com/product.php?id=1" --batch --random-agent --dbms=mysql --level=3

ب. تضمين الملفات المحلية (Local File Inclusion - LFI)

قراءة ملفات السيرفر الداخلية الحساسة. يشرح الكتاب كيفية استخدام ممرات المسار لتخطي الفلاتر:

# فلاتر وتجاوزات قراءة الملفات في خادم لينكس http://target.com/view.php?file=../../../../etc/passwd http://target.com/view.php?file=....//....//....//etc/passwd

استخدام مغلفات بي إتش بي لقراءة السورس كود مشفراً بقاعدة Base64:

http://target.com/view.php?file=php://filter/read=convert.base64-encode/resource=config.php

ج. ثغرات الـ XSS (Cross-Site Scripting)

حقن جافا سكريبت لتنفيذها في متصفح الضحية. يركز الكتاب على Blind XSS، وهي ثغرات XSS مخزنة ولكن لا يتم تنفيذها إلا في لوحات الإدارة الخلفية (Admin Panel). يوصي باستخدام XSS Hunter لتوليد البايلودز وتلقي التنبيهات مع لقطة شاشة للوحة الإدارة بمجرد تفعيل الثغرة.

7. ثغرات SSRF, XXE, IDOR المتقدمة

ثغرات معقدة برمجياً ولكن أثرها بالغ جداً على خوادم الشركات ومواردها الداخلية.

أ. تزييف الطلبات من جهة الخادم (SSRF)

إجبار السيرفر على إرسال طلبات بالنيابة عن المهاجم للوصول لخدمات داخلية بالشبكة. الهدف الأكبر في بيئات السحاب (Cloud Apps) هو الوصول لبيانات المثيل (Metadata Endpoints):

# رابط جلب بيانات الهوية والـ IAM Credentials في AWS http://169.254.169.254/latest/meta-data/

تجاوز فلاتر منع الـ localhost باستخدام التمثيل الرقمي البديل:

# تمثيل الـ Localhost بصور بديلة لتخطي الحظر http://2852039166 (تمثيل عشري لـ 127.0.0.1) http://0177.0.0.1 (تمثيل ثماني لـ 127.0.0.1)

ب. حقن كيانات XML الخارجية (XXE)

إدخال كيانات خبيثة في ملفات XML التي يحللها السيرفر لقراءة ملفات داخلية أو عمل مسح للشبكة الداخلية:

<?xml version="1.0" encoding="ISO-8859-1"?> <!DOCTYPE foo [ <!ELEMENT foo ANY > <!ENTITY xxe SYSTEM "file:///etc/passwd" >]> <foo>&xxe;</foo>

8. كتابة التقارير ودمج الثغرات (Chaining)

أنت لست مجرد هكر يبحث عن ثغرة، بل أنت تقدم استشارة أمنية للشركة. تقريرك هو المنتج النهائي الذي يحدد المكافأة المالية.

أ. هيكل التقرير الاحترافي

  • العنوان (Title): مباشر ويحدد الثغرة والمكان بدقة.
  • الوصف (Description): شرح منطقي ومبسط للثغرة.
  • خطوات إعادة الإنتاج (Steps to Reproduce): خطوات مرقمة واضحة ومختصرة يمكن لأي مطور تنفيذها للتأكد.
  • الأثر الأمني (Impact): ماذا يستفيد المخترق الحقيقي؟ (تخطي الصلاحيات، سرقة الحسابات).
  • إثبات المفهوم (PoC): صور واضحة أو رابط فيديو قصير يوضح المشكلة.

ب. دمج الثغرات (Vulnerability Chaining)

كيف تحول ثغرة ذات تأثير منخفض (Low) إلى ثغرة بالغة الأثر وعالية القيمة (High/Critical)؟ دمج ثغرة CSRF مع ثغرة Stored XSS لتغيير كلمة مرور المدير بمجرد زيارته لصفحتك الشخصية، أو دمج SSRF مع ثغرة LFI للوصول للمفاتيح السرية للسحابة.

9. أسئلة اختبار الفهم التفاعلية

🧪 أسئلة مراجعة الكتاب (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: يوصي الكتاب باستخدام خوادم استطلاع خارجية (VPS) مع تدوير وتغيير الـ User-Agent بشكل عشوائي، وتقليل معدل إرسال الطلبات (Rate Limiting) عبر خيارات التأخير، واستخدام خوادم DNS محلية موثوقة ونظيفة، وتفضيل الاستطلاع السلبي أولاً.
الإجابة النموذجية: سجل CNAME يشير إلى اسم نطاق آخر خارجي (مثل `mybucket.s3.amazonaws.com`)، وبالتالي إذا انتهت هذه الخدمة يسهل للمهاجم تسجيل نفس الاسم بالخدمة الخارجية. أما سجل A فيشير مباشرة إلى عنوان IP، ويكون استغلال الـ Takeover في سجل A ممكناً فقط إذا تم تحرير وتغيير عنوان IP من مزود السحاب (مثل AWS EC2) دون إزالة سجل DNS، فيقوم المهاجم بحجز نفس الـ IP في السحابة إن أمكن.
الإجابة النموذجية: ثغرة DOM XSS تحدث بالكامل داخل متصفح المستخدم (Client-Side) حيث يتم معالجة البيانات المدخلة وتمريرها لوظيفة تنفيذية خبيثة (Sink) مثل `innerHTML` أو `eval` بواسطة أكواد جافا سكريبت على الصفحة، دون إرسال المدخل الخبيث بالضرورة لخادم الويب. أما Reflected XSS فيمر البايلود عبر خادم الويب ويقوم الخادم بإدراجه مجدداً في استجابة الـ HTML المرسلة للمتصفح.
الإجابة النموذجية: نقوم بحقن رابط الـ Metadata الخاص بمزود السحابة (مثلاً `http://169.254.169.254/latest/meta-data/iam/security-credentials/role_name` في AWS) كقيمة لمعامل الـ SSRF. سيقوم الخادم داخلياً بجلب الرابط بالنيابة عنا ويعود بالاستجابة محملة بالـ Access Key والـ Secret Key والـ Session Token الحساسة.
الإجابة النموذجية: مغلفات PHP (مثل `php://filter`) تتيح للمهاجم قراءة ملفات السورس كود الخاصة بـ PHP وسحب كود الاتصال بقاعدة البيانات. بدلاً من محاولة تنفيذ الملف، نستخدم المغلف لتحويل المحتوى إلى ترميز Base64، فيتم عرضه كنص في متصفحنا ثم نقوم بفك تشفيره للحصول على الكود المصدري الأصلي.
الإجابة النموذجية: هو هجوم يقوم فيه المهاجم بتسجيل نطاق يملك خادم DNS متحكم به. عند محاولة الخادم فحص وحل النطاق (Resolve)، يعود خادم الـ DNS الخاص بنا بأي بي عام سليم لتخطي فلتر المنع الأولي للخادم. وبمجرد قبول الطلب، يعود خادم الـ DNS لربط النطاق بـ IP محلي (مثل 127.0.0.1) بفترة صلاحية قصيرة جداً (TTL=0)، فيقوم الخادم بإرسال الطلب الفعلي للـ IP المحلي الداخلي.
