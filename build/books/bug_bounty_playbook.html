The Bug Bounty Playbook

By Alex Thomas (Ghostlulz)
الصعوبة: متوسط إلى متقدم مدة القراءة: 3 ساعات التركيز الأساسي: الاستطلاع والاستغلال العملي
فهرس الدليل الدراسي
1. مقدمة وهيكل الكتاب 2. مرحلة التجهيز (Pre-Game) 3. استراتيجية الاستطلاع الذكي (Recon) 4. صيد الأهداف السهلة (Easy Wins) 5. استغلال ثغرات الويب الشهيرة (OWASP) 6. الثغرات المتقدمة وإدارة الـ API 7. أسئلة اختبار الفهم التفاعلية

1. مقدمة وهيكل الكتاب

كتاب Bug Bounty Playbook يمثل دليلاً انتقالياً رائعاً لمن يمتلك أساسيات الشبكات والويب ويريد البدء الفعلي في كسب مكافآت برامج Bug Bounty. يتميز هذا الكتاب بتركيزه التقني والتطبيقي المباشر بعيداً عن التنظير البحت، حيث يركز الكاتب Alex Thomas (Ghostlulz) على بناء خطة لعب منظمة مقسمة إلى ثلاث مراحل أساسية:

  1. مرحلة التجهيز (Pre-Game): إعداد الأدوات والبيئة الخاصة بك.
  2. مرحلة جمع المعلومات (Reconnaissance): رسم الخريطة الكاملة للهدف.
  3. مرحلة استكشاف الثغرات واستغلالها (Exploitation): الهجوم الحقيقي للوصول للثغرة وإثبات الأثر.
القيمة الجوهرية للكتاب: يساعدك الكتاب على تجنب العشوائية في الفحص والتنقل بين الأنظمة، ويعطيك "كتيب إرشادات" واضحاً للتحقق من ثغرات محددة بناءً على نوع النظام والتقنية المكتشفة.

2. مرحلة التجهيز (Pre-Game)

يركز الكاتب على أن الفشل في إعداد بيئتك بشكل احترافي سيتسبب في ضياع الكثير من الثغرات. يوصي الكتاب بالآتي:

  • إعداد سيرفر VPS خاص بك: لتشغيل سكريبتات الفحص والاستطلاع بشكل دائم (24/7) دون الاعتماد على موارد جهازك الشخصي وسرعة الإنترنت لديك.
  • إعداد نطاق شخصي (Custom Domain): ضروري لتلقي اتصالات الـ Out-Of-Band (مثل تفاعلات ثغرات Blind XSS أو SSRF) باستخدام أدوات مثل Interactsh أو إعداد خادم DNS خاص بك.
  • ترتيب وتنظيم البيانات: تنظيم المخرجات في مجلدات مقسمة لكل هدف، وتوثيق جميع النطاقات الفرعية المكتشفة لمتابعة الفحص لاحقاً.
نصيحة الكاتب: استخدم خوادم VPS رخيصة (مثل DigitalOcean أو Linode) وقم بتنصيب أدوات الاستطلاع الأساسية بها مع عمل حماية قوية للسيرفر عبر SSH keys.

3. استراتيجية الاستطلاع الذكي (Recon)

يشرح الكتاب منهجية صارمة لعملية جمع المعلومات مقسمة إلى مرحلتين أساسيتين:

أولاً: جمع النطاقات الفرعية (Subdomain Enumeration):

الجمع يعتمد على الدمج بين الاستطلاع السلبي (Passive Recon) باستخدام قواعد البيانات السحابية، والاستطلاع النشط (Active Recon) عبر تخمين الأسماء:

  • الاستطلاع السلبي: استخدام أدوات مثل amass و subfinder و assetfinder لجمع النطاقات الفرعية المؤرشفة مسبقاً.
  • الاستطلاع النشط: فحص تخميني عبر قواميس كلمات ضخمة باستخدام أداة دمج ذكية وسريعة مثل shuffledns أو gobuster.

ثانياً: التحقق وفحص المنافذ والخدمات (Port Scanning & Fingerprinting):

بمجرد جمع النطاقات الفرعية، يجب التحقق من النطاقات التي تعمل بالفعل وتستجيب لطلبات HTTP/HTTPS باستخدام أداة مثل httpx، ثم فحص المنافذ المفتوحة للخدمات غير المرتبطة بالويب باستخدام Nmap:

# فحص سريع للمنافذ المفتوحة nmap -sV -sC -p- -T4 target-subdomain.com

4. صيد الأهداف السهلة (Easy Wins)

يركز هذا القسم على الثغرات التي يمكن اكتشافها بسرعة فائقة بمجرد الانتهاء من مرحلة الاستطلاع وقبل الخوض في اختبارات الثغرات المعقدة:

1. ثغرات السيطرة على النطاقات الفرعية (Subdomain Takeover):

تحدث عندما يشير سجل الـ DNS (سواء سجل CNAME أو A) لنطاق فرعي معين إلى خدمة سحابية خارجية (مثل GitHub Pages، أو AWS S3، أو Heroku)، ولكن هذه الخدمة تم حذفها أو لم يتم إعدادها بشكل صحيح بداخل حساب مالك النطاق. يمكن للمهاجم تسجيل حساب في تلك الخدمة وحجز الاسم لصالحه ليعرض محتوى خبيث تحت اسم النطاق الأصلي للشركة!

كيف تكتشفها؟ ابحث عن الأخطاء الشهيرة مثل "NoSuchBucket" (في أمازون) أو "There is no app here at this subdomain" (في هيروكو) أثناء استعراض النطاقات الفرعية عبر متصفحك أو أدوات الفحص التلقائي مثل subjack.

2. استغلال أنظمة إدارة المحتوى (CMS Exploitation):

تطبيقات الويب التي تستخدم أنظمة مشهورة مثل WordPress أو Joomla تمثل هدفاً متكرراً. يوضح الكتاب كيفية فحص المكونات الإضافية (Plugins) والقوالب القديمة والبحث عن ثغرات عامة معروفة لها (1-days) باستخدام أدوات مثل wpscan:

# فحص موقع ووردبريس وكشف الإضافات الضعيفة بشكل كامل wpscan --url https://target.com --enumerate vp,vt,u

5. استغلال ثغرات الويب الشهيرة (OWASP)

يفصل الكتاب طرق البحث والاستغلال اليدوي لأخطر ثغرات تطبيقات الويب:

1. ثغرات حقن قواعد البيانات (SQL Injection):

يشرح الكتاب كيف يتم فحص المدخلات والـ Parameters للتأكد من تأثرها بالحقن، ويغطي الأنواع المختلفة:

  • In-band SQLi: حيث تظهر نتائج الاستعلام مباشرة في محتوى الصفحة.
  • Blind SQLi: عندما لا تظهر النتائج بالصفحة ولكن يمكن استنتاج البيانات من خلال تغييرات في منطق الصفحة (Boolean-based) أو فترات الانتظار الزمنية للاستجابة (Time-based).
# مثال لاختبار حقن زمني في قاعدة البيانات http://target.com/item.php?id=1' AND (SELECT 502 FROM (SELECT(SLEEP(5)))b)--

2. ثغرات الملفات والممرات (LFI & Path Traversal):

تحدث عندما يسمح التطبيق للمستخدم بإدخال مسار ملف يتم تحميله داخلياً دون تنقية كافية. يوضح الكتاب كيفية قراءة الملفات الحساسة في السيرفر مثل ملف الإعدادات أو ملف المستخدمين:

# قراءة ملف المستخدمين في سيرفرات لينكس http://target.com/page.php?file=../../../../etc/passwd

3. ثغرات حقن النصوص البرمجية عبر المواقع (XSS):

يقوم المهاجم بحقن أكواد جافا سكريبت لتنفيذها داخل متصفح زوار الموقع الآخرين. يركز الكتاب على طرق تخطي حمايات المواقع (WAF Bypass) واستهداف الـ DOM XSS بشكل ذكي.

6. الثغرات المتقدمة وإدارة الـ API

ينتقل الكتاب في الفصول الأخيرة لشرح ثغرات بالغة الأثر الأمني وتعتبر من أكثر الثغرات قيمة مالية في برامج المكافآت حالياً:

1. ثغرة تزييف الطلبات من جهة الخادم (SSRF):

تحدث عندما يجبر المهاجم خادم الويب على إرسال طلب HTTP إلى خوادم داخلية في شبكة الشركة لا يمكن الوصول إليها من الخارج مباشرة (مثل الوصول لبيانات خوادم إدارة الهوية AWS Metadata بقيمة http://169.254.169.254/latest/meta-data/).

2. ثغرة حقن كيانات XML الخارجية (XXE):

عندما يقبل التطبيق ملفات XML ويقوم بمعالجتها دون تعطيل الكيانات الخارجية، مما يتيح للمهاجم قراءة ملفات السيرفر الداخلية أو إحداث هجمات SSRF.

3. ثغرات التحكم بالوصول والـ IDOR:

تلاعب بالمعرفات الحساسة (مثل معرفات الحسابات والطلبات) للوصول لبيانات مستخدمين آخرين دون قيود أو صلاحيات صحيحة.

7. أسئلة اختبار الفهم التفاعلية

🧪 أسئلة مراجعة الكتاب (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: الاستطلاع السلبي يجمع المعلومات عن الهدف دون التفاعل معه مباشرة (مثال: الاستعلام من سجلات شهادات SSL أو محركات البحث)، بينما الاستطلاع النشط يتضمن إرسال طلبات وحزم مباشرة للهدف (مثال: فحص المنافذ Nmap وتخمين المجلدات Ffuf) مما قد يؤدي لتسجيل عنوان IP الخاص بك في سجلات الهدف.
الإجابة النموذجية: تحدث عندما يشير سجل CNAME الخاص بنطاق فرعي لخدمة خارجية تم إلغاؤها أو لم يتم حجزها بالخدمة. يمكن التحقق منها برمجياً باستخدام أدوات مثل `subjack` أو `nuclei` والتي تقارن استجابة الخادم مع بصمات أخطاء الخدمات الشهيرة (مثل رسائل أخطاء GitHub Pages أو AWS).
الإجابة النموذجية: لأنها تسمح للمهاجم بالوصول لخدمات البيانات الوصفية للمثيل السحابي (Metadata Services) مثل عنوان `169.254.169.254` في AWS، والتي قد تحتوي على كلمات مرور مؤقتة وصلاحيات ربط واستيقان (IAM credentials) تتيح للمهاجم السيطرة الكاملة على البنية السحابية للشركة.
