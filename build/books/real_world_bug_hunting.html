1. مقدمة وكيفية قراءة تقارير الثغرات (Analyzing Real Reports)

يشرح هذا الفصل الأهمية القصوى لدراسة تقارير الثغرات العامة المنشورة من قبل باحثين آخرين على منصات مثل HackerOne لمعرفة مسارات المطورين المعتادة في ارتكاب الأخطاء.

2. ثغرات التحويل المفتوح Open Redirect

شرح لكيفية تصفية وتجاوز حمايات الـ Redirect المعتمدة على النطاقات الفرعية، وحقن روابط خارجية بالهندسة الاجتماعية.

3. ثغرات حقن النص البرمجي XSS

تحليل لحالات حقيقية تم اكتشافها في موقع Yahoo و Slack، والتركيز على استخراج الـ Cookies وتخطي الـ CSP.

4. ثغرات تزوير الطلب CSRF

كيفية استغلال عدم وجود حماية ضد الـ CSRF في تغيير البيانات الشخصية وحسابات الدفع وتخطي الـ API Keys.

5. ثغرات IDOR وتخطي الصلاحيات

من أكثر الثغرات شيوعاً وقبولاً في برامج الـ Bug Bounty. يعلمك الكاتب كيفية التحقق من صلاحيات الـ backend عن طريق تغيير الـ Parameters.

6. ثغرات SSRF وتسريب السحاب

استعراض لكيفية وصول المهاجم لمعلومات الـ Metadata لسيرفرات AWS و Google Cloud عن طريق إجبار السيرفر على جلب طلبات خارجية.

7. ثغرات XXE ومعالجة XML الحساسة

قراءة ملفات النظام الداخلية وقاعدة البيانات عن طريق معالجة ملفات XML ضعيفة التهيئة والتسبب بحجب الخدمة DoS.

8. كتابة وصياغة التقارير الفائزة (Reporting Generator)

هيكل التقرير الأمني النهائي المولد أوتوماتيكياً للثغرة المكتشفة:

Title: IDOR in /api/v1/user/settings?id=

Summary:
I have discovered a security vulnerability in /api/v1/user/settings?id= due to lack of server-side authorization controls.

Steps to Reproduce:
1. Log in to the application as User A.
2. Intercept the request to /api/v1/user/settings?id=
3. Modify the ID parameter value to User B's ID.
4. Observe that we can access and manipulate sensitive settings data of User B.

Impact:
This vulnerability allows for Account Takeover.

💡 نصائح وتوجيهات الكاتب الميدانية (Real-World Golden Advice)

جمعنا لك هنا زبدة الخبرة العملية والنصائح الذهبية التي يركز عليها الكاتب في فصول كتابه للتنقيب والعمل الميداني الفعلي:

  • تتبع تحديثات المكونات مفتوحة المصدر: فحص ثغرات الـ N-Days في الإضافات والمكتبات المستخدمة بالهدف يجلب ثغرات سهلة ومضمونة.
  • افحص بوابات الربط مع الأطراف الخارجية (Integrations): عمليات تسجيل الدخول بـ OAuth أو الدفع بـ Stripe غالباً ما يخطئ المطورون في ربطها.
  • ابحث عن الـ Parameter Pollution: عند قيام واجهات الـ APIs بتوجيه الطلبات داخلياً، قد يؤدي تكرار المعاملات لتجاوز التحقق بالخلفية.

10. بنك الأسئلة والاختبارات التفاعلية الـ 10

🧪 بنك أسئلة مراجعة كتاب Real-World Bug Hunting (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: يمكن تجاوزها عبر استغلال معاملات الترميز (URL encoding) أو علامات المسار (مثل `@` أو `//`) أو استخدام النطاقات الفرعية التي يملكها المهاجم ولكنها تحتوي على اسم الموقع الموثوق (مثل `attacker.com?whitelist.com`).
الإجابة النموذجية: يتم ذلك عن طريق حقن كود جافا سكريبت يقرأ كوكيز الجلسة (`document.cookie`) ويرسلها إلى خادم استماع خارجي للمهاجم، بشرط ألا يكون خيار `HttpOnly` مفعل على الكوكي المستهدفة.
الإجابة النموذجية: هي عبارة عن قيمة عشوائية يتم إنتاجها بالسيرفر وربطها بجلسة المستخدم الحالي، ويتم إرسالها مع أي طلب حساس. عندما يستقبل السيرفر الطلب، يقارن القيمة المرسلة بالقيمة المخزنة لديه بالجلسة، وفي حال اختلافها أو غيابها يرفض الطلب تماماً.
الإجابة النموذجية: الـ IDOR هي تخطي صلاحيات أفقي (Horizontal)، حيث يقوم مستخدم بالدخول لبيانات مستخدم آخر يملك نفس المستوى من الصلاحيات. أما الرأسية (Vertical)، فهي عندما ينجح مستخدم عادي في رفع صلاحياته لتنفيذ عمليات الأدمن أو المشرف.
الإجابة النموذجية: نقوم بحقن عنوان الـ IP الداخلي الخاص بالـ AWS وهو `169.254.169.254` في الرابط المصاب بالـ SSRF. سيقوم السيرفر بطلب العنوان داخلياً، ويعيد لنا معلومات الـ IAM Credentials ومفاتيح الاستيقان الخاصة بحساب السحابة.
الإجابة النموذجية: يتم منع ثغرات XXE نهائياً عن طريق تعطيل معالجة الكيانات الخارجية (External Entities) والـ DTDs في إعدادات محلل الـ XML (XML Parser) المستخدم في لغة البرمجة بالموقع الخلفي.
الإجابة النموذجية: هي ثغرة تسمح بتمرير أوامر لقاعدة البيانات مباشرة وتجاوز التحقق من الهوية، وتختلف عن بقية أنواع الحقن (مثل حقن الأوامر) في أنها تستهدف محرك قاعدة البيانات فقط وليس نظام التشغيل بشكل مباشر، وإن كان يمكن ترقيتها للحصول على شل.
الإجابة النموذجية: لأنها تتيح لمراجعي الثغرات تشغيل الهجمة الفوقية فوراً بمجرد فتح الملف بمتصفحهم، مما يوضح التأثير الحقيقي للهجمة ويسهل فحصها وتأكيدها.
الإجابة النموذجية: نبحث في الموقع عن أي واجهات أو سجلات عامة تسرب معرفات الـ UUID الخاصة بالمستخدمين الآخرين (مثل صفحة البروفايل العامة أو نتائج البحث)، ثم نستخدم المعرف المسرب في الطلب المستهدف بالـ IDOR للتأكد من الثغرة.
الإجابة النموذجية: تؤدي إلى توقف خادم الويب عن العمل بالكامل وحجب الخدمة عن المستخدمين (DoS)، نتيجة استهلاك معالج الذاكرة بالكامل لفك الكيانات المتداخلة المكررة.
