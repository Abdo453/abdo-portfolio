مختبر حالة السباق (Race Conditions)

Interactive Race Conditions Lab — 2 Levels

HIGH OWASP A04
AR

محاكي ثغرات السباق. قم بإرسال طلبات متزامنة في نفس اللحظة (Concurrently) لتخطي التحقق من الأرصدة أو القسائم المتاحة.

EN

Simulated Race Conditions lab. Send concurrent, parallel HTTP requests in the same millisecond to bypass server-side balance or usage checks.

Level: 1/2 Flags: 0/2 Progress: 0%
Wallet Balance: $50

Your current balance is $50. Target: Use the SAVE50 coupon multiple times concurrently to reach a balance of $250 or more.
[*] Hint L1: Increase the Thread Count (e.g., to 10 or 15) to fire parallel coupon redeem requests concurrently.

Theory / النظرية

AR

المستوى 1: إعادة استخدام كوبونات الخصم. إذا كان السيرفر يفحص حالة صلاحية الكود (is_redeemed) في قاعدة البيانات ثم يقوم بإضافته لاحقاً، فإن إرسال 10 طلبات متزامنة في نفس المللي ثانية قد تنجح جميعها قبل تحديث حالة الكود إلى "مستخدم".

EN

Level 1: Coupon Code Reuse. If the application checks if a coupon is redeemed, performs balance updates, and only then marks it as used, multiple threads can execute the check simultaneously, permitting multiple applications of a single-use coupon.

AR

المستوى 2: الإنفاق المزدوج المحاكي. عند تحويل الأموال، يفحص النظام أولاً: balance >= amount. إذا قمت بإرسال 5 طلبات تحويل بقيمة $40 متزامنة ولديك $50 فقط، فستنجح جميعها قبل خصم الأرصدة، مما يسبب سحب أموال أكثر من رصيدك الحقيقي.

EN

Level 2: Wallet Double Spend. During transfers, the API checks if balance is sufficient. Sending multiple parallel transfers concurrently might allow them to pass the check simultaneously before database debit updates are committed.
