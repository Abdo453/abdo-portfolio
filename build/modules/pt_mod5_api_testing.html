🔥 5. API Recon & Testing

اكتشاف واختبار واجهات برمجة التطبيقات (APIs) - استخراج وثائق API، واختبار GraphQL، واكتشاف ثغرات المصادقة والتخويل (BOLA)، إلى جانب حقن قواعد البيانات.

5.1 API Documentation Discovery

أول خطوة في اختبار واجهات برمجة التطبيقات هي العثور على التوثيق (Documentation) الخاص بها. توثيق الـ API مثل OpenAPI أو Swagger يكشف لنا جميع المسارات (Endpoints)، والمعلمات (Parameters)، والمدخلات المتوقعة، مما يسهل عملية الهجوم بشكل كبير.

OpenAPI/Swagger

Common paths to check 
/swagger.json
/swagger.yaml
/api/swagger.json
/v1/swagger.json
/openapi.json

يمكننا استخدام أدوات الفحص مثل ffuf للبحث عن مسارات التوثيق المخفية على الخادم المستهدف باستخدام قوائم مخصصة (Wordlists).

Using ffuf to find swagger docs 
ffuf -u https://target.com/FUZZ -w swagger_paths.txt -mc 200

Postman Collections

  • Look for .postman_collection.json files: ابحث عن ملفات مجموعات Postman التي غالباً ما تترك بشكل غير مقصود على الخوادم، والتي تحتوي على هيكل كامل للطلبات والمصادقات.
  • Check developer portals and documentation: تصفح بوابات المطورين الخاصة بالهدف، فقد تجد روابط مباشرة للتوثيق.
  • Search GitHub for leaked collections: ابحث في منصة GitHub عن أي تسريبات لهذه الملفات التي قد يكون المطورون قد قاموا برفعها بالخطأ في مستودعات عامة.

5.2 GraphQL Testing

تقنية GraphQL تختلف عن REST APIs، حيث تتيح للعميل تحديد هيكل البيانات المطلوبة بالضبط. هذا يوفر مرونة كبيرة لكنه يفتح باباً لثغرات خطيرة إذا لم يتم تأمينها، مثل استخراج هيكل البيانات الكامل (Introspection) أو استغلال اقتراحات الحقول.

Introspection Query

إذا كانت ميزة الـ Introspection مفعلة، يمكنك إرسال استعلام لمعرفة الهيكل الكامل لقاعدة البيانات وكل الحقول والأنواع المتاحة، وهو بمثابة خريطة كنز للمهاجم.

Introspection Query 
query {
  __schema {
    types {
      name
      fields {
        name
        type {
          name
        }
      }
    }
  }
}

Field Suggestion Abuse

في حال تم إيقاف ميزة الـ Introspection، قد يوفر خادم GraphQL ميزة "الاقتراحات". عند إرسال استعلام خاطئ، يقوم الخادم بالرد باقتراح حقول حقيقية موجودة (مثل: هل تقصد كذا؟). هذا يمكننا من تخمين الحقول المخفية.

Send malformed query to trigger suggestions 
query {
  system
}
# Response: "Did you mean 'pastes', 'paste', 'systemDebug'?"

Tools

  • InQL: أداة متخصصة لفحص واجهات GraphQL وتوليد الاستعلامات واستخراج هيكل البيانات.
  • GraphQL Voyager: أداة تستخدم لتمثيل الهيكل (Schema) الذي تم استخراجه في شكل رسومي مرئي يسهل فهمه.
  • Clairvoyance: أداة قوية لتوليد واستنتاج هيكل الـ Schema باستخدام تقنية الإكمال التلقائي (Field Suggestion) عندما تكون ميزة الـ Introspection مغلقة.

5.3 API Authentication Testing

اختبار عمليات المصادقة في الـ API للتأكد من عدم وجود ثغرات في طرق التحقق من هوية المستخدمين والتطبيقات، مثل تسريب التوكنز، ضعف المفاتيح، أو تخطي بروتوكولات المصادقة.

OAuth 2.0 Testing

  • Test for missing PKCE in mobile apps: التحقق من عدم استخدام بروتوكول PKCE في تطبيقات الجوال، مما يسهل على المهاجمين سرقة الـ Authorization Code.
  • Authorization code interception: محاولة اعتراض كود التفويض أثناء تبادله بين العميل والخادم واستخدامه لتوليد Access Token.
  • Token scope manipulation: التلاعب بنطاق الصلاحيات (Scopes) أثناء طلب التفويض لمحاولة الحصول على صلاحيات أعلى مما هو مسموح.
  • Refresh token security: فحص مدى أمان الـ Refresh Token وما إذا كان يمكن استخدامه بشكل غير محدود أو إعادة استخدامه بعد انتهاء صلاحيته.

JWT Testing

مفاتيح JWT (JSON Web Tokens) هي من أشهر طرق المصادقة. يتم اختبارها للبحث عن ثغرات في هيكلة الـ Token، أو استخدام خوارزميات ضعيفة، أو مفاتيح تشفير قابلة للكسر.

JWT Tool Tests 
# Decode JWT
jwt_tool.py <token>

# Test for algorithm confusion
jwt_tool.py -t <token> -X k

# Test for weak signing keys
jwt_tool.py -t <token> -C -d secrets.txt

API Key Testing

  • Key in URL parameters (should be in headers): البحث عن إرسال المفاتيح عبر مسار الرابط (URL Parameters) بدلاً من الترويسات (Headers)، مما يعرضها للحفظ في سجلات التصفح أو الخادم.
  • Missing key rotation: غياب سياسات تبديل وتحديث المفاتيح بشكل دوري، مما يجعل المفاتيح المسربة فعالة لفترة طويلة.
  • Keys committed to repositories: العثور على مفاتيح مسربة في مستودعات الكود مثل GitHub بسبب إهمال المطورين.

5.4 Authorization Testing (BOLA/BFLA)

اختبار التخويل (Authorization) يركز على التحقق من أن المستخدم لديه الصلاحيات للوصول إلى بيانات أو وظائف معينة. من أشهر الثغرات هنا BOLA (كسر صلاحيات الكائنات) و BFLA (كسر صلاحيات الوظائف).

Broken Object-Level Authorization (BOLA/IDOR)

تعتبر ثغرة BOLA (المعروفة أيضاً بـ IDOR) من أخطر الثغرات وأكثرها شيوعاً. تحدث عندما لا يتحقق الخادم مما إذا كان المستخدم يملك الصلاحية للوصول إلى كائن معين بناءً على معرفه (ID). يتم اختبارها بتغيير المعرفات (Object IDs) في الطلبات والتحقق مما إذا كنا نستطيع الوصول لبيانات مستخدم آخر.

Test BOLA by changing object IDs 
curl -H "Authorization: Bearer <token>" https://api.target.com/users/123/orders
curl -H "Authorization: Bearer <token>" https://api.target.com/users/124/orders  # Should fail

Authorization Matrix Testing

  • Create matrix of endpoints vs user roles: إنشاء جدول (Matrix) يقارن بين جميع المسارات (Endpoints) والأدوار المختلفة للمستخدمين (Admin, User, Guest).
  • Test each endpoint with each role: إرسال طلبات إلى كل مسار باستخدام صلاحيات لكل دور على حدة.
  • Document expected vs actual responses: توثيق الردود المتوقعة ومقارنتها بالردود الفعلية لاكتشاف أي مسارات غير محمية بصلاحيات كافية.

5.5 Rate Limiting & Abuse Testing

اختبار آليات تحديد عدد الطلبات (Rate Limiting) لمنع الهجمات المعتمدة على إرسال عدد كبير من الطلبات مثل البريوت فورس أو الـ DoS، وكذلك اختبار ثغرات السباق الزمني (Race Conditions).

Test rate limiting and Race conditions 
# Test rate limiting
for i in {1..100}; do
  curl -H "Authorization: Bearer <token>" https://api.target.com/endpoint
done

# Test for race conditions
# Send multiple simultaneous requests

نقوم بإرسال عدد كبير من الطلبات في وقت قصير (بواسطة حلقات تكرارية أو أدوات مثل Turbo Intruder) للتأكد من تفعيل أنظمة حماية الـ Rate Limit، ولمعرفة ما إذا كان بإمكاننا استغلال الـ Race Conditions عن طريق تنفيذ عمليات مالية أو إدارية في نفس اللحظة للحصول على مزايا غير مشروعة.

5.6 API Input Validation

تعتمد الـ APIs بشكل كبير على استقبال المدخلات بتنسيقات معينة (مثل JSON أو XML). عدم فحص أو تنظيف هذه المدخلات يمكن أن يؤدي إلى حقن تعليمات برمجية خبيثة.

Injection Testing

نقوم بتضمين حمولات خبيثة (Payloads) داخل حقول الـ API، سواء لاختبار حقن قواعد البيانات العادية (SQLi)، قواعد البيانات غير العلاقية (NoSQLi)، أو حتى حقن الأوامر في نظام التشغيل (Command Injection).

Various Injection payloads for APIs 
# SQL Injection in API parameters
curl -X POST https://api.target.com/search -d '{"query":"test'\'' OR '\''1'\''='\''1"}'

# NoSQL Injection
curl -X POST https://api.target.com/search -d '{"query":{"$gt":""}}'

# Command Injection
curl -X POST https://api.target.com/generate -d '{"template":"test;whoami"}'

5.7 API Testing Tools

استخدام الأدوات المناسبة يسرع من عملية استكشاف الـ API واختباره وتوثيق الطلبات بكفاءة عالية.

Postman & Insomnia

  • Postman: أداة غنية جداً، تتيح استيراد ملفات OpenAPI، إنشاء مجموعات من الطلبات (Collections)، وعمل اختبارات تلقائية باستخدام أداة Newman.
  • Insomnia: بديل خفيف وسريع لـ Postman، ويُفضل كثيراً في اختبارات الـ GraphQL بسبب واجهته الممتازة في التعامل مع هذا النوع من الـ APIs.

Command Line Tools (HTTPie & Curl)

للعمل من سطر الأوامر بسرعة وفاعلية أثناء بناء السكربتات (Scripts)، تُعتبر HTTPie و Curl خيارات مثالية لاختبار المسارات وتوثيق النتائج.

HTTPie and Curl Examples 
# HTTPie
http GET api.target.com/users Authorization:"Bearer <token>"

# Curl
curl -X POST https://api.target.com/endpoint \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer <token>" \
     -d '{"key":"value"}'
