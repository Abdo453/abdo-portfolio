☁️ 6. Cloud & Infrastructure Recon

استكشاف البنية التحتية السحابية وجمع المعلومات عن الخوادم ونطاقات الشبكة والبحث عن التسريبات

6.1 ASN & IP Range Enumeration

عملية استخراج واستكشاف الـ ASN (Autonomous System Number) والنطاقات الخاصة بعناوين الـ IP التابعة للهدف. الـ ASN هو رقم مميز يُعطى للشركات الكبيرة لتنظيم مسارات البيانات الخاصة بها على الإنترنت. بمعرفة هذا الرقم، يمكنك اكتشاف العديد من نطاقات الـ IP الخاصة بالهدف.

ASN Lookup باستخدام whois 
whois -h whois.radb.net -- '-i origin AS12345'

أداة whois هنا تستعلم من قاعدة بيانات RADb عن نطاقات الـ IP المُسجلة تحت الـ ASN المحدد (AS12345).

استخراج الـ ASN باستخدام Amass 
amass intel -asn 12345

تقوم أداة Amass القوية بجمع معلومات استخباراتية تفصيلية بناءً على الـ ASN وتساعد في اكتشاف المزيد من الأصول (Assets) المرتبطة بالهدف.

اكتشاف الـ ASN عبر asnmap 
echo target.com | asnmap

أداة asnmap من ProjectDiscovery توفر طريقة سريعة جداً لمعرفة أرقام الـ ASN المرتبطة باسم النطاق (Domain) المعطى.

فحص نطاقات الـ IP باستخدام Nmap 
nmap -sn 192.168.1.0/24

استخدام Nmap مع الخيار -sn (Ping Scan) لاكتشاف الأجهزة الحية ضمن نطاق الـ IP المستهدف دون إجراء فحص تفصيلي للمنافذ (Port Scan)، مما يسرع عملية المسح الأولية.

6.2 Cloud Storage Enumeration

عملية البحث عن وحدات التخزين السحابية (Cloud Storage) التابعة للهدف والتي قد تحتوي على بيانات حساسة، ملفات نسخ احتياطي، أو كود مصدري تم تعريضه للعامة بسبب إعدادات صلاحيات خاطئة (Misconfigurations).

AWS S3

فحص S3 Buckets بأداة S3Scanner 
s3scanner -b buckets.txt

تقوم الأداة بفحص قائمة من أسماء الحاويات (Buckets) المخزنة في الملف buckets.txt لاكتشاف أي حاويات مفتوحة أو قابلة للوصول بدون تصريح.

استكشاف S3 باستخدام Ffuf 
ffuf -u https://FUZZ.s3.amazonaws.com -w bucket_names.txt -mc 200,403

استخدام أداة ffuf للقيام بعملية تخمين (Fuzzing) على أسماء الحاويات. وتصفية النتائج لتظهر فقط الصفحات التي ترد بكود 200 (الحاوية موجودة ومفتوحة) أو 403 (الحاوية موجودة ولكن محمية بكلمة مرور/صلاحيات).

استعراض محتويات S3 عبر AWS CLI 
aws s3 ls s3://bucket-name
aws s3 ls s3://bucket-name --recursive

في حال حصولك على صلاحيات (Credentials) مسربة، يمكنك استخدام أداة AWS CLI الرسمية لعرض محتويات الحاوية. الخيار --recursive يقوم بسرد كافة الملفات والمجلدات الداخلية بشكل متكرر وشامل.

أنماط تسمية شائعة لحاويات S3:
<company>-assets
<company>-backup
<company>-dev
<company>-staging
<company>-production

Azure Blob Storage

استكشاف Azure Blob باستخدام Ffuf 
ffuf -u https://<company>.blob.core.windows.net/FUZZ -w container_names.txt -mc 200,403

بحث مخصص لوحدات التخزين الخاصة بخدمات Microsoft Azure لاكتشاف الحاويات (Containers) المتاحة للعامة عن طريق التخمين.

Google Cloud Storage

استكشاف GCS باستخدام Ffuf 
ffuf -u https://storage.googleapis.com/FUZZ -w bucket_names.txt -mc 200,403

تطبيق نفس أسلوب التخمين للبحث عن حاويات التخزين المفتوحة على بيئة Google Cloud Storage الخاصة بالشركة المستهدفة.

6.3 GitHub/GitLab Recon

البحث في المستودعات (Repositories) الخاصة بالمنظمة على منصات إدارة الأكواد مثل GitHub و GitLab. الهدف هنا هو إيجاد أكواد برمجية، مفاتيح سرية (API Keys/Secrets)، أو كلمات مرور تم رفعها بالخطأ أو بسبب إهمال المطورين.

البحث المتقدم باستخدام GitDorker 
python3 GitDorker.py -tf TOKENSFILE -q "target.com" -d dorks.txt -o output.txt

أداة GitDorker تقوم بأتمتة البحث باستخدام قائمة من عبارات البحث المخصصة (Dorks) لاستخراج المعلومات الحساسة من GitHub. استخدام ملف الـ Tokens يتيح لك تخطي حظر البحث السريع الذي تفرضه المنصة.

فحص التسريبات السرية باستخدام TruffleHog 
trufflehog git https://github.com/org/repo.git
trufflehog filesystem /path/to/code

أداة TruffleHog تفحص المستودعات بدقة للبحث عن الكلمات السرية ومفاتيح التشفير التي ربما تكون مخفية في تاريخ الالتزامات البرمجية (Commit History). يمكن تشغيلها على مستودع بعيد عبر رابط الـ git أو على مجلد محلي تم تحميله مسبقاً.

اكتشاف المفاتيح المسربة عبر Gitleaks 
gitleaks detect -v --source /path/to/repo

أداة Gitleaks متخصصة وسريعة جداً في العثور على الأسرار الثابتة (Hardcoded secrets) كرموز الوصول (Access Tokens) والمفاتيح في الكود المصدري لمنع تسرب البيانات الحساسة.

البحث اليدوي باستخدام GitHub Dorks 
org:targetcompany "password"
org:targetcompany "api_key"
org:targetcompany "secret"
org:targetcompany "AWS_ACCESS_KEY_ID"
org:targetcompany "private_key"
filename:config.json org:targetcompany
filename:.env org:targetcompany

استخدام محرك بحث GitHub يدوياً عبر عوامل التصفية (Filters) لاستهداف منظمة معينة (org:) والبحث عن كلمات دلالية تدل على وجود بيانات حساسة، أو استهداف ملفات معينة باستخدام filename: كملفات البيئة .env أو ملفات الإعدادات config.json التي غالباً ما تحتوي على كلمات مرور ومفاتيح قواعد البيانات.

6.4 Cloud Metadata Service Testing

استغلال خدمات الـ Metadata السحابية. في بيئة السحابة، توجد عناوين IP داخلية مخصصة (مثل 169.254.169.254) تزود الخادم الافتراضي (Instance) بمعلومات عن نفسه. في حال اكتشاف ثغرة مثل SSRF (تزوير الطلبات من جهة الخادم) يمكنك استدعاء هذه العناوين لسحب معلومات حساسة ومفاتيح دخول.

AWS Metadata (IMDSv1 - vulnerable)

استرجاع Metadata من AWS 
curl http://169.254.169.254/latest/meta-data/
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

النسخة الأولى من خدمة IMDS في AWS لا تتطلب توثيقاً صارماً، ويمكن استغلالها بسهولة لجلب بيانات وأسماء الأدوار (IAM roles) الخاصة بالخادم والتي تؤدي للحصول على مفاتيح وصول مؤقتة وصلاحيات أعلى.

AWS Metadata (IMDSv2 - requires token)

توليد Token واسترجاع Metadata 
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/

النسخة الثانية IMDSv2 تعتبر أكثر أماناً لأنها تتطلب إرسال طلب PUT للحصول على رمز مؤقت (Token) أولاً بمدة صلاحية محددة، ثم استخدام هذا الرمز في رأسيات (Headers) الطلبات اللاحقة للوصول إلى الـ Metadata.

Azure Metadata

استرجاع Metadata من Azure 
curl "http://169.254.169.254/metadata/instance?api-version=2021-02-01" -H "Metadata: true"

في خوادم Azure، لابد من إرسال ترويسة محددة وهي Metadata: true لتجنب الحماية وضمان استجابة الخادم الداخلي وتزويدك بالمعلومات.

GCP Metadata

استرجاع Metadata من Google Cloud 
curl "http://metadata.google.internal/computeMetadata/v1/instance/" -H "Metadata-Flavor: Google"

بالمثل في Google Cloud، يتطلب الاستعلام إرسال الترويسة Metadata-Flavor: Google كإجراء أمني لمنع هجمات SSRF البسيطة التي لا تستطيع إضافة ترويسات مخصصة.

6.5 CDN & WAF Bypass

تخطي شبكات توصيل المحتوى (CDN) وجدران الحماية للويب (WAF) للوصول المباشر لخادم الاستضافة الفعلي (Origin IP). معرفة الـ IP الحقيقي يمكنك من تجاوز الحمايات الموضوعة والوصول إلى خدمات غير مخصصة للعامة وتنفيذ هجمات مباشرة على الخادم.

اكتشاف استخدام CDN 
# Check if behind CDN
whois target.com | grep -i "cloudflare\|akamai\|fastly\|cloudfront"

# Find origin IP
# 1. Check historical DNS records
# 2. Search Shodan for SSL certificates matching target
# 3. Check for DNS records not proxied through CDN

يمكن استخدام الاستعلام whois لمعرفة ما إذا كان الدومين يخفي عنوانه الحقيقي خلف مزودات CDN مشهورة مثل Cloudflare أو Akamai. وتوضح التعليقات طرقاً أخرى لاكتشاف الـ IP الحقيقي كالتحقق من سجلات DNS التاريخية.

اكتشاف عنوان الـ IP الأصلي (Origin IP) 
# Using Shodan
shodan search ssl:"target.com"

# Using Censys
censys search "target.com"

# Check mail server IPs (often not behind CDN)
dig MX target.com

استخدام محركات البحث المتخصصة مثل Shodan أو Censys للبحث عن الشهادات الأمنية (SSL) المرتبطة بالنطاق قد يكشف عن الـ IP الأصلي المنسي بدون حماية. كذلك يمكن الاستعلام عن خوادم البريد الإلكتروني (MX) التي نادراً ما يتم حمايتها عبر CDN وغالباً ما تكشف عنوان الـ IP الفعلي.

6.6 Infrastructure Recon Pipeline

مسار تسلسلي (Pipeline) يجمع الخطوات السابقة في آلية واحدة لاستكشاف البنية التحتية بالكامل وتحديد النقاط الضعيفة والأصول المتصلة بشبكة الهدف.

التسلسل الزمني لعملية الاستكشاف 
# Step 1: ASN enumeration
echo target.com | asnmap > asn_info.txt

# Step 2: IP range discovery
amass intel -org "Target Company" -o ip_ranges.txt

# Step 3: Cloud storage enumeration
ffuf -u https://FUZZ.s3.amazonaws.com -w s3_buckets.txt -mc 200,403 -o s3_results.txt

# Step 4: GitHub recon
gitdorker -tf tokens -q "target.com" -d dorks.txt -o github_leaks.txt

# Step 5: Metadata service testing (if SSRF found)
# Test 169.254.169.254 endpoints through SSRF vectors

يوضح هذا المسار الترتيب المنطقي للعمليات:
الخطوة 1: استخراج الـ ASN الخاص بالهدف لمعرفة البنية العامة.
الخطوة 2: اكتشاف نطاقات عناوين الـ IP التابعة للمنظمة بناءً على اسم الشركة.
الخطوة 3: فحص وبحث مستودعات التخزين السحابية (S3) المفتوحة وتسجيل النتائج.
الخطوة 4: البحث المكثف داخل GitHub لاكتشاف أي بيانات أو مفاتيح مسربة.
الخطوة 5: اختبار خدمات الـ Metadata في حال اكتشاف ثغرة SSRF لمحاولة السيطرة على الخوادم الداخلية واستخراج بيانات حساسة.
