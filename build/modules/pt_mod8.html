08. API Testing

استكشاف ثغرات الواجهات البرمجية (REST و GraphQL) والتحكم بالصلاحيات (BOLA, BFLA)

[Theory] الجانب النظري

الواجهات البرمجية (APIs) هي عصب التطبيقات الحديثة. تنقسم بشكل أساسي إلى REST و GraphQL. التحديات الأمنية فيها تتمحور حول آليات التفويض (Authorization) والمصادقة (Authentication). من أخطر الثغرات وأكثرها شيوعاً هي Broken Object Level Authorization (BOLA/IDOR) حيث يمكن للمهاجم الوصول إلى بيانات مستخدمين آخرين بتغيير معرف الكائن (Object ID)، وثغرة Broken Function Level Authorization (BFLA) حيث يتم الوصول إلى وظائف إدارية بامتيازات مستخدم عادي. وفي بيئة GraphQL يزداد الهجوم تعقيداً حيث يمكن استخدام هجمات Introspection لاستخراج المخطط (Schema) أو تنفيذ هجمات استعلامات معقدة (Nested Queries) للتسبب في حرمان من الخدمة (DoS).

[Manual Testing] الفحص اليدوي

  • تحليل الاستجابات والطلبات: استخدم Burp Suite أو Postman لفهم تركيبة الطلب وهيكل البيانات، ولاحظ الرؤوس (Headers) الخاصة بالتوثيق (مثل JWT).
  • اختبار BOLA (IDOR): قم بإنشاء حسابين (User A و User B). التقط الطلب الموجه بامتيازات User A وقم بتغيير المعرف (ID) في المسار أو جسم الطلب إلى معرف User B ولاحظ ما إذا كان النظام يقبل التعديل.
  • اختبار BFLA: جرب تغيير طريقة الطلب (HTTP Method) من GET إلى POST، PUT، أو DELETE للوصول إلى الوظائف الممنوعة. حاول استدعاء مسارات إدارية مثل /api/admin/users باستخدام Token خاص بمستخدم عادي.
  • فحص GraphQL: أرسل استعلام { __schema { types { name } } } لمعرفة ما إذا كان Introspection مفعلاً. جرب استعلامات دائرية متداخلة لاختبار ثغرات DoS بسبب استهلاك الموارد.

[Automation] أدوات الأتمتة والسكربتات

يمكن استخدام أدوات متخصصة لاكتشاف المسارات وفحص الصلاحيات بشكل مؤتمت:

  • Kiterunner: أداة قوية لاكتشاف مسارات واجهات برمجة التطبيقات (API Endpoints).
  • ffuf / wfuzz: أتمتة عملية الـ Fuzzing لاكتشاف المسارات المخفية ومعرفات الكائنات.
  • Autorize: إضافة رائعة لـ Burp Suite تقوم بأتمتة اختبارات BOLA و BFLA عن طريق إعادة إرسال الطلبات بصلاحيات مستخدم آخر أو بدون صلاحيات نهائياً.
  • InQL / Clairvoyance: أدوات متخصصة لتحليل وتخمين GraphQL Schemas.

[Tools] الأوامر والأدوات التطبيقية

استخدام Kiterunner لاكتشاف مسارات API:

$ kr scan https://api.target.com/ -w /opt/wordlists/routes-large.kite

استخدام ffuf للبحث عن إصدارات الـ API ومساراتها:

$ ffuf -w /path/to/wordlist.txt -u https://target.com/api/v1/FUZZ -mc 200,401,403

تصدير مخطط GraphQL باستخدام InQL (Burp Extension) أو سكريبت بايثون:

$ inql -t https://target.com/graphql

[Lab] بيئات التدريب والتجارب

  • PortSwigger Web Security Academy: مختبرات مخصصة لثغرات API و JWT و BOLA/IDOR.
  • crAPI (Completely Ridiculous API): تطبيق مفتوح المصدر مصمم خصيصاً ليحتوي على أهم ثغرات الـ API (OWASP API Top 10).
  • VAmPI (Vulnerable API): واجهة برمجية مصابة بثغرات متعددة للتدرب على اكتشاف واستغلال أخطاء الـ API.
  • HackTheBox / TryHackMe: آلات تركز على REST APIs و GraphQL واختراقاتها المعقدة.

[Quiz] اختبار المعرفة

Scenario: أنت تختبر تطبيقاً يستخدم REST API. التطبيق يسمح للمستخدم بتحديث ملفه الشخصي عبر طلب PUT إلى `/api/v1/user/update`. عندما حاولت تغيير المعرف في جسم الطلب `{"user_id": 1234, "email": "hacker@evil.com"}` إلى معرف مستخدم آخر، تم تغيير بريد ذلك المستخدم بنجاح. ما هو التصنيف الصحيح لهذه الثغرة؟

[Cheat Sheet] المرجع السريع

Vulnerability / Task Description / Approach
BOLA (IDOR) تلاعب بمعرفات (IDs) المستخدمين أو الكائنات في الـ URL، أو الهيدرز، أو جسم الطلب (JSON).
BFLA الوصول إلى وظائف أعلى (كالإدارة) باستخدام Token مستخدم عادي. جرب تغيير HTTP Methods.
Mass Assignment إضافة حقول غير متوقعة في الـ JSON (مثل `"is_admin": true`) لمعرفة ما إذا كان الـ API سيقبلها.
GraphQL Introspection استخدام استعلام __schema أو __type لاكتشاف كامل الـ Schema والوظائف المخفية.
API Discovery استخدام أدوات مثل Kiterunner و FFUF للبحث عن Endpoints مخفية و Swagger/OpenAPI files (/swagger.json).
