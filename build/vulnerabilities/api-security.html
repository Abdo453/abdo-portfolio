أمان واجهات برمجة التطبيقات (API Security) — الدليل الشامل

API Security & OWASP API Top 10 — The Ultimate Deep Dive

CRITICAL OWASP API Security Top 10
AR

واجهات برمجة التطبيقات (APIs) هي العصب الذي يربط تطبيقات الويب والموبايل بقواعد البيانات والخدمات الخلفية (Backend). نظراً لأن الـ APIs مصممة للتعامل المباشر مع البيانات (أحياناً بدون واجهة مستخدم رسومية)، فإن أي خطأ في التحكم في الوصول (Access Control) أو التحقق من المدخلات (Input Validation) يؤدي إلى تسريب ضخم للبيانات. يركز أمان הـ API على تأمين نقاط النهاية (Endpoints) من الهجمات المباشرة مثل BOLA/IDOR والتلاعب بالبيانات الجماعية (Mass Assignment).

EN

API Security focuses on protecting Application Programming Interfaces from misuse. Unlike traditional web apps, APIs often expose underlying data structures directly to the client. This architectural difference shifts the security focus towards deeply nested authorization flaws (like BOLA), excessive data exposure, and robust rate limiting to prevent automated attacks.

أهم ثغرات الـ API (OWASP API Top 10)

1. API1:2023 - Broken Object Level Authorization (BOLA)

المعروفة سابقاً بـ IDOR. تحدث عندما يتمكن المستخدم من الوصول إلى كائنات (Objects) تخص مستخدمين آخرين بمجرد تغيير الـ ID في مسار הـ API (مثال: GET /api/receipts/105 بدلاً من 104).

2. API2:2023 - Broken Authentication

ضعف أو غياب حماية تسجيل الدخول. مثل عدم وجود Rate Limiting على /api/login مما يسمح بهجمات Credential Stuffing، أو ضعف في التحقق من صحة الـ JWT، أو تعريض كلمات المرور في مسار الـ URL بدلاً من الـ Body.

3. API3:2023 - Broken Object Property Level Authorization

تجمع بين Excessive Data Exposure و Mass Assignment. الخادم يعيد بيانات سرية لا يحتاجها العميل (مثل الـ Password Hash)، أو يسمح للمهاجم بتعديل خصائص لم يكن مسموحاً له بتعديلها (مثل تمرير {"is_admin": true} في طلب تحديث الملف الشخصي).

4. API4:2023 - Unrestricted Resource Consumption

عدم وجود قيود على معدل الطلبات (Rate Limiting) أو حجم البيانات. יכול المهاجم إرسال آلاف الطلبات لاستهلاك موارد السيرفر (CPU/Memory)، أو تحميل ملفات ضخمة، مما يؤدي إلى Denial of Service (DoS).

تقنيات البحث والاستغلال المتقدمة / Methodology

AR

خطوات اختبار أي API:

  • Discovery & Recon: البحث عن الـ API Documentation (مثل Swagger/OpenAPI على /api-docs أو /swagger.json).
  • Versioning Bypasses: إذا كان /api/v2/users محمياً، جرب /api/v1/users (غالبًا تكون النسخ القديمة مصابة بـ BOLA).
  • Method Fuzzing: إذا كان GET /api/users/1 ممنوعاً، جرب POST، PUT، أو حتى DELETE.
  • Content-Type Manipulation: تغيير application/json إلى application/xml لاختبار ثغرات XXE في الـ API.
  • Mass Assignment Check: في أي طلب PUT/POST لتحديث البيانات، قم بإضافة حقول إضافية مثل "role":"admin" أو "credit":9999 وراقب الرد.

أدوات فحص وتأمين الـ APIs

Postman / Insomnia Kiterunner ffuf (API Fuzzing) Autorize (Burp Ext)

أداة Kiterunner ممتازة لاكتشاف مسارات الـ API المخفية (Endpoint Discovery). أداة Autorize في Burp Suite تعتبر ساحرة لاكتشاف BOLA؛ تقوم بتشغيل المتصفح كمستخدم "أدمن" أو "عادي"، والإضافة تعيد إرسال كل طلباتك تلقائياً باستخدام توكن (Token) مستخدم عادي لتكتشف ما إذا كانت الصلاحيات مخترقة أم لا.

كيفية الحماية (Remediation)

  • Zero Trust: لا تثق أبداً بالمعرفات (IDs) التي يرسلها العميل. تحقق دائماً في قاعدة البيانات: "هل المستخدم صاحب هذا التوكن (Token) لديه صلاحية الوصول للـ ID المطلوب؟"
  • Strict Schema Validation: حدد بوضوح وبصرامة الحقول التي يُسمح للمستخدم بإرسالها (لمنع Mass Assignment) والحقول التي يمكن إرجاعها (لمنع Data Exposure).
  • Rate Limiting & Throttling: تطبيق قيود صارمة على عدد الطلبات بناءً على الـ IP وعلى الـ User Token لمنع الـ Brute-Force والـ DoS.
