اختراق واجهات GraphQL — الدليل الشامل

GraphQL Security & Injection — The Ultimate Deep Dive

HIGH OWASP API Security Top 10
AR

GraphQL هي لغة استعلام لواجهات برمجة التطبيقات (APIs) تتيح للعميل طلب البيانات التي يحتاجها بدقة (لا أكثر ولا أقل). على عكس REST API التي تعتمد على مسارات متعددة (Endpoints)، تستخدم GraphQL مسارًا واحدًا (غالباً /graphql). مرونة GraphQL وقدرتها على ربط البيانات المعقدة تجعلها هدفاً مفضلاً للمهاجمين للبحث عن ثغرات التحكم في الوصول (IDOR/BOLA)، الاستعلامات المتداخلة (Batching)، وتخطي الحماية عبر الحقول المخفية (Introspection).

EN

GraphQL gives clients the power to ask for exactly what they need and nothing more. However, this flexibility introduces unique security challenges. Attackers often abuse features like Introspection to map the entire API schema, query batching for brute-forcing, and nested queries to cause Denial of Service (DoS) or bypass authorization controls.

مفاهيم GraphQL الأساسية للمخترقين

  • Queries: تُستخدم لجلب البيانات (مثل GET في REST).
  • Mutations: تُستخدم لتعديل أو إضافة بيانات (مثل POST/PUT/DELETE في REST).
  • Introspection: نظام مدمج في GraphQL يسمح لك بسؤال الـ API عن جميع الاستعلامات (Queries) والتعديلات (Mutations) المتاحة، وأنواع البيانات. (بمثابة خريطة كنز للمهاجم!).
  • Fragments: أجزاء قابلة لإعادة الاستخدام داخل الاستعلام.

الثغرات الشائعة في GraphQL (Attack Vectors)

1. تفعيل Introspection (كشف الهيكل)

الخطوة الأولى في أي تقييم هي محاولة سحب "مخطط البيانات" (Schema). إذا كانت الاستبطان (Introspection) مفعلة، يمكنك إرسال استعلام __schema لمعرفة كل شيء:

{ __schema { types { name fields { name } } } }

حتى لو كانت مقفلة، يمكن تخطيها أحياناً بتبديل الحروف (مثلاً __Schema) أو الاعتماد على رسائل الخطأ (Suggestions) مثل "Did you mean ...?" لاستنتاج الحقول المتاحة.

2. ثغرات IDOR / BOLA في GraphQL

نظراً لأن العميل يحدد ما يريده، قد ينسى المطور وضع صلاحيات على الحقول الفرعية. مثال:

query { user(id: 5) { username, passwordHash, ssn } }

بينما قد يكون مسموحاً لك بجلب الـ username، هل يُسمح لك بجلب الـ ssn للمستخدم 5؟ غالباً ما يكون التحكم في الوصول معيباً في الجذور العميقة.

3. Query Batching & Brute-Force

تدعم GraphQL إرسال مصفوفة من الاستعلامات في طلب HTTP واحد. يمكن استغلال هذا لعمل Brute-Force لكلمات المرور أو الـ OTP (كود التحقق) دون إطلاق تنبيهات جدار الحماية (WAF Rate Limiting) لأنك ترسل طلب HTTP واحد يحتوي على 10,000 استعلام!

4. هجمات الحرمان من الخدمة (DoS & Nested Queries)

بفضل العلاقات المتداخلة، يمكنك طلب بيانات تشكل حلقة لا نهائية تؤدي لإسقاط الخادم (Server Crash):

{ author { posts { author { posts { author ... } } } } }

أدوات الأتمتة واختبار GraphQL

InQL (Burp Ext) Clairvoyance GraphQL Voyager

إضافة InQL في Burp Suite هي الخيار الأقوى لتحليل الـ Introspection واستخراج كافة الـ Queries لتجربتها. إذا كان الـ Introspection مغلقاً، استخدم أداة Clairvoyance التي تعتمد على التخمين ورسائل الخطأ لبناء الـ Schema.

كيفية الحماية (Remediation)

  • تعطيل Introspection في بيئة الإنتاج (Production).
  • إيقاف ميزة الاقتراحات (Field Suggestions) لمنع تخمين الحقول (Clairvoyance Attack).
  • تطبيق حدود للعمق (Query Depth Limit) والتعقيد (Query Complexity Limit) لمنع الـ DoS.
  • تعطيل أو تقييد الـ Query Batching لمنع الـ Brute-Force الخفي.
  • أهم نقطة: تطبيق التحقق من الصلاحيات (Authorization) على مستوى كل حقل (Field Level) وليس فقط على مستوى الاستعلام الأساسي (Query Level).
