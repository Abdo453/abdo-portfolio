أمان توكنات JWT — الدليل الشامل

JSON Web Token (JWT) Security — The Ultimate Deep Dive

HIGH OWASP A01:2021 / A05:2021
AR

توكنات JWT (JSON Web Tokens) تُستخدم على نطاق واسع في واجهات برمجة التطبيقات الحديثة (APIs) لتخزين بيانات جلسة المستخدم وهويته دون الحاجة لحفظها في قاعدة بيانات (Stateless). تتكون من ثلاثة أجزاء مفصولة بنقاط (Header.Payload.Signature). تكمن الخطورة في كونها تُفك تشفيرها بسهولة (Base64 URL)، وإذا كان التوقيع (Signature) ضعيفاً أو أخطأ المطور في التحقق منه، يمكن للمهاجم تزوير التوكن والصعود بصلاحياته (Privilege Escalation).

EN

JSON Web Tokens (JWT) are a stateless standard used for authentication and authorization in modern web applications. Because the payload is only Base64URL encoded (not encrypted), it is readable by anyone. Security vulnerabilities arise when the cryptographic signature verification is flawed, allowing attackers to forge tokens and impersonate arbitrary users or admins.

آلية العمل والسبب الجذري / Root Cause Analysis

يتكون الـ JWT من:

  • Header: يحتوي على نوع التوكن وخوارزمية التوقيع المستخدمة (مثل {"alg": "HS256"}).
  • Payload: يحتوي على بيانات المستخدم (الـ Claims) مثل {"user_id": 1, "role": "user"}.
  • Signature: التوقيع المشفر الذي يضمن عدم تعديل الـ Payload. يعتمد على سر (Secret) لا يملكه إلا الخادم.

إذا أراد المهاجم تغيير دوره من user إلى admin، يجب عليه إعادة حساب التوقيع باستخدام الـ Secret الأصلي، وإلا سيرفض الخادم التوكن. الثغرات تظهر عندما يفشل الخادم في التحقق بشكل صحيح.

أنواع ثغرات الـ JWT وهجماتها / Attack Vectors

1. None Algorithm (alg: none)

بعض مكتبات JWT القديمة تدعم خوارزمية "none"، مما يعني أن التوكن لا يحتاج إلى توقيع! يقوم المهاجم بتغيير alg إلى none وإزالة قسم التوقيع بالكامل، ويقبله الخادم كتوكن صحيح.

2. Weak Secret Cracking (Brute-Force)

إذا كانت الخوارزمية هي HS256 (Symmetric Key)، فهذا يعني أن السر هو مجرد نص. إذا كان السر ضعيفاً (مثل "secret" أو "12345")، يمكن للمهاجم كسره أوفلاين باستخدام أدوات مثل Hashcat أو John the Ripper.

3. Key Confusion (RS256 to HS256)

تستخدم RS256 مفتاحين (Public Key للتحقق و Private Key للتوقيع). إذا كان المهاجم يمتلك الـ Public Key (غالباً متاح علنًا)، يمكنه تغيير الخوارزمية في الـ Header إلى HS256. إذا كان الخادم مصاباً، سيستخدم الـ Public Key كسِر (Secret) في خوارزمية HS256 للتحقق من التوقيع! وبما أن المهاجم يملك الـ Public Key، يمكنه توقيع التوكن الخاص به.

4. JKU / JWK Header Injection

قد يعتمد الخادم على روابط أو مفاتيح مدمجة في الهيدر (مثل jku أو jwk) لجلب المفتاح العام. يستطيع المهاجم تمرير رابطه الخاص لخدمة استضافة (مثل خادمه الخاص) تحتوي على مفتاحه العام، وبالتالي توقيع التوكن بمفتاحه الخاص بنجاح.

تقنيات البحث والاستغلال المتقدمة / Exploitation Workflow

AR

طريقة العمل (Methodology) لاختبار JWT:

  • فك التشفير: قم بنسخ التوكن ولصقه في jwt.io لترى محتوى הـ Payload والـ Header.
  • اختبار الثقة (Trust Test): غير قيمة داخل הـ Payload (مثل role: admin) دون تغيير التوقيع وأرسله. إذا تم قبوله، فالتطبيق لا يتحقق من التوقيع أصلاً!
  • تخطي الخوارزمية (None): جرب تغيير الـ alg إلى None، none، NONE وأزل الجزء الأخير من التوكن مبقياً على النقطتين ...
  • كسر التشفير (Cracking): استخدم hashcat -a 0 -m 16500 jwt.txt wordlist.txt لمحاولة كسر سر الـ HS256.

أدوات الأتمتة (Automation Tools)

jwt_tool (Python) JSON Web Tokens (Burp Ext) Hashcat (mode 16500)

أداة jwt_tool هي أداة رائعة يمكنها فحص كل أنواع الهجمات بشكل آلي (None, Cracking, Key Confusion):

python3 jwt_tool.py "YOUR.JWT.TOKEN" -M pb

كيفية الحماية (Remediation)

  • لا تثق بقيمة alg الموجودة في الهيدر. قم بإجبار الخادم على استخدام خوارزمية واحدة محددة سلفاً (مثل RS256 حصراً).
  • استخدم كلمات مرور قوية جداً (Secrets) لخوارزميات HS256 لضمان عدم تعرضها للكسر (Brute-Force).
  • تجنب الاعتماد على الهيدرات مثل jku, jwk, kid لجلب مفاتيح غير موثوقة أو تحقق بدقة شديدة من النطاقات المسموح بها (Whitelisting).
