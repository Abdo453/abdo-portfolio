حقن قواعد البيانات (SQL Injection) — الدليل الشامل

SQL Injection (SQLi) — The Ultimate Deep Dive

CRITICAL OWASP A03:2021
AR

ثغرة حقن قواعد البيانات (SQLi) هي من أخطر وأقدم ثغرات الويب. تحدث عندما يقوم التطبيق بأخذ مدخلات المستخدم (User Input) ويمررها مباشرة داخل استعلام SQL (SQL Query) في الخلفية دون إجراء عمليات تنظيف (Sanitization) أو استخدام (Parameterized Queries). هذا يسمح للمهاجم بالتلاعب بالاستعلام الأصلي ليقوم بقراءة بيانات حساسة، تعديلها، حذفها، أو حتى تنفيذ أوامر على مستوى نظام التشغيل (RCE).

EN

SQL Injection (SQLi) is a critical vulnerability that occurs when an application improperly concatenates user-supplied data into a backend database query. This allows an attacker to manipulate the query logic, leading to unauthorized data access, modification, or even Remote Code Execution (RCE) via functions like xp_cmdshell.

آلية العمل والسبب الجذري / Root Cause Analysis

تخيل أن التطبيق يستعلم عن المستخدم هكذا:

SELECT * FROM users WHERE username = '$user_input' AND password = '$password_input'

إذا أدخل المهاجم في حقل الـ username القيمة admin' -- ، سيصبح الاستعلام كالتالي:

SELECT * FROM users WHERE username = 'admin' -- ' AND password = '...'

علامة -- تعني (تعليق) في SQL، مما يلغي شرط الباسورد بالكامل، ويدخل المهاجم بحساب الأدمن بدون معرفة كلمة المرور.

أنواع ثغرات الـ SQL Injection (التصنيف العميق)

1. In-band SQLi (Classic)

يتم إرسال الهجوم واستقبال النتيجة عبر نفس القناة (نفس صفحة الويب). وهو الأسهل في الاكتشاف والاستغلال.

  • Error-based: إجبار قاعدة البيانات على إظهار رسائل خطأ برمجية تحتوي على بيانات حساسة (مثال: استخدام دالة extractvalue() في MySQL).
  • Union-based: استخدام معامل UNION لدمج نتائج استعلام المهاجم مع الاستعلام الأصلي، مما يظهر البيانات المسروقة مباشرة في الصفحة.

2. Inferential (Blind) SQLi

البيانات لا تظهر في الصفحة. المهاجم يعتمد على مراقبة سلوك التطبيق لاستنتاج البيانات (حرفاً بحرف).

  • Boolean-based: إرسال شروط (True/False). إذا تغير محتوى الصفحة أو طولها، يستنتج المهاجم أن الشرط صحيح.
  • Time-based: إذا كان التطبيق لا يغير المحتوى إطلاقاً، يستخدم المهاجم دوال مثل SLEEP(10) أو pg_sleep(10). إذا تأخر الرد 10 ثوانٍ، فالشرط صحيح.

3. Out-of-Band (OOB) SQLi

عندما يكون التطبيق محمي بشكل كامل ولا يظهر أخطاء ولا يتأثر بالوقت. يقوم المهاجم بإجبار قاعدة البيانات على إرسال البيانات (DNS Request أو HTTP Request) إلى سيرفر يمتلكه المهاجم (Burp Collaborator). يعتمد على دوال مثل load_file() أو master..xp_dirtree.

تقنيات البحث المتقدم (Bug Bounty Hunting Methodology)

AR

في برامج الـ Bug Bounty، نادرًا ما تجد SQLi واضحة في متغيرات الـ URL (GET params). يجب البحث في:

  • HTTP Headers: مثل User-Agent, X-Forwarded-For, Referer والتي يتم تخزينها غالبًا في قواعد البيانات لأغراض الـ Logging.
  • JSON/XML APIs: استبدال القيم بـ Payloads، وأحيانًا تغيير نوع البيانات (من Integer إلى Array أو String) لكشف أخطاء.
  • Second-Order SQLi: إدخال Payload في حقل (مثل اسم المستخدم عند التسجيل) لا ينفذ فوراً، ولكن يتم تنفيذه لاحقاً عند عرض ملفك الشخصي أو في لوحة تحكم الإدارة (Admin Panel).
  • WAF Bypass: تخطي حماية جدار الحماية باستخدام مسافات بديلة /**/ أو ترميز (URL Encoding, Hex Encoding) أو تلاعب بالأحرف sELecT.

أوامر الـ SQLMap المتقدمة / Advanced SQLMap Commands

# استغلال طلب تم حفظه من Burp (شامل كل الـ Headers)
sqlmap -r request.txt --batch --dbs

# محاولة تخطي الـ WAF باستخدام نصوص مساعدة (Tampers)
sqlmap -u "https://target.com/api/users?id=1" --tamper=space2comment,between,randomcase --level=5 --risk=3

# استغلال ثغرة Time-based Blind بتحديد التقنية وتجاهل الباقي لتوفير الوقت
sqlmap -u "https://target.com/page?id=1" --technique=T --time-sec=5

# محاولة الحصول على OS Shell مباشرة (يعتمد على صلاحيات الـ DB)
sqlmap -r request.txt --os-shell

كيفية الحماية (Remediation)

الخطوة الأساسية والوحيدة الفعالة هي Prepared Statements (Parameterized Queries) حيث يتم فصل أوامر SQL عن البيانات المدخلة بشكل كامل، بحيث تُعامل البيانات كنص عادي ولا يمكن تنفيذها أبداً. استخدام ORMs الحديثة (مثل Entity Framework, Hibernate, Sequelize) يوفر هذه الحماية افتراضياً.
