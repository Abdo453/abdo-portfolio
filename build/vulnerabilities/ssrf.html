تزوير الطلبات من جهة الخادم (SSRF) — الدليل الشامل

Server-Side Request Forgery (SSRF) — The Ultimate Deep Dive

CRITICAL OWASP A10:2021
AR

ثغرة SSRF تحدث عندما يقوم تطبيق الويب بجلب مورد خارجي (مثل صورة أو رابط) بناءً على مدخل من المستخدم دون التحقق منه. المهاجم يستغل هذه الثغرة لإجبار الخادم (Server) على إرسال طلبات HTTP/DNS نيابةً عنه إلى أنظمة داخلية (Internal Networks) أو خدمات سحابية لا يمكن الوصول إليها من الخارج. هذه الثغرة من أكثر الثغرات شيوعًا وتأثيرًا في البيئات السحابية (Cloud Environments).

EN

Server-Side Request Forgery (SSRF) allows an attacker to induce the server-side application to make HTTP requests to an arbitrary domain of the attacker's choosing. This is typically used to target internal systems behind firewalls, cloud instance metadata APIs, or loopback interfaces that are not externally accessible.

آلية العمل والسبب الجذري / Root Cause Analysis

تخيل ميزة في التطبيق تقوم بتحميل صورة من رابط خارجي:

POST /api/upload-avatar
Content-Type: application/json

{"avatar_url": "https://example.com/image.png"}

إذا غير المهاجم الرابط إلى http://localhost:8080/admin، سيقوم الخادم بإرسال طلب إلى لوحة تحكم الإدارة المحلية الخاصة به والتي تثق بالطلبات القادمة من localhost (Bypassing Firewall/Access Controls).

أنواع SSRF / SSRF Types

1. Basic (In-band) SSRF

استجابة الخادم الهدف (الذي تم الوصول إليه عبر الثغرة) تظهر مباشرة للمهاجم في واجهة التطبيق الأمامي. أسهل نوع وأعلاهم خطورة.

2. Blind SSRF

لا يعرض التطبيق أي استجابة أو محتوى. يتم التأكد من الثغرة عبر إرسال الطلب إلى خادم يمتلكه المهاجم (مثل Burp Collaborator أو Webhook) ومراقبة ما إذا وصل طلب من عنوان خادم الضحية.

3. Semi-Blind SSRF

لا تظهر الاستجابة الكاملة، لكن قد يظهر التطبيق رسائل خطأ مفيدة بناءً على الاستجابة أو الوقت المأخوذ للرد، مما يسمح باستنتاج المنافذ المفتوحة.

أهداف الهجوم السحابية والداخلية (The Golden Targets)

AR

عند العثور على SSRF، ما الذي يجب أن تستهدفه؟

  • AWS Metadata API: http://169.254.169.254/latest/meta-data/iam/security-credentials/ (للحصول على مفاتيح AWS Access Keys للسيطرة على الحساب).
  • GCP Metadata API: يتطلب هيدر خاص Metadata-Flavor: Google، ولكن أحياناً يتم دمجه عبر ثغرات أخرى.
  • DigitalOcean / Alibaba Cloud: لديهم نقاط وصول مماثلة على 169.254.169.254.
  • الخدمات الداخلية: مثل Redis, Memcached, Elasticsearch, Docker API, Kubelet API والتي تعمل غالباً بدون مصادقة على منافذ محلية (مثال: http://localhost:9200 أو http://127.0.0.1:6379).

تقنيات تخطي الفلاتر (Bypassing SSRF Protections)

غالبًا ما يقوم المطورون بعمل قائمة سوداء (Blacklist) للـ localhost أو 127.0.0.1 أو 169.254.169.254. إليك طرق تخطيها:

  • IPv4 Encodings: تمثيل 127.0.0.1 بطرق مختلفة:
    Hex: http://0x7f.0x0.0x0.0x1
    Decimal: http://2130706433
    Octal: http://0177.0.0.01
  • IPv6 representation: http://[::]:80/ أو http://[0000::1]:80/
  • DNS Rebinding: إعداد دومين خاص بك ليقوم بتوجيه (Resolve) للـ IP الخارجي أولاً (لتخطي التحقق)، ثم بسرعة يوجه إلى IP داخلي (كالـ Metadata) عند قيام الخادم بجلب المحتوى.
  • استخدام روابط بديلة (Wildcard DNS): خدمات مثل http://127.0.0.1.nip.io أو http://localtest.me تشير دائماً إلى 127.0.0.1.
  • تخطي عبر Redirect: إعطاء التطبيق رابط يمتلكه المهاجم ويعيد توجيه (302 Redirect) إلى السيرفر الداخلي.

بروتوكولات أخرى (Beyond HTTP)

تطبيقات الويب قد تستخدم دوال تدعم بروتوكولات غير HTTP مثل cURL في PHP:

file:///etc/passwd         # قراءة ملفات محلية
dict://127.0.0.1:11211/    # استغلال خدمة Memcached
gopher://127.0.0.1:6379/_  # كتابة حزم TCP معقدة واستغلال Redis (Gopher Protocol)

كيفية الحماية (Remediation)

تعتبر الحماية من SSRF معقدة وتتطلب نهج دفاعي متعدد الطبقات (Defense in Depth):

  • استخدام قائمة بيضاء (Whitelist) للنطاقات المسموح بها بدلاً من القوائم السوداء.
  • إيقاف تتبع إعادة التوجيه (Disable HTTP Redirections) إن لم تكن ضرورية.
  • إجبار كافة الطلبات الصادرة من التطبيق على المرور عبر Network Proxy / Egress Firewall لمنع الوصول إلى الـ Internal IPs أو خدمة الـ Metadata (169.254.169.254).
