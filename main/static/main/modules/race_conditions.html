⚡ Race Conditions & TOCTOU — ثغرات التزامن

شرح آلية استغلال ثغرات سباق التزامن (Concurrency) والـ Time-of-Check to Time-of-Use لتنفيذ هجمات مالية أو تخطي القيود

🛠️ Burp Suite Turbo Intruder · Multi-threading scripts 📋 OWASP: Concurrency Flaws | API Vulnerability CVSS Impact: High (Financial & PrivEsc)

📚 النظرية — مفهوم التعددية والتزامن وأخطاء التصميم

تحدث ثغرات Race Conditions عندما يعالج خادم الويب طلبات متعددة بشكل متوازٍ (Multi-threaded execution) دون تطبيق قيود تمنع الوصول المتزامن لنفس المورد.

⏳ فجوة الـ Check and Use (TOCTOU)

تعتمد الثغرة على وجود فجوة زمنية بسيطة جداً (أجزاء من الثانية) بين عمليتين:
1. **Time-of-Check (وقت الفحص):** السيرفر يفحص إن كان المستخدم يملك رصيداً كافياً أو لم يسبق له استخدام الكوبون.
2. **Time-of-Use (وقت الاستخدام):** السيرفر يقوم بخصم الرصيد أو تفعيل الكوبون.

إذا أرسل المهاجم 20 طلباً متزامناً في نفس اللحظة تماماً، فسيقوم السيرفر بعملية الفحص (Check) لجميع الطلبات في نفس الوقت قبل أن يكتمل أي طلب للتحديث (Use)، فيقبل السيرفر تنفيذ الطلبات العشرين كلها!

🛡️ أشهر بيئات وقوع ثغرات التزامن

  • الأنظمة المالية: سحب الأموال أو تحويلها لأكثر من جهة في نفس الوقت.
  • أنظمة الخصومات والكوبونات: تطبيق الكوبون مرة واحدة بينما ينجح استغلال التزامن في تفعيله عدة مرات.
  • أنظمة التصويت والتقييمات: زيادة عدد الأصوات بشكل غير محدود.

2️⃣
Configure Python Script for Concurrent Requests

كتابة سكريبت بايثون داخل الأداة لإطلاق الطلبات متزامنة (Gate-based synchronization) لإرسال كل الحزم في شبكة الخادم في جزء من الميكروثانية.

def queueRequests(target, wordlists):
    engine = RequestEngine(endpoint=target.endpoint,
                           concurrentConnections=30,
                           requestsPerConnection=1,
                           pipeline=False)
                           
    # Queue multiple identical requests
    for i in range(30):
        engine.queue(target.req, gate='race_gate')
        
    # Open the gate to release all requests concurrently
    engine.openGate('race_gate')

🛡️ الحماية والدفاع — معالجة ثغرات التزامن

✅ حماية قاعدة البيانات (Locks)

  • Pessimistic Locking: قفل الصف المعني في قاعدة البيانات فور بدء قراءته لمنع أي عملية تعديل متوازية حتى يكتمل الطلب الأول: SELECT ... FOR UPDATE.
  • Optimistic Locking: التحقق من عدم تغيير رقم النسخة (Version column) للبيانات قبل حفظ التحديث في قاعدة البيانات.

🔒 قفل المزامنة البرمجية (Mutexes)

  • Mutex / Semaphore: استخدام أقفال برمجية على مستوى الكود لمنع المعالجة المتوازية لنفس المستخدم في خيوط الاتصال المتزامنة (Threads).
  • Unique Constraint: وضع قيود فريدة في قاعدة البيانات (مثل تفعيل حقل فريد يربط المستخدم بالكوبون المستعمل).
