1. إدارة الحاويات بـ Docker (Container Management)

أوامر تفاعلية لبناء وإدارة وفحص الـ Docker Containers واستكشافها:

docker run -d --name nginx-web -p 80:80 nginx

2. عمليات وأوامر Kubernetes الميدانية (K8s Operations)

استكشاف الـ pods ومستويات الصلاحيات بداخل الكلاستر واستخراج تفاصيل الاتصال:

kubectl get pods --all-namespaces

3. استكشاف واختراق Active Directory

أوامر استخدام PowerView للبحث عن النطاقات والثغرات داخل بيئة الـ Active Directory:

Get-NetDomain
Get-NetUser | select cn,title
Get-NetComputer -Unconstrained | select name

4. عمليات السحابة (Cloud Auditing & Pentesting)

أوامر استكشاف وتدقيق الصلاحيات لسيرفرات سحابة AWS cli:

aws iam list-users
aws s3 ls
aws ec2 describe-instances

6. بنك الأسئلة والاختبارات التفاعلية الـ 10

🧪 بنك أسئلة مراجعة كتاب Operator Handbook (انقر لعرض الإجابة النموذجية)
الإجابة النموذجية: الحاويات (Containers) تشترك في نفس نواة نظام التشغيل (OS Kernel) الخاص بالخادم المضيف، وتعمل كعمليات معزولة، مما يجعلها خفيفة جداً وسريعة التشغيل. بينما الآلات الوهمية (VMs) تحتوي على نظام تشغيل كامل مستقل (Guest OS) يعمل فوق الـ Hypervisor، مما يستهلك موارد عتاد أكبر بكثير.
الإجابة النموذجية: هي سكربت PowerShell يتيح استكشاف وقراءة بيانات الـ Active Directory باستخدام حساب مستخدم عادي دون الحاجة لصلاحيات المدير، للبحث عن صلاحيات ضعيفة مثل أجهزة تدعم الـ Unconstrained Delegation أو حسابات ذات صلاحيات عالية غير مؤمنة.
الإجابة النموذجية: يتم ذلك عبر تشغيل الأمر التفاعلي `kubectl exec -it [pod_name] -- /bin/bash` أو `/bin/sh`، مما يتيح لك تنفيذ أوامر مباشرة وتفقد الملفات وإعدادات الحاوية داخل الكلاستر بالوقت الحقيقي.
الإجابة النموذجية: نقوم بتشغيل الأمر الأولي `aws sts get-caller-identity`. هذا الأمر آمن ولا يترك بصمة ضخمة، ويعيد لنا اسم الحساب ورقم الـ Account ID والـ ARN للمستخدم الحالي، لمعرفة مستوى الصلاحيات المرتبط بالـ API key المسرب.
الإجابة النموذجية: الـ RBAC هو نظام ينظم تحديد الصلاحيات للمستخدمين والخدمات بداخل الكلاستر بناءً على الأدوار. يجب تأمينه لأن أي ضعف في تهيئته قد يسمح للـ pods ذات الصلاحيات المحدودة بالوصول للـ API server ورفع صلاحياتها للسيطرة بالكامل على الكلاستر.
الإجابة النموذجية: يتم منع ذلك عن طريق تحديد مستخدم عادي (Non-root user) بداخل ملف التكوين الـ `Dockerfile` باستخدام التعليمة `USER 1000` أو تطبيق معايير الأمان (Security Context) بداخل ملفات Kubernetes لمنع الحاوية من تشغيل العمليات كـ root.
الإجابة النموذجية: هي ثغرة تسمح لأي مستخدم عادي بطلب تذكرة خدمة (TGS Ticket) لأي حساب خدمة يملك اسم الخدمة (SPN). تذكرة الـ TGS تكون مشفرة بالهاش الخاص بكلمة مرور حساب الخدمة، ويمكن للمهاجم سحبها وتخمين كلمة المرور دون اتصال (Offline Cracking) للحصول على صلاحيات الخدمة.
الإجابة النموذجية: يحدث عندما ينجح مهاجم داخل الحاوية في استغلال صلاحيات ضعيفة (مثل تشغيل الحاوية بالخيار `--privileged`) أو استغلال ثغرة بنواة النظام المضيف للوصول لموارد السيرفر الرئيسي وقراءة ملفاته والتعديل على العمليات الخارجية بالكامل.
الإجابة النموذجية: الـ GPO هي أداة تتيح للمسؤولين تطبيق إعدادات حماية وسياسات موحدة (مثل سياسة كلمات المرور، أو تعطيل الـ USBs، أو تثبيت البرامج) على جميع الأجهزة والمستخدمين المنضمين للشبكة بشكل مركزي وتلقائي.
الإجابة النموذجية: يتم منع ذلك عبر تعطيل ربط مجلدات المضيف الحساسة (مثل `/var/run/docker.sock` أو `/etc/shadow`) مع الحاويات غير الموثوقة، وتحديد صلاحيات القراءة فقط (Read-only root filesystem) بداخل الحاوية لتقليل أثر أي اختراق محتمل.
