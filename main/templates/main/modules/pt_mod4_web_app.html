🔥 Web Application Testing

فحص وتأمين تطبيقات الويب، اكتشاف التقنيات، وتحديد الثغرات بناءً على منهجيات احترافية.

4.1 Technology Fingerprinting

مرحلة البصمة التقنية لمعرفة اللغات، الإطارات (Frameworks)، والخوادم التي يعتمد عليها الهدف.

WhatWeb 
whatweb https://target.com

أداة WhatWeb تقوم بتحليل الموقع لمعرفة التقنيات المستخدمة مثل إصدارات CMS وقواعد البيانات وغيرها من البصمات الحرجة.

Wappalyzer (Browser extension or CLI) 
wappalyzer https://target.com

تعمل Wappalyzer على استخراج قائمة كاملة بالمكتبات (Libraries) والتقنيات التي يعمل بها النظام سواء عبر سطر الأوامر أو كإضافة للمتصفح.

BuiltWith - Online tool 
https://builtwith.com

موقع BuiltWith هو أداة استخباراتية رائعة (OSINT) تمنحك نظرة تاريخية وحالية عن البنية التحتية البرمجية للموقع المستهدف.

Nmap with http-enum 
nmap --script=http-enum -p 80,443 target.com

يستخدم هذا السكربت في Nmap لاكتشاف المسارات المخفية والمجلدات الشائعة التي تدل على لوحات تحكم أو تطبيقات محددة مثبتة على الخادم.

Httpx (ProjectDiscovery) 
echo target.com | httpx -title -tech-detect -status-code -o tech_info.txt

أداة Httpx من الأدوات الفعالة والسريعة جداً. هذا الأمر يستخرج العنوان وحالة الاستجابة والتقنيات ويحفظ النتائج بشكل منسق في ملف نصي.

4.2 WAF Detection

مرحلة اكتشاف جدار الحماية لتطبيقات الويب (Web Application Firewall) لمعرفة الحمايات وتحديد طرق تجاوزها.

Wafw00f 
wafw00f https://target.com

تعتبر Wafw00f من أشهر الأدوات المخصصة لإرسال استعلامات خبيثة (Malicious Payloads) ومراقبة الاستجابة لتحديد نوع الـ WAF الموجود بالضبط مثل Cloudflare أو Akamai وغيرها.

Nmap WAF detection 
nmap --script=http-waf-detect,http-waf-fingerprint -p 80,443 target.com

استخدام سكربتات Nmap المدمجة لفحص البصمات الخاصة بجدران الحماية وتحليل سلوك الشبكة لتأكيد وجود WAF.

4.3 Screenshotting (Visual Recon)

المسح البصري وأخذ لقطات شاشة للنطاقات الفرعية لتوفير الوقت ومعرفة الأهداف المثيرة للاهتمام بسرعة.

Gowitness 
gowitness file -f urls.txt --threads 4
gowitness single -u https://target.com

تقوم Gowitness بالتقاط صور للنطاقات باستخدام متصفح حقيقي بالخلفية (Headless Chrome). الأمر الأول مخصص لمجموعة مسارات عبر ملف، والثاني لمسار واحد فقط.

Aquatone 
cat urls.txt | aquatone

أداة Aquatone تأخذ قائمة من الروابط وتقوم بعمل لقطات شاشة وتجميعها في تقرير HTML شامل وجميل جداً ليسهل على المختبر تصفحه.

4.4 Vulnerability Scanning with Nuclei

استخدام أداة Nuclei القوية والمبنية على قوالب (Templates) للفحص الآلي واكتشاف الثغرات المعروفة بسرعة.

Nuclei Commands 
# Basic scan
nuclei -target target.com

# Using templates
nuclei -target target.com -t cves/ -t vulnerabilities/

# Severity filtering
nuclei -target target.com -severity critical,high

# Tags
nuclei -target target.com -tags cve,exposure

# List of targets
nuclei -list targets.txt

# Rate limiting
nuclei -target target.com -rate-limit 100

# Output formats
nuclei -target target.com -o results.txt -json

شرح الأوامر:
- Basic scan: الفحص الافتراضي للهدف.
- Using templates: لتحديد مسارات محددة للقوالب مثل الثغرات أو الـ CVEs.
- Severity filtering: لفلترة البحث للثغرات الحرجة (Critical) والعالية (High) فقط، لتجنب الضوضاء.
- Tags: للبحث عن قوالب تحمل علامات محددة (مثل الانكشافات وتسريبات البيانات).
- List of targets: تمرير ملف يحتوي على قائمة من الروابط للفحص الجماعي.
- Rate limiting: تحديد عدد الطلبات في الثانية لتجنب حجبك (Banning) من قِبل الـ WAF.
- Output formats: استخراج النتائج كملف نصي عادي أو بصيغة JSON لربطها مع أدوات أخرى.

4.5 Manual Testing Methodology (OWASP Top 10 2025)

منهجية الفحص اليدوي العميقة والتي تتبع أحدث معايير OWASP لاكتشاف الثغرات المنطقية والحرجة التي لا تلتقطها الأدوات الآلية.

الصنف (Category) مفاهيم الاختبار والتطبيق (Testing Concepts) شرح عميق بالعربية
A01: Broken Access Control - Test horizontal IDOR (change ID parameter)
- Test vertical IDOR (user to admin)
- Test forced browsing to unlinked paths
- JWT manipulation (algorithm confusion, weak keys)
- Missing authorization on API endpoints 		كسر في التحكم بالوصول. يشمل محاولة التلاعب بالمعرفات (IDOR) أفقياً للوصول لمعلومات مستخدم آخر، أو عمودياً للوصول لصلاحيات الأدمن. إضافة إلى محاولة الدخول لمسارات غير معلنة (Forced Browsing)، والتلاعب بتوكنات JWT مثل تغيير الخوارزمية (Algorithm Confusion)، واختبار واجهات الـ API التي تفتقر للتحقق من الصلاحيات.
A02: Cryptographic Failures - Check weak hashing (MD5, SHA1)
- Test TLS config (TLS 1.0/1.1, weak ciphers)
- Missing HSTS, secure cookie flags
- Sensitive data in URLs
- Hardcoded keys in client-side code 		الفشل التشفيري. يعتمد على فحص استخدام خوارزميات قديمة وضعيفة للهاشات (MD5)، وجود ضعف في بروتوكول TLS، أو عدم وجود حمايات الكوكيز (Secure Flags). وكذلك التأكد من عدم تسريب معلومات حساسة في رابط الـ URL، والبحث عن مفاتيح التشفير المبرمجة مسبقاً (Hardcoded keys) في أكواد العميل كالـ JS.
A03: Injection - SQL Injection (blind SQLi on sort/search)
- NoSQL Injection (operator injection)
- OS Command Injection
- LDAP Injection
- Server-Side Template Injection (SSTI) 		ثغرات الحقن المتعددة. مثل حقن الـ SQL (بما فيه الـ Blind) عند فلاتر البحث أو الترتيب، وحقن قواعد بيانات الـ NoSQL، وحقن أوامر النظام OS، وحقن الـ LDAP في أنظمة التوثيق، وأخيراً حقن القوالب (SSTI) في محركات مثل Jinja2، التي يمكن أن تؤدي لاختراق الخادم بالكامل.
A04: Insecure Design - Missing rate limiting on OTP/login
- Weak account recovery flows
- Business logic abuse (race conditions)
- Insecure direct API design 		التصميم غير الآمن. يركز على عيوب بنية النظام نفسها كعدم وجود قيود للطلبات (Rate Limiting) على الـ OTP أو تسجيل الدخول. بالإضافة لمهاجمة مسار استعادة الحساب وتجاوزاته، والتلاعب بالمنطق التجاري (Business Logic) كاستغلال التزامن (Race Conditions) لمضاعفة الرصيد، والتصميم الخاطئ للـ API.
A05: Security Misconfiguration - Exposed admin interfaces (Jenkins, Swagger)
- Verbose error messages
- Missing security headers (CSP)
- Default credentials
- Directory listing & Misconfigured CORS 		الأخطاء في الإعدادات الأمنية. وتشمل ترك واجهات التحكم متاحة للعامة (مثل Jenkins أو Swagger)، وعرض رسائل خطأ تفصيلية تكشف مسارات السيرفر، وعدم تطبيق ترويسات الحماية مثل CSP. وأيضا استخدام كلمات مرور افتراضية، وإتاحة تصفح المجلدات (Directory Listing)، أو السماح لجميع النطاقات بالاتصال (CORS).
A06: Vulnerable and Outdated Components - Dependency scanning (npm/pip audit, snyk)
- Framework version disclosure
- Verify exploitable libraries 		المكونات القديمة أو المصابة بثغرات. ويتضمن الكشف عن النسخ والإصدارات الخاصة بالمكاتب والإطارات المستخدمة من خلال رسائل الـ Header، ثم إجراء فحص دقيق (Audit) للمكاتب وتأكيد ما إذا كانت هذه الثغرة قابلة للاستغلال فعلياً أم لا في بيئة التطبيق الحالية.
A07: Identification and Authentication Failures - Weak password policy
- Username enumeration
- Missing/weak MFA
- Session management flaws
- OAuth/SAML/SSO flaws 		فشل التحقق والهوية. يغطي سياسات كلمات المرور الضعيفة، وثغرة تخمين الأسماء (Enumeration) عبر تسجيل الدخول أو الاستعادة، وغياب المصادقة الثنائية (MFA). إلى جانب مشاكل إدارة الجلسات (Sessions) كعدم وجود وقت انتهاء أو ضعف التوكن، ومشاكل تطبيقات OAuth وتسجيل الدخول الموحد (SSO Bypass).
A08: Software and Data Integrity Failures - Insecure deserialization
- Auto-update without signatures
- CI/CD pipeline trust issues
- JS supply chain risk (missing SRI) 		فشل النزاهة والتكامل البرمجي. من أخطرها إلغاء التسلسل غير الآمن (Insecure Deserialization)، أو تحديثات تلقائية دون التحقق من التوقيع الرقمي. وتشمل أيضاً الثغرات في سلاسل التوريد البرمجية (CI/CD)، كاستدعاء مكتبات JS خارجية بدون تفعيل تحقق التكامل (SRI)، مما يسمح للهاكر بحقن أكواده فيها.
A09: Security Logging and Monitoring Failures - Verify events are logged
- Check integration with SIEM
- Alerting on suspicious patterns 		فشل التسجيل والمراقبة الأمنية. يعتمد على التحقق من أن السيرفر يسجل الأحداث المهمة كالفشل في تسجيل الدخول، وأنه مرتبط بنظام مركزي (SIEM) لتحليل السجلات، وأنه يمتلك نظام تنبيه سريع يكتشف الأنماط المشبوهة والهجمات وقت حدوثها بدلاً من اكتشافها لاحقاً.
A10: Server-Side Request Forgery (SSRF) - SSRF on URL parameters (webhook, PDF)
- Blind SSRF via out-of-band (Burp Collaborator)
- SSRF to cloud metadata (169.254.169.254) 		تزوير الطلبات من جهة الخادم (SSRF). وتحدث عندما يجبر الهاكر خادم الويب على إجراء طلبات لموارد داخلية. يمكن استغلالها في الـ Webhooks، أو مولدات الـ PDF. يمكن استخدام Burp Collaborator لاكتشاف (Blind SSRF)، والأخطر هو الوصول لبيانات خوادم السحابة (Cloud Metadata) كـ AWS لسرقة التوكنات الداخلية.

4.6 Burp Suite Workflow

مراحل العمل وتجهيز بيئة الاختبار في أداة Burp Suite.

Setup: إعداد وتجهيز الـ Proxy لربطه مع المتصفح، تثبيت شهادة (CA Certificate) لالتقاط طلبات الـ HTTPS المشفرة، وضبط إعدادات الـ Scope لتحديد النطاق المصرح باختباره لتجنب فحص مسارات غير مطلوبة. وهي المرحلة الأساسية قبل بدء أي فحص يدوي باستخدام الـ Repeater والـ Intruder وغيرها من خصائص الأداة.
