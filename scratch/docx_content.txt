1. Active Reconnaissance & Subdomain Enumeration
1.1 Overview
Active subdomain enumeration involves direct interaction with the target’s DNS infrastructure to discover subdomains that may not be publicly indexed. This is critical for expanding the attack surface beyond what passive reconnaissance reveals.
1.2 DNS Brute Forcing
Tools & Commands
Gobuster DNS Mode:
gobuster dns -d target.com -w wordlist.txt -t 50 --delay 100ms -o gobuster_results.txt
FFuF for Subdomain Fuzzing:
ffuf -w subdomains.txt -u https://FUZZ.target.com -mc 200,301,302 -t 100 -rate 50 -o fuf_results.json
Puredns (MassDNS Wrapper):
puredns -r resolvers.txt bruteforce wordlist.txt target.com
MassDNS (High-speed resolver):
massdns -r lists/resolvers.txt -t A domains.txt -o results.txt
ShuffleDNS (ProjectDiscovery):
# Subdomain resolvingshuffledns -d example.com -list subdomains.txt -r resolvers.txt -mode resolve# Subdomain bruteforcingshuffledns -d hackerone.com -w wordlist.txt -r resolvers.txt -mode bruteforce

Altdns:
python altdns.py -i input_domains.txt -o ./output/path -w altdns/words.txt

1.8 Best Practices & Anti-Detection
Rate limiting: Add delays between requests (--delay 100ms)
Rotate DNS resolvers: Use multiple resolver lists
Use VPS: Run active enumeration from cloud instances
Respect scope: Always check bug bounty scope before brute forcing
Wildcard detection: Filter wildcard responses using puredns or shuffledns

3.1 Nmap Scanning Techniques
Basic Scan Types
SYN Scan (Stealth/Half-open):
nmap -sS 192.168.1.0/24
TCP Connect Scan:
nmap -sT 192.168.1.0/24
UDP Scan:
nmap -sU 192.168.1.1nmap -sU --top-ports 100 192.168.1.1nmap -sU -p 53,67,68,69,123,161,162,514 192.168.1.1
Stealth Scans (NULL, FIN, Xmas):
nmap -sN 192.168.1.1  # NULL scannmap -sF 192.168.1.1  # FIN scannmap -sX 192.168.1.1  # Xmas scan (FIN, PSH, URG)
ACK Scan (Firewall mapping):
nmap -sA 192.168.1.1
Advanced Nmap Options
# Full port scannmap -p- 192.168.1.1# Fast scan (top 100 ports)nmap -F 192.168.1.1# Version detectionnmap -sV 192.168.1.1nmap -sV --version-intensity 9 192.168.1.1# OS detectionnmap -O 192.168.1.1# Default script scannmap -sC 192.168.1.1# Aggressive scan (OS + version + scripts + traceroute)nmap -A 192.168.1.1# Vulnerability scanning scriptsnmap --script=vuln 192.168.1.1# Specific service enumerationnmap --script=http-enum 192.168.1.1# Decoy scan (hide source)nmap -sS -D RND:10 192.168.1.1# Timing templatesnmap -T4 192.168.1.1  # Aggressivenmap -T5 192.168.1.1  # Insane (noisy)# Fragment packetsnmap -f 192.168.1.1# Host discovery onlynmap -sn 192.168.1.0/24# Skip host discoverynmap -Pn 192.168.1.1

3.5 UDP Scanning Best Practices
UDP is connectionless - responses are not guaranteed
Use protocol-specific probes (DNS query for 53, NTP for 123, SNMP for 161)
Rate limiting is crucial - ICMP port unreachable responses are often rate-limited
Combine with -sV for better service detection
Use --host-timeout to skip slow hosts

4.5 Manual Testing Methodology (OWASP Top 10 2025)
A01: Broken Access Control
Test horizontal IDOR (change ID parameter to access other users’ resources)
Test vertical IDOR (regular user accessing admin functions)
Test forced browsing to unlinked admin paths
JWT manipulation (algorithm confusion, weak signing keys)
Missing authorization on API endpoints
A02: Cryptographic Failures
Check for weak hashing (MD5, SHA1)
Test TLS configuration (TLS 1.0/1.1 enabled, weak ciphers)
Check for missing HSTS, secure cookie flags
Look for sensitive data in URLs/query strings
Search for hardcoded keys in client-side code
A03: Injection
SQL Injection (blind SQLi on sort, search, filter parameters)
NoSQL Injection (MongoDB/DynamoDB operator injection)
OS Command Injection (file processing endpoints)
LDAP Injection (authentication systems)
Server-Side Template Injection (SSTI) on Jinja2/Twig/Handlebars
A04: Insecure Design
Missing rate limiting on login, password reset, OTP endpoints
Weak account recovery flows
Business logic abuse (race conditions on payment, coupon stacking)
Insecure direct API design
A05: Security Misconfiguration
Exposed admin interfaces (Jenkins, Kibana, Swagger, phpMyAdmin)
Verbose error messages leaking stack traces
Missing security headers (CSP, X-Frame-Options, X-Content-Type-Options)
Default credentials
Directory listing enabled
Misconfigured CORS
A06: Vulnerable and Outdated Components
Dependency vulnerability scanning (npm audit, pip audit, snyk, Trivy)
Framework version disclosure in HTTP headers
Verify if vulnerable libraries are actually exploitable
A07: Identification and Authentication Failures
Weak password policy
Username enumeration on login/signup/password-reset
Missing or weak MFA
Session management flaws (predictable tokens, missing timeout)
OAuth/SAML implementation issues
SSO bypass via direct authentication endpoints
A08: Software and Data Integrity Failures
Insecure deserialization (Java/Python/Ruby)
Auto-update mechanisms without signature verification
CI/CD pipeline trust issues
JavaScript supply chain risk (loading third-party scripts without SRI)
A09: Security Logging and Monitoring Failures
Verify security-relevant events are logged
Check integration with SIEM
Alerting on suspicious patterns
A10: Server-Side Request Forgery (SSRF)
SSRF on URL parameters (webhook, image upload, PDF generation)
Blind SSRF via out-of-band channels (Burp Collaborator)
SSRF to cloud metadata endpoints (169.254.169.254 on AWS)

9.4 Testing Types
Black Box Testing
No prior knowledge of target
Simulates external attacker
Time-consuming but realistic
May miss internal vulnerabilities
Grey Box Testing
Limited knowledge (credentials, basic architecture)
Most common approach
Balances realism with efficiency
Tests access control and lateral movement
White Box Testing
Full knowledge (source code, documentation)
Most thorough analysis
Identifies root causes
Does not simulate real attacker
Red Team vs Penetration Test
Penetration Test: Time-boxed, scoped, cooperative
Red Team: Extended duration, broader scope, stealthy, tests detection/response
